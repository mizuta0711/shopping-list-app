# 設計書反映チェック＋更新

直近の実装変更を分析し、設計書を更新してください。

## Step 1: 変更されたファイルの特定

```bash
git diff --name-only HEAD~${0:-1}
```

$ARGUMENTS が数値の場合はそのコミット数分を対象。未指定なら直近1コミット。
未コミット変更がある場合は `git diff --name-only` も含める。

## Step 2: チェックリスト照合

変更されたファイルに基づき、以下のチェックリストを照合:

| 変更内容 | 更新対象 |
|---------|---------|
| `src/app/api/` 配下の変更 | `docs/設計書/API一覧.md` + `.claude/01_development_docs/03_api_design.md` + `docs/設計書/API・サービス・リポジトリ・フック対応表.md` |
| `src/lib/services/` 配下の変更 | `docs/設計書/サービス・リポジトリ一覧.md` |
| `src/features/*/hooks/` 配下の変更 | `docs/設計書/フック一覧.md` |
| `prisma/schema.prisma` の変更 | `npx tsx tools/scripts/generate-table-docs.ts` を実行してテーブル定義書を自動生成 + `docs/設計書/ER図.md` |
| `src/lib/ai/prompts/` 配下の変更 | `docs/設計書/ai-prompt-design.md` |
| `docs/features/yyyymmdd_*.md` に基づく実装 | 該当する機能設計書のタスクステータスを更新 |

## Step 3: 設計書の更新

該当する設計書を読み、変更内容を反映。

**改訂履歴の記入ルール:**
- 改訂履歴テーブルにコミットハッシュ列を含める（まだない場合は追加）
- フォーマット: `| 版数 | 日付 | コミット | 内容 | 担当 |`
- コミットハッシュは短縮形（7文字）を使用

**対応表の連鎖更新ルール（`src/app/api/` または `src/lib/services/` 変更時）:**
- `route.ts` の import 文が変わった場合、対応表のサービス列も必ず更新する
- サービスのリネーム・分割時は、対応表の該当行を grep で検索し、旧名が残っていないか確認する
```bash
# 対応表に旧サービス名が残っていないか確認
grep "旧サービス名" "docs/設計書/API・サービス・リポジトリ・フック対応表.md"
```

## Step 3.5: 機能設計書のタスクステータス更新＋完了判定

変更が `docs/features/yyyymmdd_*.md` に基づく実装である場合（Step 2 で該当した場合）:

1. **タスクステータス更新**: 実装済みタスクのステータスを ✅完了 に更新
2. **全タスク完了判定**: 機能設計書内の全タスクが ✅完了 になっているかチェック
   - ⏸️保留 や ❌却下 は「未完了」とはみなさない（理由付きで意図的に除外されたもの）
3. **全タスク完了時: sync-check を自動実行**（🔵/🟡 が残っていない場合のみ）:
   - 機能設計書の「作成・変更ファイル」セクションから変更対象を読み取る
   - 変更対象に応じて関連設計書のみ照合する（全量照合ではなくスコープを絞る）:
     - `src/app/api/` 変更 → API一覧のエンドポイント照合
     - `src/lib/services/` 変更 → サービス一覧のメソッドシグネチャ照合（`grep -rn "^export async function\|^export function"` で実装側を抽出して比較）
     - `prisma/schema.prisma` 変更 → `npx tsx tools/scripts/generate-table-docs.ts` を実行して `git diff` で差分確認
     - `src/features/*/hooks/` 変更 → フック一覧のファイル照合
   - 乖離が見つかったら修正し、改訂履歴を更新
   - 乖離がなければ「sync-check: ✅ 同期済み」と報告
4. **移動判定**:
   - **✅ と ❌ のみ**（⏸️なし） → `completed/` に移動
   - **⏸️保留が1つ以上ある** → `pending/` に移動（保留解除待ち）

## Step 4: 同期トラッキングファイルに記録

`docs/設計書/.doc-sync.md` の同期記録テーブルに追記:

```markdown
| {コミットハッシュ} | {日付} | {同期先の設計書と版数、または「更新不要（理由）」} | ✅ |
```

設計書の更新が不要だったコミットも `更新不要（理由）` として必ず記録すること。

**例外**: `.doc-sync.md` 自体の更新のみのコミットは記録不要（鶏と卵問題を回避）。

## Step 5: 結果報告

ユーザーには更新した設計書名と件数のみ簡潔に報告。更新不要の場合は「更新不要」と1行で報告。
