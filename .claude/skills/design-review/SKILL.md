# 設計レビュー

機能設計書の Stage 1 / Stage 2 に対応するレビューを実施する。

## モード

| モード | 対象 | 使用エージェント | 用途 |
|-------|------|-----------------|------|
| `feature` | 機能設計書の「3. 機能・画面設計（Stage 1）」 | code-reviewer + product-advisor（並列実行） | L規模の Stage 1 レビュー。UX・企画 + UI/UX設計・設計書整合性 |
| `tech` | 機能設計書の「4. 技術設計（Stage 2）」 | code-reviewer のみ | M/L規模の Stage 2 レビュー。API・DB・サービス・フックの整合性 |
| (省略) | 全設計書 | code-reviewer のみ（互換モード） | 単一設計書（非 Stage 構造）のレビュー。後方互換用 |

## Step 1: 対象設計書の特定

$ARGUMENTS をチェック:
- `feature` → Stage 1 モード
- `tech` → Stage 2 モード
- 上記以外 or 未指定 → 互換モード

対象の機能設計書（`docs/features/yyyymmdd_*.md`）は以下で特定:
1. 直近の未コミット変更から `docs/features/` 配下のファイルを探す（`git diff --name-only`）
2. 見つからない場合は、直近コミットから探す（`git diff --name-only HEAD~1`）
3. それでも見つからない場合はユーザーに確認

## Step 2: 設計書の部分読み（トークン効率化）

**全文読みはしない**。必要なセクションのみ offset/limit で読む。

```bash
# メタ情報の取得（冒頭30行）
Read(file, offset=0, limit=30)

# Stage 1 セクションの開始行を特定
grep -n "^## 3\\. 機能・画面設計" {設計書パス}

# Stage 2 セクションの開始行を特定
grep -n "^## 4\\. 技術設計" {設計書パス}
```

モードに応じて読む範囲:

| モード | 読むセクション |
|-------|-----------------|
| `feature` | メタ情報 + Stage 1（3章）+ 関連する `docs/設計書/`（UIパターン確認に `.claude/02_design_system/`）|
| `tech` | メタ情報 + Stage 2（4章）+ 関連する `docs/設計書/`（API/DB/サービス設計） |
| (省略) | 設計書全体 + 関連する `docs/設計書/` |

## Step 3: エージェント起動

### feature モード（並列実行）

code-reviewer と product-advisor を並列起動する:

```
# code-reviewer（Stage 1: 設計整合性 + UI/UX設計）
Agent(subagent_type: "code-reviewer", model: "sonnet", prompt:
  "Stage 1 設計レビュー。対象: {設計書パス} の「3. 機能・画面設計」セクション。
   以下の観点で評価して:
   【設計書整合性】
   - 設計書間の矛盾（他の docs/設計書/ との整合）
   - 「5. ブラウザ評価計画」の記入状況
   【UI/UX 設計観点】
   - デザインシステム準拠 (.claude/02_design_system/ との整合)
   - 既存画面のレイアウトパターンとの一貫性
   - 情報階層・視覚優先度の妥当性
   - アクセシビリティ (ARIA, キーボードナビ, コントラスト)
   - レスポンシブ対応方針 (スマホ375px / PC1280px)
   - コンポーネント再利用の可否
   結果は『要修正』テーブル形式で返して。")

# product-advisor（企画・UX体験）
Agent(subagent_type: "product-advisor", model: "sonnet", prompt:
  "Stage 1 設計レビュー。対象: {設計書パス} の「3. 機能・画面設計」セクション。
   以下の観点で企画妥当性を評価して:
   - 課題-解決策フィット
   - ユーザーフローの自然さ・摩擦ポイント
   - 代替案の検討余地
   - 優先度の妥当性
   - 既存機能との整合性 (重複・矛盾)
   結果は『課題・懸念』テーブル形式（優先度: 高/中/低付き）で返して。")
```

**矛盾・タイムアウト時**:
- 片方のエージェントが失敗した場合、成功した側の結果だけで一旦レポート作成、失敗したエージェントは再実行を promptする
- 両者の指摘が真逆のケース（技術制約 vs UX要求）は自動解決せず「要判断」として明示しユーザーに委ねる

### tech モード

code-reviewer を起動:

```
Agent(subagent_type: "code-reviewer", model: "sonnet", prompt:
  "Stage 2 設計レビュー。対象: {設計書パス} の「4. 技術設計」セクション。
   以下の観点で評価して:
   - API 設計のレスポンス構造 ⇔ DB 設計のテーブル定義
   - サービス設計の戻り値型 ⇔ API 設計のレスポンス
   - フック設計 ⇔ API 設計のレスポンス形式
   - 全エンドポイントにリクエスト・レスポンス JSON 例があるか
   - ページネーションキーが items に統一されているか
   - 型安全性・命名規則
   結果は『要修正』テーブル形式で返して。")
```

### 互換モード（モード未指定）

単一設計書（Stage 分離されていないもの）向け。従来の全項目チェック:

- API関連: リクエスト・レスポンスJSON例、API一覧.md と 03_api_design.md の一致、ページネーションキー items統一
- DB関連: APIレスポンスのフィールドがDBスキーマに存在、型一致
- サービス/フック: 対応表の記載、サービスメソッドがAPI設計をカバー
- UI/画面: 既存レイアウトとの一貫性、レスポンシブ、コンポーネント再利用
- ブラウザ評価計画: セクション存在、項目の具体性、実現可能性、エッジケース、レスポンシブ、前提条件

評価計画が未記入の場合、設計書の「1. 概要」「3. 設計詳細」から対象画面を特定し、草案を生成して設計書に追記する。

## Step 4: 結果統合 + ファイル保存

### feature モード の統合フォーマット

`docs/reviews/YYYYMMDD_HHMMSS_design-review-feature.md` に保存:

```markdown
# Stage 1 設計レビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: {設計書パス}
- モード: feature (Stage 1)

## 総合判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し

## code-reviewer（設計整合性 + UI/UX設計）

| # | 重要度 | 内容 | 対象 | 対応状況 |
|---|--------|------|------|---------|

## product-advisor（企画・UX体験）

| # | 優先度 | 観点 | 指摘内容 | 対応状況 |
|---|--------|------|---------|---------|

## 両者の指摘が競合する場合

（code-reviewer の技術制約と product-advisor のUX要求が矛盾するケース等）

| # | 競合内容 | code-reviewer の主張 | product-advisor の主張 | ユーザー判断 |
|---|---------|-------------------|----------------------|-------------|
```

### tech モード の保存先

`docs/reviews/YYYYMMDD_HHMMSS_design-review-tech.md`（code-reviewer の結果のみ）

### 互換モード の保存先

`docs/reviews/YYYYMMDD_HHMMSS_design-review.md`（従来形式）

## Step 5: 機能設計書への記録

レビュー結果ファイルを保存した後、機能設計書の該当セクションにレビュー記録を追記する:

- feature モード: 機能設計書の「3-3. Stage 1 レビュー記録」テーブルに追記
- tech モード: 機能設計書の「4-5. Stage 2 レビュー記録」テーブルに追記

記録内容: 日付 / レビューファイルパス / 判定 / 対応状況

## Step 6: ユーザーへの報告

以下のみ報告（詳細はレビューファイル参照）:

- 総合判定
- 重要な指摘の件数（要修正: N件、競合: M件）
- レビューファイルのパス
- **feature モードの場合**: 「ユーザー承認後、Stage 2 の記入に進んでください」
- **tech モードの場合**: 「ユーザー承認後、実装フェーズに進んでください」
