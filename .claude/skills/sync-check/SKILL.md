# 設計書 ↔ 実装 網羅的同期チェック

`/update-docs` が「変更駆動」で差分を反映するのに対し、このスキルは「全量照合」で設計書と実装の乖離を検出する。

**実行タイミング**: フェーズ完了時、push前、大規模リファクタリング後

---

## Step 1: サービス一覧の照合（ファイル + メソッドシグネチャ）

### 手順

#### Step 1a: サービスファイルの照合

1. `src/lib/services/` 配下の全 `.ts` ファイル一覧を取得（サブディレクトリ含む）
2. `docs/設計書/サービス・リポジトリ一覧.md` を読む
3. 以下を照合:
   - **コードに存在するが設計書にないサービス** → 追加漏れ
   - **設計書にあるがコードに存在しないサービス**（削除済み表記を除く） → 削除漏れ

```bash
# 実装側: サービスファイル一覧
find src/lib/services -name "*.ts" | sort
```

#### Step 1b: メソッドシグネチャの照合

1. 実装側のエクスポート関数を一括抽出:

```bash
# 全サービスの export 関数シグネチャを抽出
grep -rn "^export async function\|^export function" src/lib/services/ --include="*.ts" | sort
```

2. 設計書側のメソッド一覧と照合:
   - **実装に存在するが設計書にない関数** → 追加漏れ
   - **設計書にあるが実装に存在しない関数** → 削除漏れ（実装が削除されたのに設計書に残っている）
   - **引数の型や数が変わっている関数** → シグネチャ変更漏れ

3. 差分がある関数のみ、実装のシグネチャを読んで設計書と比較する。全関数を読む必要はない。

**重要**: 内部関数（`export` されていない `function` や `async function`）は照合対象外。設計書に書くべきはパブリックAPIのみ。

---

## Step 2: フック一覧の照合

### 手順

1. `src/features/*/hooks/` と `src/lib/hooks/` 配下の全フックファイル一覧を取得
2. `docs/設計書/フック一覧.md` を読む
3. 以下を照合:
   - **コードに存在するが設計書にないフック** → 追加漏れ
   - **設計書にあるがコードに存在しないフック**（削除済み表記を除く） → 削除漏れ

```bash
# 実装側: フックファイル一覧
find src/features -path "*/hooks/*.ts" -o -path "*/hooks/*.tsx" | sort
find src/lib/hooks -name "*.ts" 2>/dev/null | sort
```

---

## Step 3: 対応表のサービスマッピング照合

### 手順（grep ベースで効率的にチェック）

**全 route.ts を1つずつ読まないこと。** 以下の手順で機械的に照合する:

1. 対応表を読み、サービス列に記載されたサービス名を抽出
2. 各サービス名が対応する route.ts の import 文に実際に存在するか grep で確認

```bash
# Step 3a: route.ts の数と対応表の行数を比較（大まかな漏れチェック）
find src/app/api/v1 -name "route.ts" | wc -l
grep -c "^|" docs/設計書/API・サービス・リポジトリ・フック対応表.md

# Step 3b: 全 route.ts のサービス import を一括抽出
grep -rh "from.*services/" src/app/api/v1/ --include="route.ts" | sort -u

# Step 3c: 対応表に書かれたサービス名が実際の import に存在するか確認
# 差分があるルートのみ route.ts を個別に読む
```

3. **差分があるルートのみ** route.ts を読んで詳細確認（全ファイルは読まない）

---

## Step 4: テーブル定義書の照合

テーブル定義書は `tools/scripts/generate-table-docs.ts` で自動生成される。
schema.prisma を変更した場合、自動生成スクリプトを再実行するだけで同期が取れる。

### 手順

1. 自動生成スクリプトを実行して最新化:
```bash
npx tsx tools/scripts/generate-table-docs.ts
```

2. `git diff docs/設計書/テーブル定義書.md` で差分を確認
   - **差分なし** → 同期済み
   - **差分あり** → スキーマ変更後に自動生成が未実行だった。差分をコミットに含める

---

## Step 5: 機能設計書のタスクステータス照合

### 手順

1. `docs/features/` 配下（`completed/` と `pending/` を除く）の全 `.md` ファイルを取得
2. 各機能設計書のタスク一覧を読み、以下をチェック:
   - **🔵未実施 / 🟡実装中 のタスクが、実際には実装済みではないか**
     - タスクに記載されたファイルやコンポーネントが既に存在するか grep/glob で確認
     - 存在すれば ✅完了 に更新すべき
   - **全タスクが ✅完了 / ⏸️保留 / ❌却下 なのに `completed/` や `pending/` に移動されていないか**
     - 🔵/🟡 が残っていなければ移動対象
3. `docs/features/completed/` 内の設計書に 🔵未実施 / 🟡実装中 のタスクが残っていないか（誤移動チェック）

```bash
# 未完了タスクが残っている features ファイルを検出
grep -l "🔵\|🟡" docs/features/*.md 2>/dev/null

# completed/ 内に未完了タスクが残っていないか
grep -l "🔵\|🟡" docs/features/completed/*.md 2>/dev/null
```

## Step 7: 乖離の修正

乖離が見つかった場合:

1. 各設計書を修正（改訂履歴も更新、コミットハッシュは `(pending)`）
2. 修正内容をユーザーに報告し、確認を求める
3. 確認後コミット → コミットハッシュを改訂履歴に反映 → `.doc-sync.md` に記録
