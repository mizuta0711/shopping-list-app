# コードレビュー＋指摘対応

## Step 1: レビュー対象の特定

$ARGUMENTS が指定されている場合はそのファイル/ディレクトリを対象。
未指定の場合は直近の未コミット変更を対象:

```bash
git diff --name-only
```

変更がない場合は直近1コミットの変更ファイル:

```bash
git diff --name-only HEAD~1
```

## Step 2: レビュー実施

変更内容に該当する項目のみチェックする（全項目を毎回チェックしない）。

### API関連の変更時
- `.claude/01_development_docs/03_api_design.md` とレスポンスが一致しているか
- ページネーションキーが `items` に統一されているか

### 常にチェック
- `any` 型の使用がないか
- API呼び出しの try/catch
- withAuth による認証チェック・userId スコープ
- N+1クエリがないか

### UI変更時
- 既存レイアウトとの一貫性
- レスポンシブ対応（スマホ375px / PC1280px）
- インライン `style={{ width }}` がスマホで画面幅を超えていないか
- ⚠️ **ブラウザテスト リマインド**: UI変更を含む場合、レビュー結果の末尾に以下を付記する:
  `「UI変更を含むため、コードレビュー指摘の修正後に /browser-test を実施してください」`

## Step 3: 結果をファイルに保存 + 報告

レビュー結果を `docs/reviews/YYYYMMDD_HHMMSS_code-review.md` に直接書き込む:

```markdown
# コードレビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: （レビュー対象ファイル一覧）

## 指摘事項
| # | 重要度 | 場所 | 内容 | 対応状況 |
|---|--------|------|------|---------|
| 1 | 重要/中/低 | ファイル:行 | （指摘内容） | 修正済み/未対応 |

## 判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し
```

ユーザーには**判定・指摘件数・重要な指摘の概要**のみ報告する。「良かった点」の列挙は不要。

## Step 4: 指摘対応（自動修正）

重要度が「重要」の指摘で、明確な修正方法がある場合は自動的に修正を行う。
修正後、レビュー結果ファイルの対応状況を「修正済み」に更新する。

修正が複雑な場合や判断が必要な場合は、修正案を提示してユーザーに確認を求める。
