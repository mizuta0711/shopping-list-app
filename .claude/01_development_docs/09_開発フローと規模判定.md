# 開発フローと規模判定

本テンプレートの**変更規模判定**と**S/M/L 開発フロー**の詳細仕様。
CLAUDE.md の「設計・実装フロー」「バグ修正の原則」から参照される詳細ドキュメント。

---

## 1. 規模判定の詳細

### 1-1. 基準表（5軸）

| 観点 | S | M | L |
|------|---|---|---|
| 変更ファイル数 | 1〜2 | 3〜10 | 10超 または広範囲 |
| 新規API | なし | 既存パターンに沿った 1〜2 | 新パターン or 複数 |
| DBスキーマ変更 | なし | カラム追加のみ | 新テーブル、FK変更、破壊的変更 |
| UX変更 | なし（既存挙動維持） | 限定的（1画面内） | 複数画面・フロー全体 |
| 設計書更新 | 不要 | 必要 | 必要（複数種類） |

#### 判定ルール

1. **最も該当する項目が多い規模を採用**
2. **1つでも L 相当の項目があれば L に引き上げる**
3. **UX 変更を含む場合は自動的に L 扱い**（2段階設計が必要なため）

### 1-2. 判定プロセス

**AI が推測 → ユーザーが承認** の2ステップ。

```
1. ユーザーが作業を依頼
   例: 「ユーザープロフィールに自己紹介欄を追加して」

2. メインClaude が指示内容と影響範囲から S/M/L を推測して提示
   例: 「これは M 規模と判断します。
         既存の users テーブルにカラムを追加、
         プロフィール画面と編集画面の2画面を修正、
         関連ファイル5つ程度、UX変更は限定的。
         進めてよいですか？」

3. ユーザーが承認 or 修正
   例: 「OK、Mで進めて」または「いや、通知機能とも連動するから L で」

4. 承認された規模でフローに入る
```

### 1-3. AI バイアス補正ルール

AI が「楽な方（S 寄り）」に判定する傾向を抑制するため、以下を徹底する:

- **判定に迷ったら L 寄りで提示する**
- ファイル数や DB 変更の有無が不明確なら **M 以上で提示する**
- **過小見積もりより過大見積もりの方が安全**（ユーザーが S に下げるのは容易、L を後から発覚するのは辛い）

### 1-4. スコープ拡大時の再判定ルール

実装中に「予想より大きな変更になりそう」と気づくケースの対応:

| 気づくタイミング | 対応 |
|----------------|------|
| 着手前 | 即停止して再判定 |
| フェーズ実行中 | **現フェーズを完了**させ、フェーズ完了時の報告で再判定を提案 |
| 破壊的変更・スキーマ変更が必要と判明 | **即停止**（戻しが困難なため） |

**原則**:
- 作業途中で止めると中途半端な状態になるため、切りの良いところまで進める
- AI 単独でフロー切り替えをしない（必ずユーザーに報告）
- 引き上げ判断後は、必要な設計書追記・レビュー等を**遡って実施**する

### 1-5. 補助軸（変更の性質）

規模 (S/M/L) とは独立した軸として、変更の性質を意識する:

| 区分 | 意識すべき点 |
|------|-------------|
| 新規追加 | 既存パターンを先に確認してから設計する。既存コンポーネント・フック・サービスで再利用可能なものを特定 |
| 改修 | 「何を変えるか・何を変えないか」の境界を明示する。既存挙動を壊さない保証をレビュー観点に含める |
| バグ修正 | 根本原因を確定してから修正する（§3 参照）。症状に飛びつかない |

性質は規模と独立した概念。「M 規模の改修」「L 規模のバグ修正」等は自然な組み合わせ。

## 2. S/M/L フローの詳細

### 2-1. S 規模の詳細（軽微な変更）

対象:
- UI 微調整、テキスト修正、1ファイル程度の修正
- 軽微なバグ修正（タイポ、表示ずれ、null チェック漏れ等）

フロー:
```
実装 → /build-check → コミット → /done
```

省略する工程:
- 設計書作成 (不要)
- 設計レビュー (不要)
- コードレビュー (不要)
- ブラウザテスト (UI 変更ありなら手動確認程度)
- `/update-docs` (設計書変更なし)

### 2-2. M 規模の詳細（機能追加・API変更・複数ファイル変更）

**前提**: UX 変更を含まないこと。UX 変更を伴う場合は L に引き上げる。

対象:
- 機能追加（既存画面にフィールド・ボタン追加等）
- API 変更（既存パターンに沿った CRUD 追加等）
- 複数ファイルにまたがる改修（UX変更なし）
- 中規模のバグ修正（複数機能にまたがる不具合、API ロジック修正等）

フロー:
```
設計 → 実装 → /code-review → /browser-test → /build-check → /update-docs → コミット → /done → プッシュ
```

#### M 規模での設計書

- 機能設計書は **1段階設計**（「3. 機能・画面設計（Stage 1）」は `⚪ 不要` ステータスで空のまま）
- 「4. 技術設計（Stage 2）」と「5. ブラウザ評価計画」のみ記入
- API のリクエスト・レスポンス JSON 例は必須

#### その他の注意

- `/browser-test`: UI 変更を含む場合のみ、評価計画に基づいてスポットテストを実施
- `/update-docs` で全タスク完了を検知すると、自動で sync-check → `completed/` 移動が実行される
- レビュー前にコミットしない

### 2-3. L 規模の詳細（新機能・大規模変更・UX変更・DBスキーマ変更）

対象:
- 新機能（新画面 + 新 API + 新テーブル）
- UX 変更を含む変更（操作フローの変更、複数画面にまたがる挙動変更）
- DB スキーマ変更（新テーブル、破壊的変更）
- 大規模リファクタリング
- 大規模バグ修正（アーキテクチャレベル、セキュリティ対応等）

フロー（2段階設計必須）:
```
設計 Stage 1（機能・画面設計）
  → /design-review feature (code-reviewer + product-advisor 並列実行)
  → ユーザー承認
設計 Stage 2（技術設計）
  → /design-review tech (code-reviewer)
  → 実装 → /code-review → /browser-test → /build-check → /update-docs → コミット → /done → プッシュ
```

### 2-4. プッシュ前の共通フロー

全規模共通:
```
/pre-push-check → プッシュ
```

- `/sync-check` は `/update-docs` の全タスク完了時に自動実行
- 手動実行は `/complete-feature` で可能（フェーズ完了時・大規模変更後）

### 2-5. フロー共通ルール

- **レビュー前にコミットしない**（M, L の場合）
- 設計時、API のリクエスト・レスポンス JSON 例は必須（M, L の場合）
- サブエージェント委譲時は API 設計書のパスを明示

## 3. バグ修正の詳細

### 3-1. フロー順序（原因特定 → 規模判定 → 設計書）

バグ修正も S/M/L 判定の対象だが、**症状だけで規模を決めない**。原因特定後に判定する。

```
1. バグ発見（症状の報告）
   ↓
2. 原因特定フェーズ（対話的、設計書なし）
   - エラーメッセージ全文確認
   - ログ追加 or 最小再現手順で原因を確定
   - 「何が原因だと考えたか」を短く示す（仮説の先行共有）
   ↓
3. 規模判定フェーズ
   - 原因が確定した段階で、修正に必要な変更範囲を見積もる
   - AI が S/M/L を推測・提示 → ユーザー承認
   ↓
4. フロー選択
   ├─ S: 直接修正 → /build-check → コミット → /done
   ├─ M: 軽量な設計書作成 → /code-review → /build-check → /update-docs → コミット → /done
   └─ L: 通常の設計書作成（2段階設計）→ 以降 L フローと同じ
```

### 3-2. 緊急バグ対応の2段構え

本番障害等で原因特定が追いつかない場合:

1. **暫定修正（S 規模）**: 原因不明のまま応急処置。影響を最小化
2. **恒久対応（後日 M/L）**: 落ち着いた後、根本原因を調査して恒久修正

**暫定修正時の注意**:
- 「根本原因不明で暫定対応」と commit メッセージに明記
- 後日の恒久対応のために issue / TODO を必ず残す
- 暫定対応のまま放置しない

### 3-3. M/L 規模バグの設計書の書き方

専用テンプレートは用意せず、既存の機能設計書テンプレート（`docs/features/TEMPLATE.md`）をそのまま使う。セクションを以下のように読み替える:

| 機能設計書のセクション | バグ修正時の内容 |
|---------------------|-----------------|
| 1. 概要 > 背景 | 問題の発見経緯・症状 |
| 1. 概要 > 方針 | 根本原因の分析・修正アプローチ |
| 3. 設計詳細 | 修正内容・影響範囲・再発防止策 |
| 4. ブラウザ評価計画 | 修正箇所の動作確認項目 |

**専用ディレクトリは作らない**。運用で「機能設計書とバグ修正設計書が混ざって見つけにくい」問題が出たら `docs/features/bugfixes/` 等を後から検討する（YAGNI）。

## 4. 2段階設計の詳細（L 規模）

### 4-1. Stage 1: 機能・画面設計

**目的**: 「何を作るか」「なぜ作るか」を確定する（実装の「HOW」は扱わない）

**記入内容**:
- 対象画面の一覧（URL、新規/変更の区別）
- 前提条件（テストユーザー、シードデータ等）
- 操作フロー（ユーザーの動き）
- 受け入れ条件（ユーザー目線の「完成」の定義）
- UX 評価の重点観点（レイアウト、視認性、操作性等）

**レビュー**: `/design-review feature`
- **code-reviewer**: 設計書間の整合性 + UI/UX 設計観点（デザインシステム準拠、レイアウト、アクセシビリティ等）
- **product-advisor**: 企画・UX 体験観点（課題-解決策フィット、ユーザーフローの自然さ、摩擦ポイント等）

**承認ゲート**: ユーザー承認で Stage 2 に進む

### 4-2. Stage 2: 技術設計

**目的**: 「どう作るか」を確定する

**記入内容**:
- API 仕様（リクエスト・レスポンス JSON 例）
- テーブル構造 / スキーマ変更
- サービス・フック設計
- 実装詳細（データフロー、状態管理等）

**レビュー**: `/design-review tech`
- **code-reviewer**: 技術的な矛盾・設計書整合性（API 設計・DB 設計・サービス設計の一貫性）

**承認後**: 実装フェーズへ

### 4-3. Stage 2 → Stage 1 戻り手順

Stage 2 記入中に技術制約で実現不可と判明するケースの対応:

1. Stage 2 の記入を中断
2. 設計書メタ情報の「Stage 1 ステータス」を `🟡 再レビュー中` に戻す
3. 該当する Stage 1 セクション（受け入れ条件・操作フロー等）を修正
4. `/design-review feature` を再実行
5. ユーザー再承認
6. Stage 2 を再開

改訂履歴には「Stage 1 を再修正した版」として明記する。
