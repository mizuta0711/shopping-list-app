# Dev Container 移行エージェント用プロンプト

このプロジェクト（skillup_mock）と同じ Dev Container 環境に、Windows SSD 上の別プロジェクトを移行するためのエージェント向け指示書。

## 目的（実現したい環境）

**ゴール**: Windows SSD 上にあった既存プロジェクトを、**WSL ネイティブ FS（`~/Project/Web/<repo>`）に置き、VS Code Dev Container で開発できる状態**にする。Windows 直開発・WSL 直開発との違いを以下に整理する。

### 達成したい状態

| 観点 | 実現したい環境 |
|------|---------------|
| **配置先** | WSL ネイティブ FS（`~/Project/Web/<repo>`）。Windows FS（`/mnt/c/...`）も WSL の `\\wsl$` 経由 Windows 編集も使わない |
| **編集環境** | VS Code を WSL から起動し、`Dev Containers: Reopen in Container` でコンテナ内編集 |
| **ランタイム** | プロジェクトの言語・依存はホスト（Windows / WSL）に入れず、コンテナ内に閉じる。ホストには Docker Desktop と VS Code 拡張だけあればよい |
| **Claude Code** | コンテナ起動直後から認証済み状態で使える（WSL ホストの `~/.claude` を bind mount） |
| **ブラウザテスト** | Playwright を使うプロジェクトは、コンテナ内で Chromium が起動できる（`--ipc=host` + システム依存ライブラリ込み） |
| **ポート転送** | dev サーバー等を Windows ブラウザから `http://localhost:<port>` でアクセス可能 |
| **シークレット** | `.env.local` 等に集約。`.devcontainer/` には書かない・コミットしない |
| **再現性** | リポジトリを clone → VS Code で Reopen in Container するだけで、誰のマシンでも同じ開発環境が再現される |

### 解決したい課題（移行の動機）

- Windows ホストに各種ランタイム（Node/Python/Go 等）を入れて散らかすのを避けたい
- Windows FS（`/mnt/c/`）経由の I/O 遅延・ファイル監視不安定を避けたい
- Claude Code の認証をプロジェクトごとに再ログインせず使いまわしたい
- マシンを変えてもセットアップが再現でき、チーム内で開発環境を共有しやすくしたい

### 非目的（やらないこと）

- Windows ホスト上に Node 等を直接インストールする構成
- `/mnt/c/` `/mnt/d/` 等 Windows FS 配下に置いたまま開発する構成
- 本番デプロイ用 Dockerfile の整備（あくまで開発環境）

## なぜ WSL ネイティブ FS（`~/Project/Web/`）に置くのか

「Windows SSD（`/mnt/c/` `/mnt/d/`）に置いたほうが Windows ツールから触りやすいのでは？」という疑問への回答。

### 結論

- **動作はする**が、Next.js / Vite / Webpack 等の dev サーバーを使うプロジェクトでは性能・挙動の差が大きく、WSL ネイティブ配置を推奨する
- **Windows からのファイル編集は WSL ネイティブ配置でも可能**（後述の `\\wsl$\` 経由）

### 性能・挙動の比較

| 項目 | WSL ネイティブ FS（`~/Project/Web/`） | Windows FS（`/mnt/d/...` 等） |
|------|---------------------------------------|------------------------------|
| I/O 速度 | ext4 ネイティブ（速い） | 9P プロトコル経由（3〜10倍遅い） |
| `npm install` | 数十秒 | 数分〜十数分 |
| dev サーバー起動 | 数秒 | 十数秒〜数十秒 |
| HMR（ホットリロード） | 即座 | 遅延 or 効かないことがある |
| ファイル監視（chokidar 等） | 安定 | 取りこぼしあり（Turbopack の不具合の根本原因） |
| `node_modules` の I/O | サクサク | 律速して全体が重くなる |
| Git 操作 | 速い | 遅い（特に `git status`） |

### Windows からアクセスしたい場合

WSL ネイティブ配置のままでも、Windows 側から以下の方法でアクセスできる:

| 用途 | アクセス方法 |
|------|-------------|
| Windows エクスプローラー | `\\wsl$\Ubuntu\home\<user>\Project\Web\<repo>`（または `\\wsl.localhost\Ubuntu\...`） |
| Windows 版 VS Code | 上記 UNC パスでフォルダを開く（ただし Remote-WSL 経由のほうが快適） |
| Windows 版エディタ（Sublime / Notepad++ / JetBrains 等） | UNC パスで開ける |
| Git for Windows / Git GUI | UNC パスで操作可 |
| Windows のターミナル（PowerShell 等） | `cd \\wsl$\Ubuntu\home\<user>\Project\Web\<repo>`（一部コマンドで制限あり） |

> **メモ**: WSL2 起動中のみ `\\wsl$\` が利用可能。WSL を停止すると見えなくなる。

### どうしても Windows FS（`/mnt/d/...`）に置きたい場合

性能犠牲を受け入れる前提で、以下の対応が必要になる:

1. Next.js は `next dev --webpack` を付ける（Turbopack のファイル監視不具合回避）
2. ファイル監視が効かない場合は `WATCHPACK_POLLING=true` `CHOKIDAR_USEPOLLING=true` 等の polling 系環境変数を付与（CPU 消費増）
3. `node_modules` だけコンテナ内ボリュームに分離する手もあるが構成が複雑化する
4. dev サーバー起動・HMR の体感速度低下は「そういうもの」と割り切る

このプロンプトのデフォルトは WSL ネイティブ配置。`/mnt/d/...` 配置にする場合はユーザーが明示的に指示すること。

## 使い方

1. WSL ターミナルで `cd ~/Project/Web` し、そこで Claude Code を起動する（まだリポジトリを clone する前の状態でよい）
2. 下記「エージェントへのプロンプト」セクションを丸ごとコピーして貼り付ける
3. プロンプト内の `{{...}}` プレースホルダを埋めてから送信する

> **配置ルール**: 移行先は **`~/Project/Web/<repo-name>`** で固定。WSL ネイティブ FS（ext4）に置くことが I/O・ファイル監視・Turbopack 動作の前提条件。

## 前提条件（移行先マシン）

- Windows + WSL2（Ubuntu 等）がセットアップ済み
- Docker Desktop インストール済み・WSL2 統合 ON
- VS Code に拡張機能 `ms-vscode-remote.remote-containers` インストール済み
- WSL 側で `claude` コマンドがログイン済み（`claude auth status` で確認可能）
- リファレンスプロジェクト `skillup_mock` の `.devcontainer/devcontainer.json` を参照できる状態

## 設計方針（このテンプレートで踏襲する原則）

| 項目 | 方針 | 理由 |
|------|------|------|
| プロジェクト配置 | **WSL ネイティブ FS（`/home/<user>/...`）**に置く | Windows FS（`/mnt/c/...`）経由の 9P マウントは I/O が遅く、Turbopack/ファイル監視と相性が悪い |
| ベースイメージ | スタックに応じた `mcr.microsoft.com/devcontainers/*` を使う | 公式メンテで安定・拡張機能対応が良い |
| Claude Code 認証共有 | `~/.claude` と `~/.claude.json` を WSL の `${localEnv:HOME}` から bind mount | コンテナ起動ごとの再ログインを回避 |
| `--ipc=host` | Playwright/Chromium を使う場合のみ付与 | `/dev/shm` 不足によるクラッシュ回避 |
| 環境変数 | `.env.local` 等に集約し `remoteEnv` には書かない | シークレットを `.devcontainer` にコミットしない |
| `dev` バンドラー | Next.js プロジェクトでファイル監視に問題が出たら `--webpack` を一時付与 | Turbopack が WSL bind mount で誤動作するケースがあるため |
| Claude Code 権限モード | **コンテナ内は `bypassPermissions`、WSL ホスト側は default**（毎回確認） | コンテナはホスト資産から隔離されているのでブラストラディウスが小さい。WSL ホストは `/mnt/c` 等にアクセスできるため、誤操作防止に確認を残す |
| Windows FS への deny | `.claude/settings.json` で `/mnt/**` への Read/Edit/Write/Bash を deny | WSL ホストで起動した場合の事故防止。コンテナ内では `/mnt` が存在しないため無害 |

## 作業手順

### Step 0: 起動位置の確認

1. `pwd` を実行。`/home/<user>/Project/Web` でなければ停止し、ユーザーに `cd ~/Project/Web` してから再開するよう案内する。
2. `mkdir -p ~/Project/Web` で念のためディレクトリを担保。
3. `git --version` と `gh --version`（あれば）で前提コマンドを確認。`gh auth status` が ok なら HTTPS clone でも認証エラーが出ない。

### Step 1: リポジトリの取得（WSL `~/Project/Web/` 配下へ）

`Windows 側にローカルリポジトリがあるか` の答えで分岐する。

#### A. Windows 側にローカルがない、または GitHub の最新を素直に持ってくる場合（推奨）

```bash
cd ~/Project/Web
git clone {{GitHub URL}} {{リポジトリ名}}
cd {{リポジトリ名}}
git checkout {{使用ブランチ}}   # 省略時はスキップ
```

- SSH URL なら `~/.ssh/` に WSL 側の鍵が登録されている前提。鍵がなければ `gh repo clone {{owner/repo}}` を案内する（gh の OAuth トークンで HTTPS clone される）。
- `gh` も未認証なら、ユーザーに `gh auth login` または SSH 鍵登録を促して停止する。

#### B. Windows 側にローカルリポジトリがあり、未コミット作業を持ち越したい場合

1. ユーザーに Windows 側で `git status` と `git stash list` の結果を確認してもらう。未コミット作業があれば**コミットまたは push してから移行**するよう案内する（Windows と WSL でファイル権限・改行コードの扱いが異なり、コピー後に差分が出るため）。
2. クリーンになったら A の手順で WSL に clone する。Windows 側のローカルは原則削除せず一時保管しておく（万一の差し戻し用）。

#### 共通

- clone 後 `ls -la` で中身を確認。`.devcontainer/`、`.env.local`、`.git/` の有無をログに残す。
- `git config core.autocrlf input` をリポジトリローカルで設定（CRLF 混入防止）。
- `.env.local` 等のシークレットは Git 管理外なので clone では取得できない。Windows 側から手動コピーが必要なら、ユーザーに具体パスを案内する（例: `cp /mnt/c/Users/foo/Project/my-app/.env.local ~/Project/Web/{{リポジトリ名}}/`）。

### Step 2: 事前確認

1. `pwd` が `~/Project/Web/{{リポジトリ名}}` であることを確認。
2. `git status` がクリーンであることを確認。
3. `.devcontainer/devcontainer.json` が既に存在するか確認。存在する場合は「上書きしてよいか」「マージするか」をユーザーに確認する。
4. リファレンス（skillup_mock の devcontainer.json）の構造を読み込む。手元になければユーザーに該当ファイルの内容を貼ってもらう。

### Step 3: スタック検出

以下のファイルから自動でスタックを推定する。検出できない場合のみユーザーに質問する。

- `package.json` → Node 系（"engines.node" で Node バージョン推定）

## テンプレート: devcontainer.json（フル機能版）

下記をベースに不要部分を削って利用する。

```jsonc
{
  "name": "{{プロジェクト名}}",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:20",

  // Playwright/Chromium 利用時のみ
  "runArgs": ["--ipc=host"],

  // Claude Code 認証共有（WSL から VS Code を起動する前提）
  "mounts": [
    "source=${localEnv:HOME}/.claude,target=/home/node/.claude,type=bind,consistency=cached",
    "source=${localEnv:HOME}/.claude.json,target=/home/node/.claude.json,type=bind,consistency=cached"
  ],

  "remoteUser": "node",

  "customizations": {
    "vscode": {
      "extensions": [
        "anthropic.claude-code"
        // スタック別に追加
      ]
    }
  },

  "postCreateCommand": "npm install && npm install -g @anthropic-ai/claude-code",

  // 複数 Next.js プロジェクトを同居させる場合は requireLocalPort: true を必ず付ける（後述）
  "forwardPorts": [3000],
  "portsAttributes": {
    "3000": {
      "label": "Dev Server",
      "onAutoForward": "notify",
      "requireLocalPort": true
    }
  }

  // 環境変数は .env.local に集約。remoteEnv には書かない。
}
```

## 複数 Next.js プロジェクトを同居させる場合のポート戦略

NextAuth は `NEXTAUTH_URL=http://localhost:3000` のようにポート番号を環境変数で固定するため、ホスト側のポートが**勝手に 3001 等へフォールバックされると認証コールバックが破綻する**。複数プロジェクトを同じ Dev Container 構成で運用する場合は以下を徹底する。

### 設定（必須）

各プロジェクトの `devcontainer.json` で `requireLocalPort: true` を指定する:

```jsonc
"forwardPorts": [3000],
"portsAttributes": {
  "3000": {
    "label": "Next.js Dev Server",
    "onAutoForward": "notify",
    "requireLocalPort": true
  }
}
```

### 運用ルール

- `npm run dev` を**同時に走らせるのは1プロジェクトのみ**
- コンテナ自体は複数同時起動しても問題ない（dev サーバーが listen していなければポート衝突しない）
- 別プロジェクトで開発したくなったら、現在動いている `npm run dev` を Ctrl+C で停止してから切り替える

### 期待挙動

| シナリオ | 結果 |
|----------|------|
| Project A だけ dev 起動 | A: `localhost:3000` ✓ |
| A 起動中に B でも dev 起動 | B: ホスト 3000 が埋まっているため**フォールバックせず通知のみ**。NEXTAUTH_URL は破綻しない |
| A を停止して B で起動 | B: `localhost:3000` ✓ |
| 両方コンテナ起動・dev 停止 | 衝突なし（プロセス未 listen のためフォワード自体起きない） |

### なぜ `appPort` や `runArgs: -p 3000:3000` を使わないのか

Docker レベルで host:3000 を予約する方式は、**コンテナ起動の時点で衝突**する。複数プロジェクトのコンテナを同時に起動すらできなくなり、開発体験を損なう。`forwardPorts + requireLocalPort` は「プロセスが listen した時だけ衝突する」ので、コンテナ起動と dev サーバー起動を分離して扱える。

| 方式 | 同時にコンテナ起動 | 同時に dev | 衝突時 |
|------|------------------|----------|-------|
| `appPort: ["3000:3000"]` | ❌ | ❌ | 後発コンテナ起動失敗 |
| `forwardPorts` のみ | ✓ | ✓（だが片方が 3001 にフォールバック） | NEXTAUTH_URL 破綻 |
| **`forwardPorts` + `requireLocalPort: true`** | ✓ | ❌（明示拒否） | 通知あり・URL は守られる |

## Claude Code 権限モード戦略（WSL ホスト vs コンテナ）

WSL ホストで `claude` を直接起動するか、Dev Container 内で起動するかで、ブラストラディウスが大きく異なる。本テンプレートは以下の方針で設定済み:

### 方針

| 起動環境 | 権限モード | 設定箇所 | 理由 |
|---------|----------|---------|------|
| **Dev Container 内** | `bypassPermissions`（毎回確認なし） | `.devcontainer/devcontainer.json` の `customizations.vscode.settings.claudeCode.initialPermissionMode` | コンテナはホスト FS（`/mnt/c` 等）から隔離されているため、誤操作してもワークスペース内で完結 |
| **WSL ホスト直接** | default（毎回確認） | 設定なし（VS Code 拡張の設定は WSL 側に効かない） | `/mnt/c` `/mnt/d` 経由で Windows 側資産に届くため、誤操作のリスクが大きい |

### Windows FS への deny ルール

WSL ホストで直接起動した場合の事故防止として、本テンプレートの `.claude/settings.json` には以下の deny ルールを設定済み:

```json
"permissions": {
  "deny": [
    "Read(/mnt/**)",
    "Edit(/mnt/**)",
    "Write(/mnt/**)",
    "Bash(*/mnt/**)"
  ]
}
```

- **意図**: Claude Code が `/mnt/c/Users/...` 等の Windows ファイルを読み書きしたり、そこに対して Bash コマンドを実行したりするのを禁止する
- **コンテナ内での挙動**: コンテナは `/mnt` をマウントしないため、deny ルールは存在するだけで何もしない（無害）
- **どうしても必要な場合**: ユーザー入力 `.env.local` の Windows コピー等、一時的に必要な場合は `/mnt/...` を経由せず、ユーザーが手動で `cp` してからセッションを再開するか、`.claude/settings.local.json` で個別に allow を追加する

### 検証コマンド（参考）

```bash
# 現在のプロジェクトで効いている権限の概要を確認
cat .claude/settings.json | jq '.permissions'

# ユーザーレベルの設定（重複していないか確認）
cat ~/.claude/settings.json | jq '.permissions // empty'
```

ユーザーレベル（`~/.claude/settings.json`）にも同様の deny を追加すれば、テンプレート未適用のプロジェクトでも保険が効く。

## トラブルシュート

| 症状 | 原因と対処 |
|------|-----------|
| `claude` がログイン状態にならない | `${localEnv:HOME}` が Windows パスになっている。VS Code を WSL から起動して開き直す |
| WSL 直接起動時にコンテナ並みの自動許可になっている | `.devcontainer` の `bypassPermissions` 設定はコンテナ内のみ有効なため、本来そうはならない。もし起きていれば `~/.claude/settings.json` で `defaultMode: "bypassPermissions"` 等が設定されていないか確認する |
| `/mnt/c/...` を読みたいのに deny される | テンプレートの `.claude/settings.json` で deny 済み。一時的に必要なら `.claude/settings.local.json`（コミット対象外）に対象パスのみ allow を追記し、作業後に削除する |
| Dev Container を Reopen しても権限プロンプトが毎回出る | VS Code が古い、または `claudeCode.initialPermissionMode` 設定がコンテナに反映されていない。`Dev Containers: Rebuild Container` を実行 |
| ファイル監視が効かない（Next.js 等） | Turbopack/Webpack の bind mount 互換性。`next dev --webpack` を試す |
| Playwright が `Failed to launch browser` | `runArgs: ["--ipc=host"]` を追加 |
| `permission denied` で `.claude` に書けない | コンテナ内 uid（node=1000）と WSL の uid 不一致。`postCreateCommand` に `sudo chown -R node:node /home/node/.claude` を追加 |
| 初回 `npm install` が極端に遅い | プロジェクトを WSL ネイティブ FS に置けているか確認（`/mnt/c/` 配下だと遅い） |
| NextAuth のコールバックで認証エラー | ホストのポートが 3001 等にフォールバックされている可能性。`requireLocalPort: true` を `portsAttributes` に追加し、他プロジェクトの `npm run dev` を停止してから再起動 |
| Ports パネルに「Forwarding failed」と出る | `requireLocalPort: true` 下で別プロジェクトが 3000 を専有中。期待挙動なので、片方の dev を停止すれば解消 |
