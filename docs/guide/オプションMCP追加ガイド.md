# オプション MCP 追加ガイド

本テンプレートに標準では含まれていないが、**必要に応じてユーザーが追加できる MCP** とその手順をまとめる。

[バイブコーディング運用ガイド.md](./バイブコーディング運用ガイド.md) の補足ドキュメントとして扱う。

---

## このドキュメントについて

- 標準テンプレートには **Playwright MCP のみ** が設定されている
- 上記以外の MCP は、ユーザー自身のニーズに応じて追加する
- 追加は `.mcp.json`（プロジェクト全体）または `~/.claude/settings.json`（個人用）への追記で行う
- MCP を増やすとコンテキストに該当ツールが常時ロードされ、**トークン消費が増える**。本当に必要なものだけ追加する

---

## 追加前の検討事項

MCP を追加する前に、以下を確認する:

### 1. Claude Code 標準ツールで代替できないか

Read / Write / Edit / Glob / Grep / Bash など標準ツールで足りる場合、MCPは不要。「なんとなく便利そう」で追加しない。

### 2. トークン消費の増加を受け入れられるか

MCP を追加するとツール定義がコンテキストに常時ロードされる。軽微な作業で数千トークン増える可能性がある。

### 3. プロジェクト全体で必要か、個人的に使いたいだけか

| 範囲 | 設定ファイル |
|------|-------------|
| プロジェクト全体（チーム共通） | `.mcp.json`（Git 管理下） |
| 個人的に使いたい | `~/.claude/settings.json`（グローバル） |

個人的な MCP（例: 自分だけ Notion と連携したい等）は `~/.claude/settings.json` に書く。

## Serena MCP

### 何か

[Serena](https://github.com/oraios/serena) は、コードベースに対して **セマンティックなコード検索・編集** を提供する MCP サーバー。言語サーバー（LSP）を内部で使い、シンボル単位でのコード把握・編集が可能になる。

### 向いている場面

- **大規模コードベース**（数千ファイル以上）でシンボル横断的な検索・リファクタリングをしたい
- TypeScript / Python / Rust などの LSP 対応言語で、型情報を活用した検索が欲しい
- `grep` / `Glob` では辿りにくい「関数の呼び出し元」「型の使用箇所」を追いたい

### 向かない場面

- 小〜中規模のコードベース（Claude 標準の Grep / Read で十分）
- Python 環境（uvx）を使いたくない
- トークンコストを最優先したい

### 前提条件

- **uvx** がインストールされていること（Python パッケージマネージャ `uv` に付属）
- インストール: `pip install uv` または [uv 公式手順](https://docs.astral.sh/uv/)

### 追加手順

`.mcp.json` に以下を追記する（`playwright` エントリと並列）:

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    },
    "serena": {
      "type": "stdio",
      "command": "uvx",
      "args": [
        "--from",
        "git+https://github.com/oraios/serena",
        "serena",
        "start-mcp-server",
        "--context",
        "claude-code",
        "--project",
        "YOUR_PROJECT_ABSOLUTE_PATH"
      ],
      "env": {}
    }
  }
}
```

**重要**: `YOUR_PROJECT_ABSOLUTE_PATH` はプロジェクトの絶対パスに置き換える（例: `D:/Develop/Web/myproject`）。テンプレートのまま使わないこと。

### 確認方法

1. Claude Code を再起動
2. Serena の MCP ツール（例: `mcp__serena__find_symbol`）が利用可能になっているか確認
3. 簡単なシンボル検索を試して動作確認

### Serena を使うときの注意

- 初回起動時、Serena が **プロジェクトの LSP インデックスを構築** するため数十秒〜数分かかる
- プロジェクトパスの指定ミスで起動しない場合がある → Claude Code のログで確認
- 複数プロジェクトで共通して使う場合、`~/.claude/settings.json` 側にプロジェクトごとに分けて記述する方が管理しやすい

## Context7 MCP

### 何か

[upstash/context7](https://github.com/upstash/context7) は、**各種ライブラリの最新公式ドキュメントをリアルタイムで Claude に提供する** MCP サーバー。プロンプトに `use context7` と書くだけで、Next.js / React / Prisma などの最新 API リファレンスを動的に取得できる。

LLM の学習データはカットオフがあるため、新しいライブラリバージョン（特に破壊的変更）の挙動を正確に把握できないことがある。Context7 はその**動的な補完**として機能する。

### 向いている場面

- **新しめのスタック**を使うプロジェクト（本テンプレは Next.js 16 / React 19 / Prisma 6 など学習カットオフ後のバージョンが多い）
- ライブラリの API 変更・非推奨化を頻繁にチェックしたい場合
- AGENTS.md / CLAUDE.md で「このバージョンは破壊的変更があるから確認せよ」と指示していても、実際の最新情報は取りに行けない → Context7 で動的取得可能

### 向かない場面

- 古いバージョンで安定運用しているプロジェクト（学習データで足りる）
- オフライン環境で開発する場合（ネットワーク必須）
- プロプライエタリ / 社内ライブラリ（Context7 は公開ドキュメントのみ対応）

### 前提条件

- **Node.js / npx** が利用可能なこと（Claude Code を使っていれば既に入っている）
- インターネット接続
- 特別な API キーは原則不要（無料利用枠あり）

### 追加手順

`.mcp.json` に以下を追記する:

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    },
    "context7": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp@latest"]
    }
  }
}
```

### 使い方

Claude への指示に `use context7` や `context7 で最新の Next.js 16 ドキュメントを参照して` のように書くと、Context7 が最新ドキュメントを取得して参照してくれる。

例:
```
use context7
Next.js 16 の App Router で並行ルートを使う最新の書き方を教えて
```

### Context7 を使うときの注意

- **必ず正確とは限らない**: Context7 が取得したドキュメントも、Claude が誤解釈する可能性はある。重要な箇所は公式ドキュメントで再確認する
- **ネットワーク遅延**: リアルタイム取得のため応答が遅くなる場合がある
- **CLAUDE.md との使い分け**: プロジェクト固有ルール（「このテンプレートではこう書く」）は CLAUDE.md、最新の汎用的な API 情報は Context7、という棲み分けが自然

### 本テンプレートとの親和性

本テンプレは AGENTS.md / CLAUDE.md で「Next.js 16 は破壊的変更あり」と警告し、`node_modules/next/dist/docs/` の参照を推奨している。Context7 はその**オンライン代替**として機能するため相性が良い。

## drawio MCP

### 何か

Draw.io の図を MCP 経由で作成・編集できるサーバー。XML 形式 / Mermaid 形式 / CSV 形式の図データを受け取り、ブラウザで draw.io エディタを起動する。`docs/diagrams/` の図の作成・更新に実績あり（本テンプレの全体アーキテクチャ図・開発フロー図等はこの MCP で作成した）。

### 向いている場面

- 設計書駆動開発で**図を頻繁に更新**する
- Mermaid の簡易構文では表現しきれない**複雑な図**（スイムレーン、ダッシュボード風の比較表等）を作成したい
- 図を Git 管理したいが、手書きは面倒な場合

### 向かない場面

- シンプルなシーケンス図・フローチャートのみ（Mermaid を md に埋め込む方が軽量）
- 図の作成・更新がほぼない運用

### ⚠️ 本テンプレートでの注意点: Next.js dev server とのポート競合

**drawio MCP サーバーは既定でポート 3000 を使用する。** Next.js の dev server (`npm run dev`) も同じポート 3000 を使うため、**両方を同時に起動できない**。

以下のいずれかで回避すること:

1. **個人用途として `~/.claude/settings.json` で設定** し、プロジェクトごとに有効/無効を切り替える
2. dev server を別ポートで起動（例: `npm run dev -- -p 3001`）
3. drawio MCP のポート設定を変更（`--http-port` 等のフラグで指定、バージョンによる）

**本テンプレの `.mcp.json` にはあえて入れていない。** 全利用者が Next.js 開発中に drawio MCP を起動したいとは限らないため、個人用途として各自追加するのが現実的。

### 前提条件

- Node.js / npx が利用可能なこと
- ポート 3000 / 3333 が空いていること（利用する場合）

### 追加手順（個人用途推奨）

`~/.claude/settings.json` に以下を追記:

```json
{
  "mcpServers": {
    "drawio": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "drawio-mcp-server"]
    }
  }
}
```

別パッケージ `@drawio/mcp` もあるため、動作するほうを選ぶ。

プロジェクト固有で追加したい場合は `.mcp.json` に書くが、ポート競合に注意すること。

### 利用できる機能

- `mcp__drawio__open_drawio_xml`: 手書き drawio XML を編集画面で開く
- `mcp__drawio__open_drawio_mermaid`: Mermaid 構文から drawio 図に変換して開く
- `mcp__drawio__open_drawio_csv`: CSV 形式から図を生成

## 検討したが採用しなかった MCP

将来の再検討のために、**検討プロセスと不採用の理由を記録**しておく。

### Cipher

#### 概要

[campfirein/cipher](https://github.com/campfirein/cipher) は、**セッションをまたいだ長期記憶**を提供する MCP サーバー。ユーザー・プロジェクトに紐づく記憶を蓄積し、過去のやり取り・意思決定・コンテキストを参照できる。ベクトルDB を使って類似検索も可能。

#### 検討した時期

本テンプレートの整備フェーズ（2026-04）で MCP 追加候補として評価した。

#### 評価した良い点

- ✅ セッションをまたいだ継続性が確保できる
- ✅ 「過去にこう議論した」「あの時はこう決めた」を Claude が参照できる
- ✅ 長期プロジェクトで効果を発揮する可能性
- ✅ ベクトル検索により、同じ議論の繰り返しを減らせる期待

#### 不採用の理由（詳細）

##### 理由1: 本テンプレートの「記憶はファイルに書き残す」思想とズレる

本テンプレートは、**意思決定・設計内容・進捗をすべてファイルに明文化する**方針を採っている:

| ファイル/ディレクトリ | 記録する内容 |
|---------------------|-------------|
| `docs/features/yyyymmdd_*.md` | 機能設計書（背景・目的・タスク進捗・評価計画） |
| `docs/設計書/` | 実態の一覧・定義（API一覧、テーブル定義書等） |
| `docs/reviews/` | コードレビュー・設計レビュー結果 |
| `docs/設計書/.doc-sync.md` | 設計書とコミットの同期記録 |
| 各設計書の改訂履歴テーブル | 何を・いつ・どのコミットで変更したか |

この方針の狙いは **「人間にも AI にも透明性がある形で管理する」** こと。Git で履歴が追え、他のメンバーが確認でき、プロジェクト消失時もファイルが残る。

Cipher の記憶層は**不透明（ブラックボックス）**であり、「なぜ Claude がこの判断をしたのか」を後から説明しにくくなる。

##### 理由2: 既存の仕組みと重複する

Claude Code には既に以下の記憶メカニズムがネイティブで存在:

- **`~/.claude/memory/`**: ユーザー・プロジェクト単位のメモリ（本テンプレも `auto memory` として活用）
- **セッション履歴**: コンテキスト圧縮で長期セッションも維持
- **CLAUDE.md**: プロジェクト固有ルールの永続化

これに加えて Cipher を導入すると、**記憶の保管場所が複数に分散**し、どこに何があるか分からなくなる。

##### 理由3: 導入コスト・運用コストが高い

- 別途サーバー運用が必要（Docker / DB）
- チームで共有するには共有サーバーの設置・メンテが必要
- ベクトル DB のバックアップ・リストア戦略が別途必要
- トークン消費も増える（記憶のコンテキスト挿入分）

本テンプレートのような「軽量に始められる」性格と合わない。

##### 理由4: 検証可能性が低い

「Cipher が正しく記憶しているか」「古い記憶が混乱を招いていないか」を検証する手段が限られる。一方、ファイル管理であれば `git diff` や `grep` で容易に検証できる。

#### 再検討の判断基準

以下の兆候が出てきたら再検討する:

- [ ] 「ファイルに書いてあるはずだが、どこに書いたか分からない」状態が頻発する
- [ ] 同じ議論・同じ判断を **3回以上** 繰り返している
- [ ] 機能設計書やレビュー履歴が数百件を超え、Claude が全体を把握できなくなっている
- [ ] チームメンバー間で「あの時こう決めた」の認識齟齬が継続的に起きている

再検討時は、このセクションを削除して上の「候補MCP」セクションへ移動する。その際は**ファイルベースの記憶（公式記録）**と **Cipher の動的な記憶（補助的な検索インデックス）** の役割分担を明確にすること。

### Chrome DevTools MCP

#### 概要

Chrome DevTools の機能（Performance タブ、Network タブ、Console、DOM 検査等）を MCP 経由で操作できる。Playwright MCP が「ブラウザ操作・テスト実行」に特化するのに対し、こちらは「実機デバッグ・パフォーマンス分析」に特化する。

#### 検討した時期

Playwright MCP 取り込み後、**ブラウザ関連 MCP の追加候補**として評価した（2026-04）。

#### 評価した良い点

- ✅ Performance プロファイリングを Claude に自動化させられる
- ✅ Network リクエストの解析（失敗原因の特定など）が容易
- ✅ Console エラーの検知・分析が自動化できる
- ✅ Playwright MCP との役割分担（テスト実行 vs 実機デバッグ）が明確

#### 不採用の理由

##### 理由1: Playwright MCP と機能が部分的に重複する

- Playwright MCP 経由でもコンソールログは取得できる
- ページスナップショット（DOM 状態）も Playwright MCP の機能に含まれる
- 追加で得られる情報（Performance、Network 詳細）は **Claude に自動判断させるより人間が DevTools を開く方が速い** ことが多い

##### 理由2: パフォーマンス改善は「意図的に」行うもの

本テンプレートの設計思想では、パフォーマンス改善は以下のいずれかのタイミングで意図的に行う:

- 新機能の設計フェーズで計測項目を「4. ブラウザ評価計画」に盛り込む
- `/code-review` でパフォーマンス観点のチェック
- 実際に問題が起きた時に手動で DevTools を開く

**常時ロードで自動デバッグさせる必要性は薄い**。

##### 理由3: トークン消費が増える

DevTools の情報は詳細で量が多い。コンテキストに常時ロードすると、軽微な作業でも**数千〜数万トークン増える**可能性がある。

#### 再検討の判断基準

- [ ] パフォーマンス課題が継続的に発生している
- [ ] ブラウザテスト（browser-tester）だけでは検出できないエラーが頻発する
- [ ] Performance 計測を **自動化する価値** が明確になった（例: 毎回の /browser-test で Core Web Vitals を記録したい）

### Sequential Thinking MCP

#### 概要

複雑なタスクを**段階的な思考ステップに分解**して Claude に推論させる枠組みを提供する MCP。各ステップで「今何を考えているか」を明示化し、最終的な結論の質を高めることを狙う。

#### 検討した時期

「複雑な設計判断の品質向上」の観点で評価した（2026-04）。

#### 評価した良い点

- ✅ 判断プロセスが可視化される
- ✅ 論理的な飛躍を減らせる可能性
- ✅ デバッグ時の原因特定で有用な場面がある

#### 不採用の理由

##### 理由1: 本テンプレートが既に「段階的思考」を強制している

本テンプレートは以下で段階的思考を組み込み済み:

| 段階 | 仕組み |
|------|-------|
| 設計フェーズ | `/new-feature` → 機能設計書のセクション別記入 |
| 設計レビュー | `/design-review` → 8項目の具体的チェック |
| 実装 | `coding-specialist` エージェントの `Phase 0〜3` プロセス |
| レビュー | `/code-review` → 重要度別指摘 |

**スキル + エージェント定義 + フックで思考フローを強制している**ため、MCP 層でさらに思考ステップを追加する必要性が薄い。

##### 理由2: 「考えるべきタイミング」は人間が決める方が効率的

Sequential Thinking は常時有効にしておくと、単純なタスクでも段階的思考が発動してトークンを消費する。一方、本テンプレは「複雑な判断が必要な時だけ `product-advisor` エージェントや `/design-review` を呼ぶ」構造なので、**判断タイミングが人間の手元にある** 方が効率的。

##### 理由3: トークン消費が大きい

思考ステップごとに中間出力が生成されるため、完了までのトークン消費が 2〜5 倍になることがある。

#### 再検討の判断基準

- [ ] Claude の推論品質が明らかに低下してきた（誤った結論が頻発）
- [ ] 複雑なアルゴリズム設計・数学的推論が頻繁に発生するプロジェクトになった
- [ ] 既存のスキル構造で「思考が浅い」と感じる場面が継続的にある

### Memory Bank MCP

#### 概要

Cipher の軽量版のような位置付け。ファイルベースで「プロジェクトの記憶（決定事項・設計方針・コンテキスト）」を管理する MCP。専用の memory bank ディレクトリを作り、セッションをまたいで参照する。

#### 検討した時期

Cipher 不採用の代替として評価した（2026-04）。

#### 評価した良い点

- ✅ Cipher と違ってサーバー不要・軽量
- ✅ ファイルベースなのでテンプレの思想と親和性が多少ある
- ✅ 構造化された形でプロジェクト記憶を保持できる

#### 不採用の理由

##### 理由1: 既存の機能設計書・設計書と役割が完全に重複する

本テンプレートでは以下が既に「プロジェクト記憶」として機能している:

- `docs/features/yyyymmdd_*.md`: 機能単位の設計・進捗・判断の記録
- `docs/設計書/`: API・DB・サービス等の実態

Memory Bank MCP を追加すると、**同じ役割のファイル群が2箇所に分散**する。どちらに書くべきか・どちらを正にするかで混乱する。

##### 理由2: 専用 MCP にするほどの複雑さがない

本テンプレートの記憶管理は「markdown ファイルへの読み書き」で完結する。標準の Read / Write / Edit で十分であり、MCP 層での抽象化は過剰。

##### 理由3: Git 管理との親和性

本テンプレのファイル群は Git 管理下にある。Memory Bank MCP 独自のファイル形式だと、PR レビューや履歴追跡の一貫性が崩れる可能性がある。

#### 再検討の判断基準

- [ ] 機能設計書や設計書では表現しづらい「暗黙知」が増えてきた
- [ ] 複数プロジェクトにまたがる記憶共有が必要になった
- [ ] 既存のファイル構造では管理しきれない情報量になった
