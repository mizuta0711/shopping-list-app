# 新規 Next.js プロジェクト作成手順

このテンプレート（`nextjs-claude-template`）を雛形に、新しい Next.js プロジェクトを WSL ネイティブ FS 上に作成するための手順書。**別セッションの Claude Code に作業を依頼する前提**で記述している。

## 想定する作業フロー

```
[あなた]                                  [Claude Code セッション]
   │                                            │
   │ 1. WSL ターミナルで ~/Project/Web へ移動      │
   │ 2. claude を起動（テンプレ外で起動するのが重要） │
   │                                            │
   ├──────プロンプト送信──────────────────────────▶│
   │                                            │
   │                                            ├─ テンプレを複製
   │                                            ├─ package.json などを書き換え
   │                                            ├─ 不要な依存を削除
   │                                            ├─ git init + 初回コミット
   │                                            ├─ gh repo create + push（任意）
   │                                            └─ 完了報告
   │◀──────完了報告──────────────────────────────┤
   │                                            │
   │ 3. VS Code で開いて Reopen in Container     │
   │ 4. npm run dev で起動確認                   │
```

## 前提条件

| 項目 | 確認方法 |
|------|---------|
| WSL2（Ubuntu）が動いている | `wsl -l -v`（PowerShell から） |
| WSL ネイティブ FS に作業ディレクトリがある | `ls -la ~/Project/Web/`（なければ `mkdir -p ~/Project/Web`） |
| テンプレが手元にある | `ls ~/Project/Web/nextjs-claude-template/` |
| Claude Code がログイン済み | `claude` 起動 → `/status` または `claude auth status` |
| `gh` がログイン済み（GitHub に push する場合） | `gh auth status` |
| Docker Desktop が起動・WSL2 統合 ON（Dev Container を使う場合） | `docker info` |
| VS Code 拡張 `ms-vscode-remote.remote-containers`（同上） | VS Code の拡張機能一覧 |

## ステップ 1: Claude Code セッションを起動する

**重要**: テンプレ自体（`~/Project/Web/nextjs-claude-template/`）の **外側** で起動する。テンプレ内で起動すると Claude が誤ってテンプレ自体を変更する事故につながる。

```bash
cd ~/Project/Web   # ← ここで起動。テンプレの中に入らない
claude
```

WSL ホストで素の `claude` を起動した場合、テンプレの `.claude/settings.json` の deny ルール（`/mnt/**`）は効かない（プロジェクト外なので）。Windows FS への誤操作を避けるため、`~/.claude/settings.json` にも同等の deny を入れておくことを推奨（[サンドボックス環境移行ガイド](./サンドボックス環境移行ガイド.md#claude-code-権限モード戦略wsl-ホスト-vs-コンテナ) 参照）。

## ステップ 2: プロンプトを送信する

下記プロンプトテンプレートを使う。`{{...}}` を埋めてから貼り付ける。

### 推奨プロンプト（フル版）

````markdown
~/Project/Web/nextjs-claude-template を雛形に、新しい Next.js プロジェクトを
WSL ネイティブ FS 上にセットアップしてほしい。日本語で進めて。

## 入力情報

- プロジェクト名（リポジトリ名）: {{例: my-todo-app}}
- 配置先: ~/Project/Web/{{プロジェクト名}}
- プロジェクトの目的（README に書く1行説明）: {{例: 個人タスク管理アプリ}}
- GitHub リポジトリ:
    - 既存URL: {{なし or git@github.com:foo/bar.git}}
    - 新規作成する: {{yes / no}}（yes の場合は gh repo create で作る）
    - 公開設定: {{public / private}}
- 使用するライブラリ（不要なら削除する）:
    - Prisma + PostgreSQL: {{使う / 使わない}}
    - NextAuth: {{使う / 使わない}}（使う場合のプロバイダ: {{Google / Credentials / GitHub など}}）
    - Zustand: {{使う / 使わない}}
    - Playwright（ブラウザテスト）: {{使う / 使わない}}
    - shadcn/ui: {{使う / 使わない}}
- フォワードしたいポート: {{例: 3000（DBを使うなら 5432 も）}}
- Dev Container を使うか: {{yes / no}}（推奨: yes）

## やってほしいこと

**責任範囲**: 雛形のコピーだけでなく、**Dev Container で Reopen in Container して
`npm run dev` が立ち上がるところまで** 設定を仕上げる責任を持つ。設定ファイルの
JSON 構文・postCreateCommand の各コマンド・必要拡張機能の整合性は事前にエージェント
側で検証する。ユーザーには最終的な起動操作だけを依頼する。

### Step 1: 事前確認
1. `pwd` を実行。`~/Project/Web` でなければ停止して案内する。
2. `~/Project/Web/{{プロジェクト名}}` が既に存在しないか確認。存在する場合は
   ユーザーに上書き可否を確認する（破壊的操作なので必ず確認）。
3. 雛形 `~/Project/Web/nextjs-claude-template/` の存在を確認。

### Step 2: テンプレの複製
1. `cp -r` または `rsync` で雛形を複製する。以下は除外:
   - `.git/` `node_modules/` `.next/` `.serena/` `package-lock.json`
   - `tools/dump.sql` `tools/backup/` `tests/browser-evidence/*/`
2. 引き継ぐべきもの（必須）:
   - `.claude/`（agents, skills, hooks, 各種ドキュメント）
   - `.devcontainer/devcontainer.json`
   - `.gitignore` `.mcp.json` `CLAUDE.md`
   - `docs/`（設計書テンプレート、guide 配下）
   - `tools/export-to-sql.ts`（DBを使う場合のみ）

### Step 3: プロジェクト固有の書き換え
1. `package.json` の `name` を {{プロジェクト名}} に変更。version は `0.1.0` にリセット。
2. `.devcontainer/devcontainer.json` の `name` を {{プロジェクト名}} に変更。
3. `README.md` を {{プロジェクト名}} 用に新規作成（旧テンプレ説明は削除し、
   1行説明・起動手順・Dev Container 手順だけ残す）。
4. `docs/設計書/*.md` のサンプル記述（テンプレに含まれる例）は残しつつ、
   トップに「初期テンプレートのまま。実装が進んだら更新する」とコメントを追加。

### Step 4: 不要な依存・コードの削除
入力情報で「使わない」と答えたものについて、以下を実施:
- Prisma を使わない場合: `prisma/`, `tools/export-to-sql.ts`,
  `.claude/hooks/pre-migrate-backup.js`, schema 関連の package.json 依存を削除。
  CLAUDE.md の「DB スキーマ変更時の必須ルール」セクションも削除。
- NextAuth を使わない場合: 関連依存・サンプルコードを削除。
- Zustand を使わない場合: 依存削除。
- Playwright を使わない場合: `.claude/skills/browser-test/`,
  `tests/browser-evidence/`, `.mcp.json` の playwright エントリ,
  `.devcontainer/devcontainer.json` の `--ipc=host` を削除。
- shadcn/ui を使わない場合: 関連依存削除。

### Step 5: Dev Container 設定の仕上げ（必須・スタックに合わせて整える）

雛形の `.devcontainer/devcontainer.json` をベースに、**選択したスタックで実際に
Reopen in Container して起動できる状態まで仕上げる**。コピーしただけで終わらせない。

具体的にやること:

1. **`name`** を `{{プロジェクト名}}` に変更。
2. **`runArgs`** の調整:
   - Playwright を使う場合: `["--ipc=host"]` を残す。
   - 使わない場合: 削除する。
3. **`mounts`** は触らない（`~/.claude` と `~/.claude.json` の bind mount は維持）。
4. **`postCreateCommand`** をスタックに合わせて組み立て直す。`&&` で連結:
   - 必須: `npm install`
   - 必須: `npm install -g @anthropic-ai/claude-code`
   - Prisma を使う場合: `npx prisma generate`
   - Playwright を使う場合: `npx -y playwright install --with-deps chromium && claude mcp add playwright npx '@playwright/mcp@latest'`
   - 例（Prisma + Playwright あり）:
     ```
     npm install && npx prisma generate && npm install -g @anthropic-ai/claude-code && npx -y playwright install --with-deps chromium && claude mcp add playwright npx '@playwright/mcp@latest'
     ```
5. **`forwardPorts`** と **`portsAttributes`** をユーザー入力のポートで更新:
   - Next.js は `3000` 固定で `requireLocalPort: true` を必ず付ける（NextAuth 対応）
   - DB を使う場合は 5432 等を追加
6. **`customizations.vscode.extensions`** の調整:
   - 必須: `anthropic.claude-code`
   - Playwright を使う場合: `ms-playwright.playwright` を追加
   - Prisma を使う場合: `Prisma.prisma` を追加
   - その他常時: `dbaeumer.vscode-eslint`, `esbenp.prettier-vscode`
7. **`customizations.vscode.settings`** はテンプレの値（`claudeCode.initialPermissionMode: bypassPermissions`,
   `terminal.integrated.defaultProfile.linux: bash`）を維持。
8. **`.mcp.json`** も整える:
   - Playwright を使わない場合は playwright エントリを削除（空なら `.mcp.json` 自体を削除）。

**設定の検証（必須）**:
- `cat .devcontainer/devcontainer.json | python3 -c "import json,re,sys; t=sys.stdin.read(); t=re.sub(r'//.*','',t); t=re.sub(r',(\s*[}\]])',r'\1',t); json.loads(t)"`
  でコメント混じり JSON が壊れていないか確認。
- `.env.example` を作って、必要な環境変数（`DATABASE_URL`, `NEXTAUTH_URL`, `NEXTAUTH_SECRET` 等）の
  プレースホルダを記載する。`.env.local` 自体はコミットしない。

**Dev Container の起動確認まで完遂する責任範囲**:
- 上記設定後、ユーザーに「VS Code で Reopen in Container して起動確認してほしい」と依頼する。
- ただし**設定ファイルの正しさはエージェント側で担保**する（JSON 構文、postCreateCommand の各コマンドが
  単独で動くか、必要な拡張機能が揃っているか）。
- 起動失敗が報告されたら、原因（権限・パス・コマンド）を特定して修正する。

### Step 6: Next.js 16 の初期化
雛形にはアプリ本体のソースがほぼ含まれていないため、必要に応じて Next.js の
最小構成（`src/app/layout.tsx`, `src/app/page.tsx`, `next.config.ts`,
`tsconfig.json`, `tailwind.config.ts`, `postcss.config.mjs`）を生成する。
- 既にテンプレに該当ファイルがあればそちらを優先し、上書きしない。
- ない場合は `create-next-app` ではなく、手動で最小ファイルを作る
  （余計な依存を増やさないため）。

### Step 7: 依存インストールとビルド確認
1. `npm install` を実行（時間がかかるため timeout に注意）。
2. `npm run build` で初回ビルドが通ることを確認。
3. 通らない場合は原因を特定して修正。2回失敗したら作業を止めて報告。

### Step 8: Git の初期化と初回コミット
1. `git init`
2. `git add .` の前に `git status` で意図しないファイル（.env, dump.sql 等）が
   含まれていないか確認。`.gitignore` で除外されているはず。
3. 初回コミット: `chore: initial commit from nextjs-claude-template`

### Step 9: GitHub リポジトリ作成（指定された場合）
- 新規作成 yes の場合: `gh repo create {{プロジェクト名}} --{{public/private}} --source=. --remote=origin --push`
- 既存URL 指定の場合: `git remote add origin {{URL}} && git push -u origin main`
- どちらも no の場合: スキップしてユーザーに案内のみ。

### Step 10: Dev Container の起動確認支援
ユーザーに以下の手順を案内し、起動失敗が報告されたら原因を特定して修正する。

1. VS Code を WSL から起動: `cd ~/Project/Web/{{プロジェクト名}} && code .`
2. コマンドパレット → "Dev Containers: Reopen in Container"
3. 初回ビルド完了後（数分）、コンテナ内ターミナルで以下を確認:
   - `claude /status` → ログイン済み（認証共有 OK）
   - `npm run dev` → http://localhost:{{ポート}} で Next.js トップページ表示
   - Prisma を使う場合: `npx prisma --version` がエラーにならない
   - Playwright を使う場合: `npx playwright --version` がエラーにならない、
     かつ `claude` 起動時に Playwright MCP が認識される

**起動失敗の典型パターンと対処（エージェントが自分で対処すること）**:
- `${localEnv:HOME}` が解決できない → ユーザーに「VS Code を WSL から起動」を再案内
- `postCreateCommand` のいずれかが失敗 → ログから失敗箇所を特定し、コマンド分解 or バージョン固定で対処
- ポート 3000 が他コンテナと衝突 → `requireLocalPort: true` が入っているか再確認
- `permission denied: /home/node/.claude` → `postCreateCommand` の先頭に
  `sudo chown -R node:node /home/node/.claude /home/node/.claude.json &&` を追加

## 制約

- **配置先は WSL ネイティブ FS のみ**。`/mnt/c` `/mnt/d` 等 Windows FS には絶対に置かない。
- **シークレットをコミットしない**。`.env*` は `.gitignore` 済みだが念のため `git status` で確認。
- **CLAUDE.md の規模判定**: この作業は M 規模相当として扱う（複数ファイル変更あり）。
  ただし設計書は不要、報告は所定フォーマットで。
- **破壊的操作（既存ディレクトリ上書き、git push --force 等）は事前確認**。
- **不明点は推測せず確認**。特に「使う/使わない」が曖昧な場合は止まって聞く。
- **2回失敗したら立ち止まる**。同じエラーで2回失敗したら作業継続せず状況を報告。

タイトルは「{{プロジェクト名}} プロジェクト作成 完了報告」とする。
````

## ステップ 3: 完了後の動作確認

Claude が完了報告を返したら、以下を順に確認する。

### 3.1 ファイル構成のスポットチェック

```bash
cd ~/Project/Web/{{プロジェクト名}}
ls -la                    # .claude/, .devcontainer/, docs/, src/ などが揃っているか
cat package.json | head   # name が正しいか
git log --oneline         # 初回コミットができているか
git status                # クリーンか
```

### 3.2 Dev Container で起動

```bash
code .   # VS Code を WSL から起動
```

VS Code 内で:
1. コマンドパレット → `Dev Containers: Reopen in Container`
2. ビルド完了を待つ（初回は数分）
3. ターミナルで `npm run dev`
4. ブラウザで http://localhost:3000 を開いて Next.js のトップページが表示されることを確認

### 3.3 Claude Code の認証共有確認

Dev Container 内のターミナルで:

```bash
claude
# 起動して /status を確認、ログイン済みなら成功
```

## ステップ 4: 開発開始

ここから先は通常の開発フロー。新規機能を作る時は:

```bash
claude
# Claude セッション内で:
/new-feature {{機能名}}
```

`new-feature` Skill が `docs/features/yyyymmdd_{{機能名}}.md` を生成し、
Stage 1（機能・画面設計）→ Stage 2（技術設計）→ 実装 → レビュー → コミット
の流れに沿って進む。詳細は [CLAUDE.md](../../CLAUDE.md) の「設計・実装フロー」を参照。

## トラブルシュート

| 症状 | 対処 |
|------|------|
| Claude が「`pwd` がテンプレ内」と言って止まる | 別ターミナルで `cd ~/Project/Web && claude` してやり直す |
| 既存ディレクトリと衝突した | Claude に上書き可否を伝えるか、別の名前を提案する |
| `npm install` が極端に遅い | 配置先が WSL ネイティブ FS か `pwd` で確認（`/home/...` ならOK、`/mnt/...` ならNG） |
| `gh repo create` で認証エラー | WSL ホスト側で `gh auth login` を実行してから再開 |
| Dev Container Reopen で「.claude が読めない」 | VS Code を WSL から開いていないと `${localEnv:HOME}` が Windows パスになる。一度閉じて WSL ターミナルから `code .` |
| 初回ビルドが失敗 | エラーメッセージを Claude に貼って原因特定を依頼。2回失敗したら `/clear` で仕切り直す |
| 雛形複製後に Prisma 関連が残っている（不要なのに） | プロンプト Step 4 の指定漏れ。Claude に「Prisma 関連を削除して再ビルド」と追加指示 |
