## はじめに

### バイブコーディングとは

AI エージェント（Claude Code 等）に指示を出しながら、対話的にコードを書き進める開発スタイル。人間が「なにを作りたいか」を伝え、AI が「どう書くか」を提案・実装する。

### 期待値調整（重要）

- AI は **下書きを高速に出す** のが得意。**最終承認は人間** が行う
- **1発で完璧なコード** は基本的に出ない。イテレーション前提
- AI は **「動くコード」** と **「正しいコード」** の区別が苦手。**動く＝正しい、ではない**
- AI は **忖度する**。「ユーザーが喜びそうな答え」を優先する傾向がある

---

## 3つの原則

本テンプレートは、以下の3原則をベースに設計されている。これを覚えておくだけでも失敗パターンの多くを回避できる。

### 原則1: AI 出力は draft、承認は人間

- AI が「完了しました」と報告しても、**必ず差分を目視確認** する
- ビルド・テストが通っても、論理の正しさは別途検証する
- 副作用（router.push / fetch / DB 操作）がある箇所は挙動を追跡確認

### 原則2: 仕組みで強制する

- CLAUDE.md にルールを書いても、**AI は後半を無視する** ことがある
- 致命的なルール（型チェック・DBバックアップ等）は **hook で物理的に強制** する
- 繰り返し必要なチェックは **skill** として手順化する

### 原則3: Why を設計書に残す

- AI に良い結果を出させる鍵は、**Why（なぜこれを作るか）** を明示すること
- 本テンプレは `/new-feature` で機能設計書を先に書かせる仕組みにしている
- 口頭で伝えるのではなく、**ファイルに書く** ことで AI も人間も参照できる

### A. セキュリティ・外部攻撃

#### 1. セキュリティ設定の勝手な削除

- **あるある**: AI がエラー解消のため、認証チェックやバリデーションを「邪魔者」として削除する。実害として、Tea社のデートアプリで AI 開発起因で72,000件の個人情報が露出した事例が報告されている
- **本テンプレでの対策**:
  - `/code-review` スキルで認証・認可・バリデーションのチェック項目
  - `.claude/01_development_docs/` に認証方針を明記し、coding-specialist が参照
- **あなたが意識すべきこと**:
  - 認証・認可・RLS 設定の削除・変更は AI に単独で触らせない（必ず人間レビュー）
  - エラー対処で「一旦コメントアウトして…」と言われたら警戒する

#### 2. ハードコードされたシークレット

- **あるある**: AI が API キー・パスワードを環境変数ではなくコードに直接埋め込む
- **本テンプレでの対策**:
  - `.gitignore` に `.env*` を含める設定済み
  - `/code-review` にセキュリティチェック項目
- **あなたが意識すべきこと**:
  - `.env.example` を用意して「環境変数で管理する」前提を AI に伝える
  - CI で `git-secrets` / `truffleHog` 等のシークレットスキャンを別途導入すると安心

#### 3. プロンプトインジェクション

- **あるある**: MCP 経由で取得した外部コード・ドキュメント・ページ内容に悪意ある命令が仕込まれ、AI がそれに従ってしまう。攻撃成功率は最大84%という研究結果もある
- **本テンプレでの対策**:
  - 標準 MCP は Playwright のみで、外部情報の取得経路が限定的
- **あなたが意識すべきこと**:
  - MCP ツールの権限は最小化する（特にファイル書き込み系 MCP は慎重に）
  - 外部から取得したコード/ドキュメントを設計書にそのまま貼らない（目視確認してから）
  - 公開サイトから読み込んだ内容を AI が「指示」として解釈する可能性を疑う

#### 4. SQLインジェクション・入力バリデーション漏れ

- **あるある**: raw SQL を使ったとき、ユーザー入力がそのまま結合される / API のリクエストボディにバリデーションがない
- **本テンプレでの対策**:
  - Prisma 前提のスタックで raw SQL が原則不要
  - `/code-review` で入力検証の有無をチェック
- **あなたが意識すべきこと**:
  - API 設計時に **バリデーション仕様を機能設計書に明記**（Zod 等の schema を先に決める）
  - raw SQL を使う場合は必ずパラメータバインドを使う

### B. 品質・実装

#### 5. ハルシネーション（存在しないAPI・古いバージョン）

- **あるある**: Next.js 15 の書き方で 16 向けコードを書く、存在しない関数を使う、非推奨API を提案する
- **本テンプレでの対策**:
  - 型チェックフック（`git commit` 前に `tsc --noEmit`）で存在しない API を検知
  - CLAUDE.md に「Next.js 16 は破壊的変更あり」明記、`node_modules/next/dist/docs/` 参照を推奨
  - オプションで Context7 MCP を追加可能
- **あなたが意識すべきこと**:
  - スタックのメジャーバージョン更新時は CLAUDE.md を見直す
  - 「そんな関数あったっけ？」と感じたら実際にドキュメントを確認する

#### 6. 「動く=正しい」と思い込む / サイレントデグレード

- **あるある**: ビルド・テストが通るから OK と判断してしまう。AI が「修正完了」と報告しても実際には直っていない、または別の機能を壊している。エラーが出ないので気づかない
- **本テンプレでの対策**:
  - `/browser-test` で実機確認（browser-tester + Playwright MCP）
  - `/done` で「ビルド確認」と「ブラウザテスト」を分けて報告
- **あなたが意識すべきこと**:
  - エッジケース・データ状態パターンを設計段階で網羅する（設計書「5. ブラウザ評価計画」）
  - 「動いた」と AI が言っても、主要な動作を自分の目で確認する

#### 7. コンテキストウィンドウ崩壊（既存コンポーネント重複実装）

- **あるある**: コードが増えると AI が全体を把握できず、既にあるコンポーネントを無視して似たものを新規実装する
- **本テンプレでの対策**:
  - `docs/設計書/サービス・リポジトリ一覧.md` / `フック一覧.md` を先に読ませる「設計書優先の原則」
  - `/sync-check` で設計書と実装の乖離を検出
- **あなたが意識すべきこと**:
  - 新機能の前に「類似機能が既にないか」を AI に確認させる
  - Serena MCP（別紙）を導入して既存シンボルの検索を高速化する選択肢もある

#### 8. スコープクリープ / 過剰実装

- **あるある**: 指示していないことまで実装する、「将来のために」と抽象化を増やす、1 回の依頼で大量のファイルを変更する
- **本テンプレでの対策**:
  - 機能設計書のタスク一覧 + 「作成・変更ファイル」セクションで事前にスコープ確定
  - CLAUDE.md の「作業の進め方」に「一度に編集するファイルは最大5ファイル」ルール
- **あなたが意識すべきこと**:
  - スコープ外の発見は **新規の機能設計書として別管理** し、その場では実装しない
  - 「ついでに直しました」系の変更があったら PR で分離する

#### 9. 命名インフレ・コピペ痕

- **あるある**: 意味のない長い変数名、似た関数の乱立、コピー元の変数名がそのまま残る
- **本テンプレでの対策**:
  - `/code-review` の品質チェック項目
- **あなたが意識すべきこと**:
  - 既存コードのスタイルに合わせるよう明示的に指示する（「src/features/X を参考にして」）

---

### C. 修正・デバッグループ

#### 10. 思い付き修正のループ

- **あるある**: 原因を特定しないまま「たぶんこれかも」で修正を繰り返し、コードがぐちゃぐちゃになる
- **本テンプレでの対策**:
  - CLAUDE.md の「バグ修正の原則」に **原因特定前の修正禁止** を明記
- **あなたが意識すべきこと**:
  - 「なぜそのバグが起きたと考えたか」を AI に先に言わせる
  - ログ追加や最小再現コードで原因を確定してから修正依頼する

#### 11. doom loop（同じバグの反復失敗）

- **あるある**: 同じバグの修正を何度も試みるが直らない。AI が思い込みで失敗を重ねる
- **本テンプレでの対策**:
  - CLAUDE.md の「バグ修正の原則」に **2回失敗したら `/clear`** のルール
- **あなたが意識すべきこと**:
  - 2回失敗したら立ち止まる。セッションを `/clear` してリセット
  - 最小再現コードに絞って仕切り直す
  - 指示の仕方自体に問題がないか見直す（「再現手順を伝えていない」等）

#### 12. AI バグ修正スパイラル（未検証修正の積み重ね）

- **あるある**: 未検証のフィックスが積み重なり、根本原因が追跡不能になる。「AIバグをAIで修正する」悪循環
- **本テンプレでの対策**:
  - CLAUDE.md の「バグ修正の原則」に **人間による原因分析** を明記
- **あなたが意識すべきこと**:
  - バグの原因分析は人間が行う。「なぜ起きたか」を理解してから修正指示を出す
  - AI が複数の仮説修正を連続で出してきたら、一度すべてロールバック（`git reset`）して仕切り直す

#### 13. エラーメッセージを完全に読まない

- **あるある**: エラー全文を見ずに「TypeError」「undefined」といったキーワードだけで修正を試みる
- **本テンプレでの対策**:
  - 型チェックフックがエラーの先頭15行を表示（`git commit` 時）
- **あなたが意識すべきこと**:
  - エラー全文を AI に渡す。切り詰めない
  - スタックトレースの「どの関数がどこから呼ばれたか」まで確認する

---

### D. 人間側の問題（仕組みで防ぎにくい・重要）

#### 14. 過信バイアス

- **あるある**: 「AI を使うと速い」という主観を持つが、実測は遅いことがある
  - Stanford 研究: AI を使った開発者ほど安全だと錯覚し、実際は脆弱性が増加
  - METR 研究: 開発者が「24% 速くなった」と予想 → 実測は「19% 遅かった」
- **本テンプレでの対策**:
  - なし（人間の認知バイアスは仕組みで防げない）
- **あなたが意識すべきこと**:
  - 「速くなった感覚」を KPI にしない
  - バグ率・脆弱性数・レビュー指摘数など **客観指標** で評価する
  - 体感が良くても、PR のサイズや修正回数などで振り返る

#### 15. 忖度への誘導

- **あるある**: 「これで良いですか？」と聞くと AI は「はい、良いですね」と返しがち
- **本テンプレでの対策**:
  - `product-advisor` エージェントが批判的思考をデフォルトにする設計
  - L規模の Stage 1 レビュー時に `/design-review feature` から自動起動されるため、形骸化せず必ず使われる
- **あなたが意識すべきこと**:
  - 「プロダクト品質を上げる意見を出して」「問題点を洗い出して」と明示的に依頼する
  - 肯定的な返事が続いたら、逆に「反対意見は？」と聞き直す

#### 16. 指摘を何でも修正してしまう

- **あるある**: レビュー結果を見て AI がすべて「修正済み」にしてしまい、本来は議論すべき指摘まで反映される
- **本テンプレでの対策**:
  - CLAUDE.md「サブエージェント運用ルール」で「必ずメインで差分確認・フロー検証」を明記
  - `/code-review` が重要度別（重要/中/低）に分けるので取捨選択しやすい
- **あなたが意識すべきこと**:
  - レビュー指摘は **妥当性を判断してから反映** する
  - 「レビューを反映して」ではなく「レビューを読んで対応方針を提案して」と依頼する

#### 17. 矛盾に気づけない

- **あるある**: 仕様やコード内で矛盾があっても AI は指示通り淡々と書く
- **本テンプレでの対策**:
  - 型チェックフック（機械的な矛盾検知）
  - `/sync-check`（設計書と実装の矛盾）
  - `/code-review` / `/design-review`（仕様とコードの突き合わせ）
- **あなたが意識すべきこと**:
  - **仕様レベルの矛盾（本質的な正しさ）** はユーザーが気づくしかない
  - 「これまでの仕様と矛盾していないか確認して」と明示依頼する習慣をつける

#### 18. コグニティブデット（幽霊コードベース）

- **あるある**: AI 生成コードが増え、チームの誰もシステム全体を理解していない状態になる。技術的負債と違い、リファクタリングで解消できない
- **本テンプレでの対策**:
  - `docs/設計書/` の実態一覧（API / サービス / フック）で全体像を可視化
  - 機能設計書に「背景・目的・方針」を残す
- **あなたが意識すべきこと**:
  - 重要なロジックは **人間が説明できるか** を確認する
  - 「なぜこの設計にしたか」を機能設計書の「方針」や改訂履歴に残す習慣
  - PR レビューで「自分の言葉で説明」できないコードはマージしない

#### 19. スキル萎縮

- **あるある**: AI に任せきりにすると、2〜3 年で専門知識が低下し、いざという時に自分で問題解決できなくなる
- **本テンプレでの対策**:
  - なし（個人のスキル維持は仕組みで強制できない）
- **あなたが意識すべきこと**:
  - 重要ロジックの **設計・レビューは人間が担当する範囲** を意図的に維持する
  - 生成コードを「読む」習慣を失わない
  - 新しい技術は最初の1回は自分で書く（AI に任せず学ぶ）

#### 20. コスト・セッション運用の放置

- **あるある**: コンテキストが膨らんだまま長期セッションを続け、1タスクで数十ドル規模のトークンを消費する。AI 生成コードの I/O は人間の8倍という研究結果もある
- **本テンプレでの対策**:
  - CLAUDE.md「作業の進め方」で「大きなフェーズは細分化」「最大5ファイル編集」
  - サブエージェントは sonnet 指定（コスト最適化）
- **あなたが意識すべきこと**:
  - セッションを小さく区切る（1機能＝1セッション目安）
  - タスク完了後や話題変更時は `/clear`
  - 長時間セッションでは後半の品質が落ちやすい、と認識する

## セッション運用の工夫

### `/clear` の使い所

- **タスクが完了したとき**: 次のタスク開始前にコンテキストを綺麗にする
- **話題が大きく変わるとき**: 無関係なファイルが読まれた状態を避ける
- **2回失敗したとき** (doom loop 対処): 思い込みを断ち切る
- **長時間セッションの後**: 集中力が落ちた AI をリフレッシュ

### M/L フローでタスクを区切る

- 1機能 = 1設計書 = 1セッション の単位を意識する
- 機能設計書が `completed/` に移ったらその機能は「閉じる」
- 新機能を始めるときは新しいセッションから

### 長時間作業のコツ

- フェーズ単位でコミット。後戻りできる地点を作る
- 1フェーズが終わったら `/done` で結果をレポート
- 次フェーズに進む前に `/clear` するかを検討

## プロンプト技術（ユーザースキル）

### 設計書を先に書く → プロンプトが具体化する

本テンプレートの `/new-feature` で機能設計書を先に書くと、以下が自動で具体化される:

- **Why**: 背景・目的・方針
- **What**: タスク一覧・作成/変更ファイル
- **How**: 設計詳細・受け入れ条件
- **検証**: ブラウザ評価計画

これにより、AI への指示が「`docs/features/YYYYMMDD_X.md` を読んで Phase 1 を実装して」という簡潔で精度の高いものになる。

### 副作用を伴う指示は追跡先まで明示

- ❌ 「ユーザーを削除する機能を追加して」
- ⭕ 「ユーザーを削除する機能を追加して。`router.push('/users')` 後のリダイレクト先、関連テーブル（セッション・通知）の削除連鎖、削除権限チェックまで確認して」

### 2回失敗したら指示自体を見直す

- 同じ指示を3回繰り返しても結果が良くならないなら、**指示の粒度や前提情報の渡し方** に問題がある
- 指示を小さく分割する、前提情報を事前にファイル化する、具体例を示す等を試す

### 否定形より肯定形で指示する

- ❌ 「any を使わないで」
- ⭕ 「型を明示して（any は禁止）。不明なら unknown + type guard で」

| 失敗パターン | 主な解決手段 |
|-------------|-------------|
| ルール違反 | **hook**（型チェック、DBバックアップ） |
| 繰り返しの品質チェック | **skill**（`/build-check`, `/code-review`, `/browser-test`） |
| 設計書と実装の乖離 | **skill**（`/update-docs`, `/sync-check`, `/pre-push-check`） |

| 考慮漏れ（技術整合性） | **skill + agent**（`/design-review tech` + code-reviewer） |
| 企画・UX 観点の見落とし | **skill + agent**（`/design-review feature` + product-advisor） |
| 実機動作の検証 | **MCP + agent**（Playwright MCP + browser-tester） |
| 忖度・批判的視点 | **agent**（product-advisor、Stage 1 レビューに自動統合） |
| 規模判定のバイアス | **プロセス**（AI推測→ユーザー承認 + 「迷ったらL寄り」） |
| 設計方針の統一 | **ドキュメント**（`.claude/01_development_docs/`） |
| 機能スコープの固定 | **ドキュメント**（機能設計書 `docs/features/` の Stage 1） |
| 過信バイアス | **人間の責任**（仕組みで防げない） |
| スキル萎縮 | **人間の責任**（仕組みで防げない） |
| コグニティブデット | **ドキュメント + 人間の責任**（機能設計書の「方針」+ 改訂履歴で判断経緯を残す） |

## 初心者向けチェックリスト

## おわりに

バイブコーディングは「AI に全部任せる」ことではなく、**AI が最大限パフォーマンスを発揮できる仕組みを整えたうえで、人間が判断・検証する** ことで成立します。

本テンプレートは多くの失敗パターンを仕組みで対策していますが、**最終的な品質責任は人間にある** という前提は変わりません。

- 仕組みで強制できるもの → テンプレートに任せる
- 仕組みで防ぎにくいもの（D 領域）→ 人間が意識を持って対応する

この棲み分けを意識しながら、AI との協業を進めてください。
