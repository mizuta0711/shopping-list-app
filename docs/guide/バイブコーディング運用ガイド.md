# Claude Code によるバイブコーディング運用ガイド

本テンプレートで実践している Claude Code 活用のノウハウをまとめる。
実プロジェクト（Next.js + Prisma + TailwindCSS）での運用を通じて得た知見に基づく。

## 2. 開発フローの全体像

### 2-1. 変更規模別フロー（S / M / L）

変更の規模に応じて開発フローを選択する。**判定は AI 推測 → ユーザー承認**。
機能開発・改修・バグ修正すべてに適用される。

| 規模 | 対象 | フロー |
|------|------|--------|
| **S** | UI微調整、テキスト修正、1ファイル程度、軽微なバグ修正 | 実装 → `/build-check` → コミット → `/done` |
| **M** | 機能追加、API変更、複数ファイル変更（**UX変更なし**） | 設計 → 実装 → `/code-review` → `/browser-test` → `/build-check` → `/update-docs` → コミット → `/done` → プッシュ |
| **L** | 新機能、大規模変更、DBスキーマ変更、**UX変更あり** | **2段階設計**: Stage 1 → `/design-review feature` → Stage 2 → `/design-review tech` → 実装 → 以降 M と同じ |

**重要な判定ルール**:
- **UX変更を含む場合は自動的に L 扱い**（2段階設計が必要なため）
- 判定に迷ったら **L 寄りで提示する**（過小見積もりより安全）
- スコープ拡大に気づいたら現フェーズ完了後に再判定。破壊的変更は即停止
- 詳細は [.claude/01_development_docs/09_開発フローと規模判定.md](../../.claude/01_development_docs/09_開発フローと規模判定.md) 参照

**プッシュ前は共通で** `/pre-push-check` → プッシュ。

### 2-2. L 規模の2段階設計フロー

```
┌────────────────────────────────────────────────────────────┐
│ 1-A. Stage 1 設計フェーズ（機能・画面設計）                │
│   /new-feature 機能名                                       │
│   → 対象画面・操作フロー・受け入れ条件を記入                │
│   → 5. ブラウザ評価計画も記入                              │
│   /design-review feature                                    │
│     (code-reviewer: 設計整合性 + UI/UX設計)                 │
│     (product-advisor: 企画・UX体験)                         │
│   → ユーザー承認                                            │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 1-B. Stage 2 設計フェーズ（技術設計）                      │
│   API仕様・テーブル構造・サービス/フック設計を記入         │
│   /design-review tech                                       │
│     (code-reviewer: 技術的整合性)                           │
│   → ユーザー承認                                            │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 2. 実装フェーズ                                            │
│   実装 (coding-specialist or メイン直接)                    │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 3. 品質チェックフェーズ                                    │
│   /code-review → /browser-test → /build-check               │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 4. ドキュメント同期フェーズ                                │
│   /update-docs → (全タスク完了なら /sync-check 自動実行)    │
│   → 全タスク完了なら completed/ へ自動移動                 │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 5. コミットフェーズ                                        │
│   git commit (型チェック + 同期通知フックが自動発火)        │
│   /done                                                     │
└─────────────────────┬──────────────────────────────────────┘
                      ↓
┌────────────────────────────────────────────────────────────┐
│ 6. プッシュフェーズ                                        │
│   /pre-push-check → git push                                │
└────────────────────────────────────────────────────────────┘
```

**M 規模（UX変更なし）の場合**: Stage 1 はスキップし、Stage 2（技術設計）から開始する。
**バグ修正の場合**: 原因特定 → 規模判定 → その規模のフローを適用（S/M/L）。詳細は [09_開発フローと規模判定.md](../../.claude/01_development_docs/09_開発フローと規模判定.md) §3 参照。

## 3. エージェントの役割

Claude Code の **サブエージェント** は、特定の役割に特化した Claude インスタンスで、メインエージェントから委譲される。`.claude/agents/` にMarkdownファイルで定義する。

### 3-1. coding-specialist（実装担当）

設計書に基づいた実装、バグ修正、リファクタリングを担当する。`CLAUDE.md` のコーディングルール（any禁止、memo<Props>パターン、Service層経由等）に従って実装する。

**使用タイミング**: 明確な設計書がある状態での実装作業。

### 3-2. code-reviewer（レビュー担当）

以下のモードを持つ（Phase 3 で拡張）:

- **Stage 1 設計レビュー**（`/design-review feature`）: 設計書整合性 + **UI/UX 設計観点**（デザインシステム準拠、アクセシビリティ、レイアウト、レスポンシブ等）
- **Stage 2 設計レビュー**（`/design-review tech`）: API・DB・サービス・フックの技術的整合性
- **設計レビュー（互換モード）**（`/design-review` 引数なし）: Stage 分離されていない従来形式の設計書向け
- **実装レビュー**（`/code-review`）: コード品質・設計書突き合わせ・セキュリティ・パフォーマンス

Stage 1 では **UX 体験観点は product-advisor が担当**するため、code-reviewer は扱わない（役割分担）。

**使用タイミング**: `/design-review` スキルと `/code-review` スキルから起動される。

### 3-3. documentation-manager（ドキュメント管理）

設計書の更新・整合性維持を担当。「方針（`.claude/` 配下）」と「実態（`docs/設計書/`）」を分離し、二重管理を防ぐ。

**使用タイミング**: `/update-docs` スキルから起動される。

### 3-4. product-advisor（企画・UXアドバイザー）

技術実装ではなく、企画・UX・ビジネス観点からの助言を行う。「何を作るべきか」「なぜ作るべきか」に集中する。

**Phase 3 で `/design-review feature` に統合**:
- L 規模の Stage 1 設計レビュー時に、code-reviewer と並列で自動起動される
- レビュー観点: 課題-解決策フィット、ユーザーフローの自然さ、摩擦ポイント、代替案、優先度の妥当性、既存機能との整合性

**使用タイミング**:
- `/design-review feature` から自動起動（Stage 1 レビュー）
- 企画単体のレビュー、MVPの課題分析、サービス改善提案、競合分析などをユーザーが直接依頼する場合

### 3-5. browser-tester（実機ブラウザテスト）

Playwright MCP を使って実際にブラウザを操作し、機能テスト・UX評価を行う。スクリーンショットを撮影し、`tests/browser-evidence/` にレポートを保存する。

**使用タイミング**: `/browser-test` スキルから起動される。UI変更の動作確認に使う。

### 3-6. エージェント運用の原則

- **原則 sonnet**: 各エージェントは `model: sonnet` で起動。コスト削減のため
- **Opus は例外**: 複雑な設計判断、アーキテクチャ決定時のみ
- **結果は必ず検証**: サブエージェントが「完了」と報告しても、メインで差分確認 + フロー検証してからコミットする
- **BE/FE分離時は型契約を先**: 別エージェントに委譲する前に、`src/types/` に共有型を定義する

## 4. スキルの役割

Claude Code の **カスタムスキル** は、スラッシュコマンドで呼び出す再利用可能な手順書。`.claude/skills/{name}/SKILL.md` に定義する。

### 4-1. 設計フェーズのスキル

| スキル | 用途 | 実行タイミング |
|--------|------|---------------|
| `/new-feature 機能名` | 機能設計書をテンプレートから新規作成。変更の性質・規模をユーザー承認 | 機能開発の最初 |
| `/design-review feature` | Stage 1 設計レビュー（code-reviewer + product-advisor 並列） | L規模の Stage 1 完了時 |
| `/design-review tech` | Stage 2 設計レビュー（code-reviewer のみ） | M/L規模の Stage 2 完了時 |
| `/design-review`（引数なし） | 互換モード。Stage 分離されていない従来形式の設計書向け | 旧形式の設計書レビュー |

### 4-2. 実装後の品質チェックスキル

| スキル | 用途 | 実行タイミング |
|--------|------|---------------|
| `/code-review [対象]` | コード品質・API設計書との突き合わせ。重要指摘は自動修正 | M/L規模の実装後 |
| `/browser-test [URL\|full]` | browser-tester を起動してブラウザテスト実施 | UI変更を含む場合 |
| `/build-check` | `npm run build` + `npm run lint` を一括実行 | 全規模のコミット前 |

### 4-3. ドキュメント同期スキル

| スキル | 用途 | 実行タイミング |
|--------|------|---------------|
| `/update-docs` | 変更駆動で設計書を更新。タスクステータスも自動更新 | M/L規模のコミット前 |
| `/sync-check` | 全量照合。実装と設計書を網羅的に突き合わせ | `/update-docs` で自動実行 |
| `/complete-feature` | 機能設計書の完了処理（手動） | `/update-docs` の自動処理で不十分な時 |
| `/pre-push-check` | 未プッシュコミットが全て `.doc-sync.md` に記録されているか確認 | 毎push前 |

### 4-4. 運用支援スキル

| スキル | 用途 | 実行タイミング |
|--------|------|---------------|
| `/done` | 作業完了報告を規定フォーマットで出力 | 全作業の最後 |

## 5. フックによる自動化

Claude Code の **フック** は、ツール実行の前後で自動的にコマンドを実行する仕組み。`.claude/settings.json` に定義する。

本テンプレートでは以下の3つのフックを設定している。

### 5-1. DBバックアップフック（PreToolUse）

**発火タイミング**: `npx prisma migrate` 実行前

**動作**: `npx tsx tools/export-to-sql.ts` を実行してDB全量バックアップを取得。失敗した場合は migrate をブロックする。

**目的**: DBスキーマ変更前に必ずバックアップを取り、誤操作によるデータ消失を防ぐ。

### 5-2. 型チェックフック（PreToolUse）

**発火タイミング**: `git commit` 実行前

**動作**: `npx tsc --noEmit --pretty false` を実行。型エラーがあればコミットをブロックし、エラー内容（最大15行）を表示する。

**目的**: 型エラーがあるコードをコミットするのを物理的に防ぐ。ビルドチェックより軽量（トランスパイルしないため）で高速。

### 5-3. 設計書同期通知フック（PostToolUse）

**発火タイミング**: `git commit` 実行後

**動作**: `.claude/hooks/post-commit-doc-check.js` を実行。直近コミットの変更ファイルをチェックし、以下に該当する変更があれば Claude に通知:
- `src/app/api/` → API一覧
- `src/lib/services/` → サービス一覧
- `prisma/schema.prisma` → テーブル定義書
- `src/lib/ai/prompts/` → ai-prompt-design
- `src/features/*/hooks/` → フック一覧

**目的**: M/L規模のコミット後に `/update-docs` の実行を促す。設計書との同期漏れを検知する。

### 5-4. フック設計の考え方

- **必須ルールはフックで強制**: 忘れると致命的な処理（DBバックアップ、型チェック）は、ルールとして書くだけでなくフックで自動実行する
- **軽微なルールは通知のみ**: 設計書同期は通知にとどめ、ブロックしない。柔軟性を残すため
- **ブロックする場合は理由を明示**: `stopReason` にエラー内容を含め、ユーザーが何をすべきか分かるようにする

## 6. MCP（Model Context Protocol）の活用

**MCP** は Claude Code が外部ツール・サービスと連携するためのプロトコル。`.mcp.json` にサーバー設定を記述する。

### 6-1. Playwright MCP

本テンプレートでは **Playwright MCP** を使ってブラウザ操作を実現している。

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}
```

**特徴**:
- `npx @playwright/mcp@latest` で起動するため、`package.json` に Playwright を追加不要
- Claude Code がページのアクセシビリティツリーを取得してブラウザを操作する
- スクリーンショット撮影、フォーム入力、クリック、ページ遷移などを自然言語で指示できる

**用途**: `browser-tester` エージェントが Playwright MCP を介してブラウザを操作し、実機テストとUX評価を行う。

### 6-2. MCP の追加について

プロジェクトに応じて他の MCP サーバー（Slack、GitHub、Notion、Serena 等）を `.mcp.json` に追加できる。ただし:

- MCP サーバーを追加するとコンテキストに該当ツールが常時ロードされ、トークン消費が増える
- 使わない MCP は外す
- プロジェクト固有で使うものは `.mcp.json` に、個人的に使うものは `~/.claude/settings.json` に分けて設定する

**📄 別紙**: 具体的な追加手順と候補MCP（Serena 等）は [オプションMCP追加ガイド.md](./オプションMCP追加ガイド.md) を参照。

## 7. 機能設計書の運用

### 7-1. 設計書の構造（Stage 1/2 対応）

`/new-feature 機能名` で作成される機能設計書は以下の構成（Phase 2 で Stage 1/2 構造に変更）:

```
メタ情報
├── 全体ステータス（🔵設計中 → 🟡実装中 → 🟢完了）
├── 変更の性質（新規追加 / 改修 / バグ修正）
├── 変更規模（S / M / L）
├── Stage 1 ステータス（🔵/🟡/✅/⚪不要）
└── Stage 2 ステータス（🔵/🟡/✅/⚪未着手）

1. 概要（背景・目的・方針）
2. タスク一覧（🔵/🟡/✅/⏸️/❌）
3. 機能・画面設計（Stage 1）         ← L 規模で記入。M（UX変更なし）は ⚪不要
   ├── 対象画面と操作フロー
   ├── 受け入れ条件（ユーザー目線）
   └── Stage 1 レビュー記録
4. 技術設計（Stage 2）               ← M/L 規模で記入
   ├── 作成・変更ファイル
   ├── API 仕様（JSON 例必須）
   ├── テーブル構造 / スキーマ変更
   ├── サービス・フック設計
   └── Stage 2 レビュー記録
5. ブラウザ評価計画（UI 変更ありの場合）
6. 関連ドキュメント
7. 注意事項
改訂履歴
```

### 7-2. タスクステータスの自動更新

`/update-docs` 実行時、実装済みのタスクを検知してステータスを `✅完了` に自動更新する。全タスクが完了（✅/⏸️/❌）になると、`/sync-check` → `completed/` 移動が自動で実行される。

### 7-3. 「5. ブラウザ評価計画」の位置づけ

設計段階で「何をどう評価するか」を決めておく。`/design-review feature`（L規模）または互換モードでこの評価計画自体がレビュー対象となる。未記入のまま実行した場合、設計書の内容から草案が自動生成される。

## 8. 設計書駆動開発とドキュメント同期

### 8-1. 設計書の役割分離

実運用で最も発生しやすい問題が「同じ情報を2箇所に書いて片方だけ更新する」パターン。これを防ぐため、ドキュメントの役割を明確に分離している。

| 場所 | 役割 | 例 | 変更頻度 |
|------|------|-----|----------|
| `.claude/01_development_docs/` | **設計方針・パターン・ルール** | 「API命名規則はケバブケース」「ページネーションはcursor方式」 | 低 |
| `.claude/02_design_system/` | **デザインシステム方針** | 「ボタンパターン」「レイアウト方針」 | 低 |
| `.claude/03_library_docs/` | **ライブラリ利用ガイド** | 「Next.js 16 の非推奨API」 | 低 |
| `docs/設計書/` | **実態の一覧・定義** | 「GET /api/v1/users のレスポンスは {...}」 | 高（コード同期） |
| `docs/features/` | **機能設計書** | 「20260418_user-profile.md」 | 高 |
| `docs/reviews/` | **レビュー結果** | コード/設計レビューのログ | 高 |

### 8-2. 同期の3階層

実装と設計書の同期は3階層で担保する:

1. **変更駆動同期（`/update-docs`）**: コミット前に、変更ファイルに対応する設計書を更新。差分ベースで高速
2. **全量照合（`/sync-check`）**: 実装と設計書を網羅的に突き合わせ、「設計書に記載がない実装」を検出。`/update-docs` で全タスク完了時に自動実行
3. **プッシュ前検査（`/pre-push-check`）**: 未プッシュコミットが全て `.doc-sync.md` に記録済みか grep で検査。push前の最後の砦

### 8-3. テーブル定義書の自動生成

`prisma/schema.prisma` の `///` コメントから `docs/設計書/テーブル定義書.md` を自動生成する（`tools/scripts/generate-table-docs.ts`）。手動でテーブル定義書を書くと乖離が最も発生しやすいため、自動生成に切り替えた。

## 9. エージェント定義の最適化

### 9-1. 共通ルールの一元化

当初、各エージェント定義に同じコーディングルール（`any`禁止、`memo<Props>`パターン等）を重複して書いていたが、以下の問題があった:
- **トークンコスト増大**: エージェント起動時に全文が読み込まれる
- **ルール変更時の更新漏れ**: 1箇所だけ直して他を忘れる

解決策として、共通ルールを `CLAUDE.md` の「コーディングルール（全エージェント共通）」に集約し、各エージェント定義からは参照する形にした。

### 9-2. エージェント定義は短く（50行前後）

含めるのは:
- 役割の1行説明
- 参照ドキュメントの対応表
- そのエージェント固有のプロセス
- 完了報告の必須事項

含めないのは:
- TypeScriptコード例（良い例/悪い例）
- mermaid図
- 重複するチェックリスト

結果: 合計約1,600行 → 約160行（90%削減）。

### 9-3. PMエージェントは不要

当初 project-manager エージェントを「司令塔」として設置していたが、以下の理由で削除した:
- PM→サブエージェントの中間委任でトークンコストが2倍になる
- コンテキストの受け渡しが不正確になる
- ユーザーが直接 coding-specialist を呼ぶ方が速く正確

タスク分解が必要な場合は Claude Code 組み込みの Plan 機能で十分。

## 10. 実運用で踏んだ落とし穴と対策

### 落とし穴1: 「省略禁止」は守られない

**問題**: 全変更に9ステップ必須のフローを定義し「省略禁止」と明記していたが、1行のバグ修正でも全ステップを実行するのは現実的でなく、次第に無視された。

**対策**: S/M/L の3段階フローに変更。Sは「実装→ビルド→コミット」の3ステップのみ。

### 落とし穴2: 設計書と実装の乖離

**問題**: `/update-docs` を毎回実行していても、「そもそも設計書に記載がない」実装は検出できない。変更駆動の同期だけでは不十分だった。

**対策**: 全量照合の `/sync-check` を導入し、`/update-docs` の全タスク完了時に自動実行する仕組みにした。

### 落とし穴3: エージェント定義の肥大化

**問題**: 各エージェント定義に詳細なTypeScriptコード例、mermaid図、チェックリストを書いた結果、1ファイル500行超になり、エージェント起動のたびに大量のトークンを消費していた。

**対策**: 共通ルールをCLAUDE.mdに一元化し、各エージェント定義を50行前後に圧縮。

### 落とし穴4: サブエージェントの品質を過信

**問題**: サブエージェントが「実装完了」と報告した内容をそのままコミットしたところ、ビルドは通るが実際の動作が設計と異なるケースがあった。

**対策**: 「ビルド成功 ≠ 正しい実装」をルール化。サブエージェントの結果はメインで必ず差分確認・フロー検証する。BE/FEを別サブエージェントに委譲する場合は事前に共有型定義を作成する。

### 落とし穴5: ドキュメントの二重管理

**問題**: API設計ルール（`.claude/01_development_docs/03_api_design.md`）と個別API定義（`docs/設計書/API一覧.md`）の両方にエンドポイント情報を書いてしまい、片方だけ更新されて矛盾が発生。

**対策**: ドキュメントの役割を「方針（How）」と「実態（What）」に明確分離。同じ情報を2箇所に書かないルールを CLAUDE.md に明記。

### 落とし穴6: ブラウザテストのやり直しコスト

**問題**: 当初は `実装 → /browser-test → /code-review` の順序だったが、コードレビューで修正が必要になると、再度ブラウザテスト（10〜20分）をやり直すハメになった。

**対策**: `実装 → /code-review → /browser-test` の順序に変更。コードの最終形に対してブラウザテストを実行することで、やり直しを防ぐ。

### 落とし穴7: テストコードを書かないと回帰できない

**問題**: ブラウザテストを手続き的に実行するだけだと、機能が増えるごとにテストコストが膨らむ一方で、回帰テストスイートは蓄積されない。

**対策（Phase 1）**: `browser-tester` のレポートに「テスト手順の記録」セクションを追加し、将来の `.spec.ts` 変換の元データとして蓄積。今すぐ自動生成はしない（複雑すぎて低品質なテストが量産される）。月間コストが許容範囲（$2〜5）に収まっているうちは Phase 1 で留める。

### 落とし穴8: 要件の誤解が DB/API 設計後に発覚

**問題**: 単一の設計レビューで「技術整合性」と「企画・UX 妥当性」を同時に見ようとすると、どちらも浅くなる。「この機能本当に必要？」のような根本的な指摘が DB/API 設計完了後まで出てこない。

**対策**: 設計を **Stage 1（機能・画面設計）と Stage 2（技術設計）の2段階** に分離（L 規模で必須）。Stage 1 レビューで **code-reviewer + product-advisor を並列実行** し、技術整合性と企画・UX 体験を同時にチェックする。Stage 1 承認後にのみ Stage 2 に進むことで、手戻りを最小化する。

### 落とし穴9: 規模判定がユーザー任せになる・過小評価される

**問題**: S/M/L の判定基準を書いても、ユーザーは「これは S で」と楽な方に流れがち。AI も指示通りに小さく実装しようとする。結果、実は L 相当の変更を S フローで進めて設計書・テストが抜ける。

**対策**: **「AI が推測 → ユーザーが承認」の2ステップ判定**をルール化。AI には「判定に迷ったら L 寄りで提示」のバイアス補正を入れ、ユーザーが「それは S で十分」と下げる方向の判断をさせる。過小評価より過大評価の方が安全。

### 落とし穴10: CLAUDE.md の肥大化（自己参照的な問題）

**問題**: ルール追加を繰り返すと CLAUDE.md が肥大化し、自ら警告していた「後半が無視される」現象が発生する。本テンプレートも Phase 1 で 319 行まで膨らんだ。

**対策**: **要点と詳細を分離**する構造に変更。CLAUDE.md には「AI が毎回必ず意識すべき要点」のみ残し、詳細な判定基準・プロセス例・判定イメージは `.claude/01_development_docs/09_開発フローと規模判定.md` のような専門ドキュメントに移す。Phase 1.5 で CLAUDE.md を 319 行 → 252 行に圧縮した。テンプレの「方針 vs 実態」分離思想と整合させる。

## 11. テンプレートの導入方法

1. このリポジトリをクローンまたはコピー
2. `CLAUDE.md` のプロジェクト概要・スタック情報を自分のプロジェクトに合わせて編集
3. `.claude/01_development_docs/` 内の設計方針ドキュメントをプロジェクトに合わせてカスタマイズ
4. 不要な設計書（AI関連、認証関連等）があれば削除
5. `.mcp.json` を確認し、不要な MCP サーバーは削除（Playwright MCP を使わない場合等）
6. `.claude/settings.json` のフックを確認（Prisma を使わない場合は DBバックアップフックを削除）
7. `/new-feature 機能名` で最初の機能設計書を作成して開発開始

## 12. まとめ

| 観点 | このテンプレートのアプローチ |
|------|---------------------------|
| **プロセス設計** | 変更規模に応じた S/M/L フローで、過剰なプロセスを回避 |
| **品質担保** | スキル（レビュー・ビルド・ブラウザテスト）+ フック（型チェック・DBバックアップ）で自動化 |
| **設計書管理** | 方針（`.claude/`）と実態（`docs/設計書/`）を分離し、3階層の同期で乖離を防止 |
| **コスト最適化** | サブエージェントは sonnet、エージェント定義は最小限、重複排除 |
| **実機検証** | Playwright MCP で browser-tester が実際にブラウザ操作し、UX評価までカバー |
| **段階的改善** | `.spec.ts` 自動生成などは Phase 2 送り。過剰設計を避け、実運用で必要性が見えてから導入 |

バイブコーディングは「AIに全部任せる」ことではなく、**AIが最大限パフォーマンスを発揮できる仕組みを整えたうえで、人間が判断・検証する**ことで成立する。このテンプレートはその仕組みの実装例である。
