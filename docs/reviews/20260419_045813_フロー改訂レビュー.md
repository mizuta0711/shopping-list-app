## 1. 総合評価

方向性は正しい。現状の曖昧さ（S/M/L 判定、バグ修正の扱い、product-advisor 孤立）を解消しようとする動機は明確で、既存ルールとの整合も概ね取れている。

ただし、**実装前に解消すべき構造的な問題が4件**ある。特に「1段階設計の扱いの未決定」と「スコープ拡大時の再判定ルールの曖昧さ」は、実装してから矛盾が発覚するリスクが高い。

## 2. 重要度別指摘一覧

### ■ 重要（実装前に解消が必須）

#### 指摘1: 1段階設計の扱いが Phase 2 送りになっており、TEMPLATE.md 変更が宙に浮く

**問題**: 設計書 §3-6 で「M 規模で UX 変更なし」の場合に Stage 1 セクションをどう扱うかを「Phase 2 実装時に決定」と保留している。しかし Phase 2 のタスク（#4）は TEMPLATE.md の Stage 1/2 構造化そのものであり、**構造を決めずにテンプレートを変更することはできない**。Phase 2 が自己参照的に詰まる。

**影響**: Phase 2 実装時に案A/B/C のどれかで再設計が発生する。後で「軽量テンプレ TEMPLATE_simple.md を別途用意」となれば新規ファイル追加が発生し、「既存ファイルの更新のみ」という §4 の方針とも矛盾する。

**対応案**: 案A（Stage 1 を `⚪ 不要` ステータスで空欄にする）を Phase 1 の時点で採用方針として決定し、TEMPLATE.md 変更前に設計書に記載する。新規テンプレは YAGNI。案B（TEMPLATE_simple.md）は削除対象。

---

#### 指摘2: スコープ拡大時の「作業を一度止める」判断基準が未定義

**問題**: §3-1 に「S で始めたが M 相当と判明したら作業を一度止めてユーザーに報告する」とある。しかし「いつ止めるか」の基準が定義されていない。

- 実装開始直後に気づいた場合
- 実装が 60% 終わった段階で気づいた場合
- コミット直前に気づいた場合

3つのケースで取るべき行動が異なるが、設計書は区別していない。特に「60% 実装済み」状態でロールバックを強いると作業コストが大きく、ユーザーが判断に迷う。

**影響**: AI が「もう途中まで作ったから止めにくい」と判断して規模引き上げを先送りにするサンクコスト効果が起きやすい。これは入門ガイド #10「思い付き修正のループ」と同じ構造の問題。

**対応案**: 「実装着手前に気づいた場合は即報告。着手後に気づいた場合は現在のフェーズ完了まで作業を継続した上で報告し、次フェーズ前に判断を仰ぐ」という段階を明記する。

---

#### 指摘3: `/design-review feature` で product-advisor と code-reviewer を並列実行した際の「統合」ロジックが未定義

**問題**: §3-4 に「両者の結果を統合してレビュー結果ファイルに保存」とある。しかし以下が未定義。

- 両エージェントが同じ設計書に対して矛盾する評価を出した場合の扱い
- 統合フォーマット（product-advisor の出力フォーマットと code-reviewer の出力フォーマットは構造が異なる）
- 並列実行中に片方がエラー/タイムアウトした場合の処理

現在の `/design-review` スキル（SKILL.md）のフォーマットは「## 要修正」テーブル形式を前提にしているが、product-advisor の出力フォーマット（`## 課題・懸念` + `## 改善提案`テーブル）は異なる。メインエージェントが統合する際に何を基準にまとめるかが不明確。

**影響**: 実装してみてから「統合が難しい」「どちらかが無視される」という問題が発覚する可能性が高い。code-reviewer が product-advisor の観点を code レベルで上書き評価する、またはその逆が起きやすい。

**対応案**: SKILL.md の `feature` モードに統合フォーマットのテンプレートを明記する。「product-advisor 観点（企画妥当性）」と「code-reviewer 観点（技術整合性）」を別セクションで並記し、統合判定はメインが最後に行う構造を明示する。

---

#### 指摘4: バグ修正で M/L 設計書を書く際、既存「バグ修正の原則」との手順の競合

**問題**: CLAUDE.md「バグ修正の原則」には「原因特定前の修正禁止」「仮説の先行共有」「2回失敗したら立ち止まる」が定められている。これは「まず原因を確認し、ユーザーと共有し、承認を得てから動く」という**インライン対話**を前提にしている。

設計書 §3-2 では M/L バグ修正に「軽量な設計書を作成」とある。しかし設計書作成は `/new-feature` → 記入 → `/design-review` という**同期的でないプロセス**であり、緊急性の高いバグ（例: 本番でデータ不整合が起きている）では設計書を書くコストが致命的な遅延になる。

「設計書を作るべきバグ」と「原則に従いインライン対話で解決すべきバグ」の判別基準が、設計書にもCLAUDE.mdにも定義されていない。

**影響**: M 規模のバグに「設計書を書け」というフローと「まず原因を共有しろ」という原則が並立し、AI がどちらを優先すべきか迷う。設計書作成を優先すると原因特定が後回しになり、「バグ修正の原則」の精神に反する。

**対応案**: バグ修正フローに「Step 0: 原因特定（CLAUDE.md 原則）」を明示的に先行させる。「原因確定後、影響範囲が広い（M/L）と判明した時点で設計書を起こす」という順序を設計書に記載する。設計書を原因調査前に作らないことを明記する。

### ■ 中（実装前に対応推奨）

#### 指摘5: AI の判定バイアス対策が「注意事項」止まり

§6「規模判定の AI 推測精度」で「Claude が勝手に S と判定して小さく進める傾向がある可能性。CLAUDE.md に補正ルールを入れるか、実装時に検討」と述べている。これは CLAUDE.md への追記というタスクを Phase 1 に含めるべき事項だが、タスク一覧（§2）には存在しない。

「判定に迷ったら L 寄りで提示する」というルールは、Phase 1 の CLAUDE.md 更新時に同時に書くべき内容。注意事項に留まっているのは漏れの可能性がある。

---

#### 指摘6: TEMPLATE.md の「3. 設計詳細（フェーズ別）」と新「3. Stage 1 / 4. Stage 2」の番号ズレが設計書内で整理されていない

現在の TEMPLATE.md: `3. 設計詳細` → `4. ブラウザ評価計画` → `5. 関連ドキュメント` → `6. 注意事項`
設計書 §3-6 の新構造: `3. 機能・画面設計（Stage 1）` → `4. 技術設計（Stage 2）` → `5. ブラウザ評価計画` → `6. 関連ドキュメント` → `7. 注意事項`

セクション番号が全てズレる。現在の SKILL.md（`/design-review`）や運用ガイドには「3. 設計詳細」を参照している箇所があり、番号ズレによるメンテナンスコストが生じる。設計書 §4 の影響ファイル一覧に `.claude/skills/design-review/SKILL.md` が含まれているが、番号変更の影響として明示されていない。

---

#### 指摘7: M 規模で「UX 変更あり」の 2段階設計が推奨にとどまり、境界が曖昧

§3-3 に「M 規模で UX 変更を含む場合（推奨）」とある。「推奨」だと AI は状況次第でスキップしやすい。UX 変更の有無は規模判定表（§3-1）の観点にも含まれているが、「UX 変更あり = Stage 1 必須」と読める記述と「推奨」という記述が混在している。

「UX 変更を含む M 規模」は実質 L 相当に引き上げるか、または「Stage 1 必須」として明記すべきかを決定する必要がある。

---

CLAUDE.md「設計書の改訂履歴ルール」には「`/update-docs` 実行時は `docs/設計書/.doc-sync.md` にも同期記録を追記すること」とある。本フロー改訂は CLAUDE.md 自体の変更を伴うが、`.doc-sync.md` の扱いがタスク一覧・影響ファイル一覧のどちらにも記載されていない。

## 3. 削除・簡素化すべき部分

### 削除推奨: 案B「TEMPLATE_simple.md」の選択肢

§6 の「1段階設計の扱い」の案B は新規ファイル追加を伴い、テンプレートの分岐を増やす。YAGNI。1段階設計に対してファイルを分けることで「どちらを使うか」の判断コストが増える。案A で十分機能する。設計書から案B の言及を削除し、方針を確定させる。

### 簡素化推奨: `product-advisor` への呼び出しプロンプト例（§3-5）

設計書に記載されている呼び出しプロンプトは SKILL.md に書くべき実装詳細であり、設計書の設計書に書くべきではない。設計書には「product-advisor をどのモードで、どの観点で呼ぶか」の意図のみを記載し、プロンプト文字列は実装時に SKILL.md に書く。

### 検討推奨: Phase 5（図の更新）の優先度

タスク #10（`02_開発フロー図.drawio` 更新）と `05_全体アーキテクチャ図.drawio` 更新は、drawio ファイルであり Claude Code が直接更新しにくいフォーマット。Phase 1〜4 が完了してフローが安定してから実施する方が手戻りが少ない。現状の Phase 5 という位置づけは正しいが、「フロー安定後に実施」という条件を明記しておく。

## 4. 見落とされている観点

### 複数機能の並行開発時のセッション管理

設計書は「1機能 = 1設計書」の単線フローを前提にしているが、Stage 1 レビュー待ち中に別機能の Stage 2 を進める場合のコンテキスト管理について言及がない。入門ガイド #20「コスト・セッション運用」では「1機能 = 1セッション」を推奨しているが、2段階設計で Stage 間にユーザー承認待ちが発生すると、別機能への文脈切り替えが発生しやすい。これが想定内なのか、あえて順番待ちを強いる設計なのかを明記する。

### Stage 1 承認後に Stage 2 で方針転換が必要になった場合の戻り手順

L 規模の技術設計（Stage 2）を進める中で「この技術的制約では Stage 1 の受け入れ条件を満たせない」と判明した場合の手順がない。Stage 1 に戻って再レビュー（`/design-review feature` を再実行）するのか、Stage 2 内で設計を修正してよいのかの基準が未定義。実際には技術制約による UX 変更は珍しくない。

### `改訂履歴` の `コミット` 列欠落

CLAUDE.md「設計書の改訂履歴ルール」は「コミット列にはトリガーとなったコミットの短縮ハッシュ（7文字）を記入」を必須としているが、本設計書（§改訂履歴）の 1.0 行にコミット列がない。フォーマット違反。
