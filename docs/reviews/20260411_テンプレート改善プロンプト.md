# 別プロジェクトへのテンプレート改善適用プロンプト

以下のプロンプトをそのまま Claude Code に貼り付けて使用してください。

---

## プロンプト

```
このプロジェクトの CLAUDE.md、エージェント定義（.claude/agents/）、AGENTS.md に対して以下の4つの改善を行ってほしい。

### 改善1: プロセスの軽量化（CLAUDE.md）

現在の「設計・実装の必須フロー」を、変更規模別の3パスに書き換える。

**変更前（イメージ）**:
全変更に9ステップの必須フローを適用

**変更後**:
- **S（バグ修正・軽微な変更）**: 実装 → /build-check → コミット
- **M（機能追加・API変更・複数ファイル）**: 設計 → 実装 → /code-review → /build-check → /update-docs → コミット
- **L（新機能・大規模変更・DBスキーマ変更）**: 設計 → /design-review → 実装 → /code-review → /build-check → /update-docs → コミット
- **プッシュ前（共通）**: /pre-push-check → プッシュ（/sync-check はフェーズ完了時のみ）
- 「省略禁止」の文言は削除し、「迷ったらMを使うこと」と記載

### 改善2: 設計書の二重管理解消（CLAUDE.md）

ドキュメント構成セクションを「役割分離」として書き換える。

- `.claude/01_development_docs/` = **設計方針・パターン・ルール**（命名規則、エラー処理方式、API設計ルール等）。変更頻度: 低
- `docs/設計書/` = **実態の一覧・定義**（API一覧, テーブル定義書等）。変更頻度: 高（コードと同期）
- 「二重管理の禁止」ルールを明記: 個別エンドポイント定義は `docs/設計書/API一覧.md` にのみ書く。`.claude/` 配下の設計ドキュメントにはルール・パターンだけを書く。

### 改善3: エージェント定義の圧縮

各エージェント定義ファイルを100行以内に圧縮する。以下を実施:

1. **CLAUDE.md に「コーディングルール（全エージェント共通）」セクションを追加**:
   - `any` 型禁止
   - `memo<Props>` + `displayName` パターン
   - API Route: `NextRequest`/`NextResponse` + try/catch + Service層経由
   - UI: 縦スクロール対応、レスポンシブ

2. **各エージェントから以下を削除**:
   - TypeScript interface風の擬似コード（`interface ProgressReport` 等）
   - mermaid図
   - コード例（良い例/悪い例）
   - CLAUDE.md と重複するチェックリスト
   - 絵文字の見出しプレフィックス

3. **各エージェントに残すもの**:
   - 役割の1行説明
   - 参照ドキュメントの対応表（作業内容→参照先）
   - そのエージェント固有のプロセス（箇条書き）
   - 完了報告の必須事項（3-4行）
   - 「CLAUDE.md のコーディングルールに従うこと」の参照

### 改善4: PMエージェントの削除

- `.claude/agents/project-manager.md` を削除する
- 理由: Claude Codeのアーキテクチャでは、PMを経由したサブエージェント委任はトークンコストが増えるだけでリターンが小さい。ユーザーが直接 coding-specialist を呼ぶ方が効率的。複雑なタスク分解が必要な場合は組み込みの Plan エージェントで十分。

### 追加: AGENTS.md の統合

もし AGENTS.md が存在し、内容が薄い（Next.jsの注意書き等のみ）場合:
- 内容を CLAUDE.md に直接統合する
- AGENTS.md を削除する
- CLAUDE.md 内の `@AGENTS.md` 参照を削除する

---

作業後、変更の差分を簡潔に報告してください。
```

---

## 補足: プロジェクト固有の調整が必要な箇所

上記プロンプトはテンプレートの改善内容をそのまま適用するものです。プロジェクト固有の事情で以下の調整が必要になる場合があります:

1. **コーディングルール**: プロジェクトで使っていないスタック（例: Zustand, Prisma）に関するルールは省略してよい
2. **フロー分岐の基準**: プロジェクトの規模や開発フェーズに応じて S/M/L の基準を調整してよい
3. **product-advisor エージェント**: このエージェントは改善対象外。プロジェクトに存在する場合はそのまま残す
4. **スキル定義（`.claude/skills/`）**: 今回の改善対象外。ただし `/code-review` や `/update-docs` のスキル内で「省略禁止」等の文言がある場合は、フロー変更に合わせて調整してよい
