## 総合評価

**結論: 段階的に取り込むべき。Phase 1 で取り込み、Phase 2 以降は実運用の課題を見てから判断**

評価の仕組み自体は良いアイデアで、UI変更を含む開発の品質向上に寄与する。ただし**一度に全機能を取り込むのは過剰**で、実運用で課題が顕在化してから段階的に強化するアプローチを推奨する。

---

## 仕組みの構造

現在の設計は3層構成:

```
設計フェーズ:      test-planner（計画の草案作成・レビュー）
                        ↓
実装後:            /browser-test → browser-tester（実行）
                        ↓
共通リソース:      .claude/browser-test-checklist.md（マスターチェックリスト）
```

---

## 良い点

### 1. 設計時から評価を組み込む発想が優れている

機能設計書のテンプレートに「4. ブラウザ評価計画」を追加し、設計段階でテスト項目を定義させる仕組みは正しい。「何をテストすべきか」を実装前に明確にすることで、評価計画が形骸化しない。

### 2. スポットテスト / フルテストの使い分け

変更ファイルから対象画面を自動判定してスポットテストを実行する仕組みは、コスト削減の観点で実用的。フルテストはフェーズ完了時のみに限定されている。

### 3. Playwright MCP 採用は正解

`package.json` に Playwright を直接インストールせず、`npx @playwright/mcp@latest` で起動する方式は、依存を増やさず、プロジェクトのビルド時間を増やさない点で優れている。

### 4. エビデンスの体系的保存

`tests/browser-evidence/YYYYMMDD_HHMMSS_テスト名/` へのスクリーンショット + `report.md` 保存、`.gitignore` で除外しつつ `.gitkeep` でフォルダ構造を保持する運用は適切。

### 5. 設計書と実態の二重管理を避けている

`browser-test-checklist.md` は「汎用チェック項目」、設計書の「4. ブラウザ評価計画」は「機能固有の評価項目」と役割が明確に分離されている。

## 改善すべき点（優先度順）

### 【優先度: 高】問題1: test-planner エージェントが /design-review と役割重複

**現状**:
- `/design-review` に「ブラウザ評価計画のチェック」セクションを追加（13行）
- L規模では `/design-review` 内から `test-planner` エージェントに委譲

**何が問題か**:
- `/design-review` スキル自体が評価計画のチェックルールを持っている
- さらに L規模では別エージェント（test-planner）に委譲する
- **スキルがエージェントを呼び出して同じことをチェックする構造**になっている
- test-planner の「レビューモード」は /design-review と完全に同じ観点
- エージェント数が増えるとトークン・管理コストが増える

**改善案**:
- `test-planner` を削除する
- 評価計画のレビュー観点は `/design-review` スキル内に直接記述する（既に13行追加されている）
- 「提案モード」は `/new-feature` スキル内の設計書生成時に組み込むか、独立した `/propose-test-plan` スキルとして切り出す（エージェントにする必要はない）

**→ Phase 1 で対応**

### 【優先度: 高】問題2: /browser-test の実行順序が不適切

**現状のフロー**:
```
実装 → /browser-test → /code-review → /build-check → /update-docs → コミット
```

**何が問題か**:
- `/code-review` で指摘された修正を行うと、再度 `/browser-test` を実行する必要がある
- ブラウザテストは10〜20分かかるため、やり直しのコストが大きい
- コードレビューで「このコードはリファクタが必要」と指摘されたあと、リファクタ後にまたブラウザテストをする手間

**改善案**:
```
実装 → /code-review（コード修正）→ /browser-test → /build-check → /update-docs → コミット
```

コードレビューで構造的問題を先に潰し、最終形に近い状態でブラウザテストを実行する方が効率的。

**→ Phase 1 で対応**

### 【優先度: 中】問題3: テストが使い捨てで回帰テスト資産にならない

**現状**: `/browser-test` は毎回手続き的にブラウザ操作を実行する。テストコード（`.spec.ts`）は一切生成・保存されない。

**何が問題か**:
- 機能Aをテストしたあと、機能Bの実装で機能Aが壊れても、変更範囲から機能Aが自動マッピングされない限り再テストされない
- 毎回同じテスト項目を Claude がブラウザ操作で実行するため、テスト資産が蓄積しない
- フルテスト1回で数十万トークン・10〜20分消費するが、成果物は次回には再現されない

**ただし、想定コストはそれほど大きくない**:

| 項目 | 1回あたり | 月間（20機能/月） | 月額換算 |
|------|-----------|-------------------|---------|
| スポットテスト | 3〜8万トークン | 60〜160万トークン | $1〜3 |
| フルテスト（2回/月） | 30〜50万トークン | 60〜100万トークン | $1〜2 |
| **合計** | | | **$2〜5** |

**`.spec.ts` 自動生成の難しさ**:
- Playwright の stable なセレクタ設計が必要（`data-testid` 付与の運用ルール化）
- 動的データ（ID、タイムスタンプ）のパラメータ化
- 認証状態の管理（`storageState` 等）
- 待機処理の適切な挿入（外すと flaky テストが量産される）
- **AI 生成のテストは脆い・冗長・重複しがち**で、質が低いと回帰資産ではなく**回帰負債**になる

**結論: 今すぐ導入すべきではない**

月 $5 程度のコストに対して、自動生成された低品質テストの保守コストの方が大きくなる可能性が高い。実運用で以下の兆候が出てから判断する。

**Phase 2 導入の判断基準**:
- クロス機能の回帰バグが実際に発生した（スポットテストで検出漏れの実例がある）
- フルテストのトークンコストが月 $20〜30 を超える
- クリティカルパス（ログイン、決済、主要CRUD等）が明確になった

**→ Phase 2 で対応**

### 【優先度: 低】問題4: browser-test-checklist.md の保守負担

**現状**: 15画面分の具体的なチェック項目が手動記述されている。

**懸念**:
- 画面追加・削除・URL変更のたびに更新が必要
- テンプレートに取り込む場合、プロジェクトごとに全書き換えが必要

**改善案**:
- 「共通チェック項目」（レスポンシブ、スクロール等の汎用観点）のみテンプレートに残す
- 画面別チェック項目は機能設計書の「4. ブラウザ評価計画」に集約

**→ Phase 1 で対応**

### 【優先度: 低】問題5: Dev サーバー依存と状態管理の曖昧さ

**現状**:
- `http://localhost:3000` で dev サーバーが起動済みであることが前提
- テストデータの扱いは「POST/PUT/DELETE はテスト用データ明示時のみ」
- 「テスト用データ」がどう準備されるかは未定義

**懸念**:
- 認証が必要な画面のテストでログイン状態をどう管理するか不明
- データが中途半端な状態でテストすると結果が不定

**改善案**:
- 設計書の「4. ブラウザ評価計画」に「前提条件」欄を追加（テストユーザー、シードデータ等）
- `browser-tester` の前提条件にこれを明記

**→ Phase 1 で対応（設計書テンプレートの「前提条件」欄追加）**

### 【優先度: 低】問題6: エビデンスの report.md 生成が漏れる可能性

**現状確認**: `.playwright-mcp/` にフルテストの痕跡はあるが、対応する `tests/browser-evidence/` フォルダに `report.md` がなかったケースが存在する。

**改善案**:
- browser-tester のプロセスを「各画面のテスト完了ごとに report.md を追記する」形式に変更（バッチではなくインクリメンタル）
- `/browser-test` の Step 4 で `report.md` の存在チェックを入れる

**→ Phase 1 で対応**

### 【優先度: 低】問題7: /code-review の警告だけでは実効性が弱い

**現状**: `/code-review` で UI変更がある場合「ブラウザテスト未実施なら警告」

**改善案**:
- M/L規模のフロー順序を「実装→/code-review→/browser-test」に変えれば自然と実施される
- 問題2（フロー順序）の修正で自動的に解決される

**→ Phase 1 で対応（問題2と同時に解決）**

## 段階的導入計画

### Phase 1: 今すぐ取り込む（基本機能 + 修正2点）

| 要素 | アクション |
|------|-----------|
| `/browser-test` スキル | 取り込み。ただしフロー順序は `/code-review` の後に変更 |
| `browser-tester` エージェント | 取り込み |
| `browser-test-checklist.md` | **共通チェック項目のみ**に絞ってテンプレ化 |
| 機能設計書「4. ブラウザ評価計画」 | `TEMPLATE.md` に追加。「前提条件」欄を含める |
| `/design-review` の評価計画チェック | 取り込み（スキル内で完結） |
| `.mcp.json` の playwright エントリ | 取り込み |
| `.gitignore` の `tests/browser-evidence/*/` | 取り込み |
| **`test-planner` エージェント** | **取り込まない**（/design-review と重複） |
| **.spec.ts 自動生成** | **今は実装しない**（Phase 2 で判断） |

### Phase 1 のフロー

**M規模**:
```
設計（評価計画も記入）→ 実装 → /code-review → /browser-test → /build-check → /update-docs → コミット → /done
```

**L規模**:
```
設計（評価計画も記入）→ /design-review（評価計画のレビュー含む）
→ 実装 → /code-review → /browser-test → /build-check → /update-docs → コミット → /done
```

### Phase 2: 実運用の課題を見て判断（目安: 3〜6ヶ月後、または機能数20〜30到達時）

以下の条件を満たしたら、**クリティカルパスのみ** `.spec.ts` 化することを検討:

- クロス機能の回帰バグが実際に発生した
- フルテストのトークンコストが月 $20〜30 を超える
- クリティカルパス（ログイン、決済、主要CRUD等）が明確になった

ただし、その時点でも:
- **全テストを自動化しない**。クリティカルパスに限定する
- **MCP で生成するのではなく、人間または Claude Code が手動で書く**（質の担保のため）
- `data-testid` 付与の設計ルールを先に整備する

### Phase 3: 成熟後（必要があれば）

- `npx playwright test` の CI 統合
- テストユーザー・シードデータの専用管理
- E2E テストの充実

## テンプレート取り込み方針（Phase 1）

### 取り込むもの

| 要素 | 備考 |
|------|------|
| `/browser-test` スキル | フロー順序を修正して取り込む |
| `browser-tester` エージェント | `.spec.ts` 保存機能は**実装しない** |
| `browser-test-checklist.md` | 共通項目のみ、画面別は空テンプレ |
| 機能設計書の「4. ブラウザ評価計画」 | TEMPLATE.md に追加、「前提条件」欄を含める |
| `/design-review` の評価計画チェック | スキル内で完結 |
| `.mcp.json` の playwright エントリ | 必須 |
| `.gitignore` の `tests/browser-evidence/*/` | 必須 |

### 取り込まないもの

| 要素 | 理由 |
|------|------|
| `test-planner` エージェント | `/design-review` と役割重複。スキル内で完結できる |
| `/new-feature` への test-planner 案内 | test-planner を削除するため不要 |
| `.spec.ts` 自動生成 | 複雑すぎる。Phase 2 で実運用の課題を見てから判断 |

## まとめ

| 観点 | 評価 |
|------|------|
| 設計思想 | ◎ 評価計画を設計段階から組み込む発想は優秀 |
| 実装の完成度 | ○ 一部重複・順序問題あるが、修正すれば十分実用的 |
| トークン効率 | ○ 想定コスト月 $2〜5 は許容範囲。将来必要なら .spec.ts 化 |
| テンプレート適合性 | ◎ Phase 1 の修正を加えれば取り込む価値あり |

**推奨アクション（Phase 1）**:
1. `test-planner` エージェントを削除し、`/design-review` スキルに機能統合
2. フロー順序を `/code-review → /browser-test` に変更
3. `browser-test-checklist.md` を汎用テンプレートに整理（共通項目のみ）
4. 機能設計書の「4. ブラウザ評価計画」に「前提条件」欄を追加
5. 上記を反映したうえでテンプレートに取り込む

**Phase 2 の判断材料**（今は実施しない）:
- クロス機能の回帰バグ発生の有無
- 月間トークンコストの推移
- クリティカルパスの明確化状況

## 改善実施記録（2026-04-18）

レビュー指摘に基づき、skillup_mock プロジェクトで以下の改善を実施した。

### 対応サマリ

| 指摘 | 判定 | 対応内容 |
|------|------|---------|
| 問題1: テスト使い捨て | 一部妥当 | `.spec.ts` 自動生成は見送り。代わりに report.md に「テスト手順の記録」セクションを追加し、回帰テスト変換の元データとして蓄積する段階的アプローチを採用 |
| 問題2: test-planner 重複 | 妥当 | `test-planner` エージェントを削除。レビュー観点と草案生成機能を `/design-review` スキルに統合。`/new-feature` の案内も `/design-review` 参照に変更 |
| 問題3: フロー順序 | 妥当 | M/L フローを `実装 → /code-review → /browser-test` に変更。コードレビュー後の最終形に対してブラウザテストを実行する構造に改善 |
| 問題4: checklist 保守負担 | 妥当 | 画面別チェック項目を全削除し、共通チェック項目のみに整理。画面固有のテスト項目は機能設計書の「4. ブラウザ評価計画」に集約 |
| 問題5: 前提条件の曖昧さ | 妥当 | TEMPLATE.md の「4. ブラウザ評価計画」に「前提条件」欄を追加（テストユーザー、シードデータ等） |
| 問題6: report.md 漏れ | 対応済み | 前回修正で `/browser-test` スキル側にフォールバック機構を実装済み（サブエージェントが保存しなければメインが書く） |
| 問題7: /code-review 警告の実効性 | 問題3で解消 | フロー順序変更により自然に解決。警告文言も「コードレビュー後に /browser-test を実施してください」に変更 |

### 見送った項目と理由

| 項目 | 理由 |
|------|------|
| `.spec.ts` 自動生成 | MCP インタラクションから正確なテストコードを生成するのは現時点では複雑すぎる。report.md の「テスト手順の記録」を蓄積し、実運用で回帰バグの発生頻度を見てから Phase 2 として判断する |
