## 重要指摘の修正詳細

### #1: PUT /api/sync/sets の unclassifiedId null ガード不足

`POST /api/sync/sets/merge` と同様に、`unclassifiedList` が null の場合に早期 return するガードを追加。

### #2: v1ToV2 マイグレーション失敗時の version 汚染

失敗時に v1 データをそのまま返すと zustand が version=2 で保存してしまい、次回起動で再試行されなくなる。
失敗時は `throw` して zustand に初期状態リセットを委ねるか、または v1 バックアップにフォールバックする場合でも
IN_PROGRESS フラグを残して次回 `migrate()` が呼ばれた時に再試行できるようにする必要がある。

現実的な修正: 失敗時は `throw` する。zustand は例外を catch して `initialState`（空配列）で version=2 を保存する。
バックアップは LocalStorage に残るのでサポート対応は可能。設計書の「v1 互換モード継続」よりデータ消失リスクが低い。

## 修正実施

### #1 修正: `src/app/api/sync/sets/route.ts`

- `unclassifiedList` が null の場合に `badRequest` で早期 return するガードを追加（L109-113）
- `resolvedListId` の最終フォールバックを `unclassifiedId ?? ""` から `existing?.listId ?? unclassifiedId` に変更。`unclassifiedId` は上記ガードにより必ず string になるため `??` 以降は到達しない（L149）

### #2 修正: `src/features/shopping/migrations/v1ToV2.ts`

- マイグレーション失敗時の catch ブロックを修正。v1 データを返す代わりに `throw err` で再スローするよう変更
- zustand が例外を catch して `initialState`（空配列）+ version=2 で保存するため、次回起動で `listId` 未設定状態が永続化されない
- バックアップ（`SETS_BACKUP_V1_KEY`）は LocalStorage に残り、IN_PROGRESS フラグも残存するためデータの手動復元は可能
- toast.error でユーザーに通知してから throw
