## 総合評価

設計書とコードの同期を重視した、非常に野心的なテンプレート。ドキュメント駆動開発をClaude Codeで実現しようとするアプローチは独自性が高い。ただし、**プロセスの重さが実用性を損なうリスク**がある。テンプレートとして他プロジェクトに展開する場合、「全部入り」が逆に導入障壁になる。

---

## 良い点

### 1. 設計書同期の仕組みが体系的
- `.doc-sync.md` によるコミット単位の追跡、`/update-docs` → `/pre-push-check` → `/sync-check` の3段階チェックは、設計書の陳腐化という普遍的な問題に正面から取り組んでいる
- 「変更駆動」と「全量照合」の2つのアプローチを使い分ける設計は理にかなっている

### 2. スキルの設計が具体的
- 各スキルが「何を読み、何をチェックし、何を出力するか」を明確に定義している
- 特に `/update-docs` の変更ファイル→更新対象の対応表は実用的

### 3. エージェントの役割分離が明確
- coding-specialist / code-reviewer / documentation-manager の責務が重ならないよう意識されている
- product-advisor の追加は、技術以外の視点を取り込む良いアイデア

### 4. コスト最適化の意識
- サブエージェントに `model: "sonnet"` を指定するルールは実用的
- ドキュメント参照ルール（「全てを読まず該当分のみ」）もトークン節約として正しい

## 問題点と改善提案

### 問題1: プロセスが重すぎる（最大の問題）

**現状**: 必須フローが9ステップある。

```
設計 → /design-review → 実装 → /code-review → /build-check → /update-docs → コミット → /pre-push-check → /sync-check → プッシュ
```

**何が問題か**:
- 1行のバグ修正でも全ステップ必須（「省略禁止」と明記されている）
- 各スキルの実行にサブエージェント起動 + ドキュメント読み込みのコストがかかる
- 現実的に、ユーザーは面倒になってフローを無視するか、Claude Codeに全部任せて結果を確認しなくなる
- **テンプレートとして他プロジェクトに渡した場合、最初の1機能で挫折する可能性が高い**

**改善案**: 変更規模に応じた軽量パスを用意する。

```markdown
## 変更規模別フロー

### S（バグ修正・軽微な変更）
実装 → /build-check → コミット

### M（機能追加・API変更）
設計 → 実装 → /code-review → /build-check → /update-docs → コミット

### L（新機能・大規模変更）
設計 → /design-review → 実装 → /code-review → /build-check → /update-docs → コミット

### プッシュ前（共通）
/pre-push-check → プッシュ
（/sync-check はフェーズ完了時のみ）
```

---

### 問題2: 設計書の二重管理

**現状**: 同じ情報が2箇所に分散している。

| 情報 | 場所1（詳細設計） | 場所2（一覧） |
|------|-------------------|---------------|
| API | `.claude/01_development_docs/03_api_design.md` | `docs/設計書/API一覧.md` |
| サービス | `.claude/01_development_docs/06_service_repository_design.md` | `docs/設計書/サービス・リポジトリ一覧.md` |
| フック | `.claude/01_development_docs/07_hooks_design.md` | `docs/設計書/フック一覧.md` |
| DB | `.claude/01_development_docs/02_database_design.md` | `docs/設計書/テーブル定義書.md` |

**何が問題か**:
- 同じ情報の同期コストが2倍になる
- どちらが正（Source of Truth）か曖昧
- `/update-docs` が両方を更新する必要があり、漏れやすい

**改善案**: 役割を明確に分離する。

- `.claude/01_development_docs/` → **設計方針・パターン・ルール**（変更頻度：低）
- `docs/設計書/` → **実態の一覧・定義**（変更頻度：高、コードと同期）

例えば `03_api_design.md` には「APIの設計ルール（命名規則、ページネーション方式、エラーレスポンス形式）」だけを書き、個別APIのエンドポイント定義は `API一覧.md` にのみ書く。重複する情報を持たないこと。

### 問題3: project-manager エージェントが機能しにくい

**現状**: PMエージェントは「Write/Edit禁止」「全てサブエージェントに委任」「Taskツールによる呼び出し必須」。

**何が問題か**:
- Claude CodeのAgent toolでは、PMがサブエージェントを起動→完了待ち→次のサブエージェント起動…というフローになり、コンテキストの受け渡しが非効率
- PMを経由する分だけトークンコストが増える（PMの判断 + 委任プロンプト + サブエージェントの実行）
- 実際の開発では、ユーザーが直接 coding-specialist を呼ぶ方が速く正確
- PMがTask経由でCSに「この設計書のパスを参照して実装せよ」と指示する構造は、ユーザーが直接同じことを言うのと変わらない

**改善案**: PMエージェントは削除するか、役割を「計画立案のみ」に限定する。

```
- PM → 削除（または「/plan」スキルとして再定義）
- 実装の指揮はユーザーが直接行う
- 複雑なタスク分解が必要な場合のみ、Plan エージェントを使う
```

PMが本当に必要なのは「何をどの順番で実装するか」の計画段階だけ。実行段階での中間管理は、Claude Codeのアーキテクチャではオーバーヘッドにしかならない。

### 問題4: .doc-sync.md の追跡方式が脆弱

**現状**: コミットの短縮ハッシュ（7文字）を `.doc-sync.md` に記録し、grepで存在チェック。

**何が問題か**:
- `git rebase` / `git commit --amend` / `git squash` でハッシュが変わると、記録が全て無効になる
- 短縮ハッシュの衝突リスク（小さいが、プロジェクトが大きくなると増える）
- `.doc-sync.md` 自体がコミット対象なので、「設計書を更新→記録を追記→コミット→そのコミット自体の記録は？」という再帰問題がある
- grepベースのチェックは「ハッシュが文字列として存在するか」しか見ておらず、実際に適切に同期されたかは保証しない

**改善案**: 追跡の粒度をコミット単位からプッシュ単位に変更する。

```markdown
## 代替案: プッシュ時チェックに簡素化

- コミット時: `/update-docs` で設計書を更新（従来通り）
- プッシュ前: 設計書と実装の差分チェック（`/sync-check` の軽量版）
- `.doc-sync.md` は廃止し、設計書の改訂履歴のみで追跡

理由: 個別コミットの追跡は運用コストに対してリターンが小さい。
プッシュ時点で「設計書と実装が一致しているか」が担保できれば十分。
```

### 問題5: AGENTS.md の内容と名前の乖離

**現状**: `AGENTS.md` の内容はNext.jsの破壊的変更に関する注意書き1行のみ。

```markdown
# This is NOT the Next.js you know
This version has breaking changes...
```

**何が問題か**:
- ファイル名から「エージェント定義」を期待するが、実際はNext.jsの注意書き
- `CLAUDE.md` から `@AGENTS.md` で参照されているため、毎回コンテキストに読み込まれる
- エージェントの定義は `.claude/agents/` にあるので、このファイルの存在意義が不明確

**改善案**: 2つの選択肢。

- **案A**: `AGENTS.md` を廃止し、Next.jsの注意書きは `CLAUDE.md` に直接記載する（2行で済む）
- **案B**: `AGENTS.md` をエージェント一覧として再定義し、各エージェントの概要・使い分けガイドを記載する（`.claude/agents/` の詳細定義へのナビゲーション）

案Aを推奨。`@AGENTS.md` で参照する意味がある量の情報がないなら、ファイルを分ける必要がない。

### 問題6: settings.local.json の許可範囲が広すぎる

**現状**: テンプレートに `Bash(rm:*)`, `Bash(bash:*)` などが含まれている。

**何が問題か**:
- `Bash(bash:*)` は事実上あらゆるコマンドを実行可能にする（`bash -c "任意のコマンド"`）
- `Bash(rm:*)` は `rm -rf src/` のような操作を許可する（deny に `/`, `~`, `C:` はあるが、プロジェクトディレクトリの削除は防げない）
- テンプレートとして配布する場合、ユーザーが許可範囲を理解せずそのまま使う可能性が高い
- `settings.local.json`（.gitignore対象）なのでテンプレートとして配布されない可能性もあるが、READMEや手順書で「この設定をコピーせよ」と案内する場合は問題

**改善案**: テンプレートとして推奨する設定は保守的にする。

```json
{
  "permissions": {
    "allow": [
      "Read", "Write", "Edit", "Glob", "Grep",
      "Bash(git:*)", "Bash(npm:*)", "Bash(npx:*)", "Bash(node:*)",
      "Bash(ls:*)", "Bash(mkdir:*)", "Bash(cp:*)", "Bash(mv:*)",
      "WebFetch", "Agent"
    ]
  }
}
```

`rm`, `bash`, `sed`, `cat` 等は必要時にユーザーが都度許可する方が安全。

### 問題7: エージェント定義の情報量が過剰

**現状**: 各エージェント定義ファイルが非常に長い（TypeScriptのinterface定義、mermaid図、詳細なチェックリスト等を含む）。

**何が問題か**:
- エージェント起動時に全文がコンテキストに読み込まれる
- coding-specialist と code-reviewer で重複するチェック項目が多い（any型禁止、UI実装チェック等）
- TypeScriptのinterface風の擬似コードは、Claude Codeが実際に参照するわけではない（人間向けのドキュメントならわかるが、AI向けなら自然言語の方が効率的）
- トークンコストに直結する

**改善案**:
- 共通のチェック項目は `CLAUDE.md` のコーディングルールセクションに1箇所で定義し、エージェント定義からは「CLAUDE.mdのルールに従うこと」と参照する
- TypeScript interface風の擬似コードは削除し、自然言語のリストに置き換える
- 各エージェント定義は**そのエージェント固有の振る舞い**のみに絞る（目安: 100行以内）

### 問題8: テンプレートとしての汎用性が低い

**現状**: Next.js 16 + React 19 + TailwindCSS 4 + Zustand 5 + Prisma 6 + NextAuth 4 + AI機能を前提としたテンプレート。

**何が問題か**:
- AI機能を使わないプロジェクトでも `08_ai_prompt_design.md` が存在する
- 認証を使わないプロジェクトでも `03_authentication_guide.md` が存在する
- 特定のスタック（Zustand, Prisma, NextAuth）に強く依存した設計書が「テンプレート」として含まれている
- プロジェクトの規模感（個人開発？チーム開発？）が想定されていない

**改善案**: レイヤー構造にする。

```
テンプレート構成:
├── core/          # 必須（CLAUDE.md, 基本スキル, 基本エージェント）
├── stack/         # スタック別（prisma/, nextauth/, zustand/ 等）
└── optional/      # オプション（ai/, design-system/ 等）
```

実際にはディレクトリを分けなくても、READMEで「AI機能を使わない場合は `08_ai_prompt_design.md` を削除してください」等のガイドがあれば十分。

### 問題9: スキル間の依存関係が暗黙的

**現状**: `/code-review` → `/build-check` → `/update-docs` の順序は `CLAUDE.md` に書かれているが、各スキル自体はこの依存を知らない。

**何が問題か**:
- ユーザーが `/update-docs` を単体で実行した場合、コードレビュー未実施の状態で設計書が「同期済み」になる
- 順序を間違えても警告がない
- 「省略禁止」のルールはCLAUDE.mdを読んだ場合のみ有効で、スキル単体では強制力がない

**改善案**: 2つのアプローチ。

- **案A（軽量）**: 各スキルの冒頭に前提条件チェックを追加（例: `/update-docs` 実行時に `docs/reviews/` に当日のレビューファイルが存在するか確認）
- **案B（統合）**: `/ship` のような統合スキルを作り、全ステップを順番に実行する。個別スキルは上級者向けとして残す

案Bを推奨。テンプレートの利用者が毎回正しい順序を覚える負担を減らせる。

---

### 問題10: 機能設計書のライフサイクル管理が複雑

**現状**: ステータス5種類、`completed/` と `pending/` への移動、`/update-docs` での自動判定。

**何が問題か**:
- `pending/`（保留タスクあり）の存在意義が薄い。保留理由はファイル内に書かれるので、ディレクトリで分ける必要がない
- 「全タスク完了判定」のロジック（⏸️保留は未完了とみなさない）が直感に反する
- ディレクトリ移動はgitの差分を汚す

**改善案**:
- ディレクトリ移動を廃止し、メタ情報テーブルのステータスのみで管理
- `completed/` も `pending/` も不要。ファイル名の日付でソートすれば十分
- どうしても分類したければ、一覧のMarkdownファイルにリンクを分類して書く方がgit-friendly

## まとめ: 優先度順の改善リスト

| 優先度 | 問題 | 改善案 | 効果 |
|--------|------|--------|------|
| **高** | プロセスが重すぎる | 変更規模別の軽量パスを用意 | 導入障壁の大幅低下 |
| **高** | 設計書の二重管理 | 役割分離（方針 vs 実態） | 同期コスト半減 |
| **高** | エージェント定義が過剰 | 共通ルールの一元化、100行以内に圧縮 | トークンコスト削減 |
| **中** | PM エージェントが非実用的 | 削除またはPlanスキルに再定義 | 構成の簡素化 |
| **中** | .doc-sync.md が脆弱 | プッシュ時チェックに簡素化 | 運用の安定化 |
| **中** | スキル間依存が暗黙的 | `/ship` 統合スキルの導入 | ユーザー負担の軽減 |
| **中** | AGENTS.md の名前と内容の乖離 | 廃止してCLAUDE.mdに統合 | 構成の明確化 |
| **低** | settings.local.json が寛容すぎる | 推奨設定を保守的に | セキュリティ向上 |
| **低** | テンプレートの汎用性 | オプション機能のガイド追記 | 適用範囲の拡大 |
| **低** | 機能設計書のライフサイクル | ディレクトリ移動の廃止 | git履歴の簡素化 |

## 結論

このテンプレートの最大の価値は「設計書とコードの同期を仕組みで担保する」という思想にある。これは多くのプロジェクトが諦める部分であり、Claude Codeのスキル機構を使って実現しようとするアプローチは評価できる。

一方、最大のリスクは**プロセスの重さによる形骸化**。9ステップの必須フローを毎回実行するのは現実的でなく、テンプレートとして配布した場合「重すぎて使われない」か「面倒で全部スキップされる」のどちらかになりやすい。

**最も効果的な改善は「変更規模に応じたフロー分岐」の導入**。これだけで、テンプレートの実用性が大幅に向上する。
