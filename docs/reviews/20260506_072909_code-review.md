## 安全性・正確性の確認

### 肯定的に確認できた点

- **migration.sql の安全性**: INSERT 前に User 存在確認（FK 制約）、listId バックフィルはサブクエリで userId 一致のみ参照 → 他ユーザーへの誤フィルはない。NOT NULL 制約は step 5 でバックフィル完了後に追加 → 順序正しい
- **partial unique index**: `WHERE "system" = TRUE` の WHERE 句が正しく `userId` 単位の制約になっている
- **userId 所有権チェック**: PUT upsert ループ、PUT delete ループ、merge の非 system upsert ループで全件 `existing.userId !== userId` チェックあり
- **T6 pending 分離**: `pendingListUpsertIds` / `pendingListDeleteIds` が独立した Set で管理され、相互に除名するロジックが実装されている（`markListUpsert` で deletes から除名、`markListDelete` で upserts から除名）
- **useInitialMerge の 2 フェーズ**: lists merge 完了 → `remapListIds` 実行 → `useShoppingStore.getState().items.map(itemToDTO)` で remap 済みの最新 items を再取得して merge に渡している（L107 が remap 後の `getState()` 呼び出し）
- **addItem/addItems の重複判定**: `item.listId === listId && item.name === trimmed && item.scope === scope && item.status === 'PENDING'` でリストスコープの重複チェックを実施
- **applyServerChanges の listId**: `ShoppingItem.listId` は always present（非 optional）なので、サーバーから受信した items は常に listId を持つ構造
- **migrateToV3 の冪等性**: `isAlreadyV3` フラグにより 2 回目以降はスキップ（ただし C6 の余分な localStorage アクセスあり）
- **編集モードと移動モードの排他**: `editMode ON → exitMoveMode()` / `enterMoveMode → editMode を false に` の双方向 useEffect で完全排他。`SortableItemRow.useSortable` でも `disabled: ... || moveMode` でドラッグ無効化
- **システムリスト保護**: PUT upsert で `if (input.system) continue`、PUT upserts の system チェック、merge でも `if (input.system) continue`、delete で `if (list.system)` → rejected でサーバー側の多層保護
- **GET のサイドエフェクトなし**: T9 対応済み。GET では未分類自動生成を行わない
