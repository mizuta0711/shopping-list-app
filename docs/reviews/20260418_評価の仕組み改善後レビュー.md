## 改善後の良い点

### 1. 段階的アプローチの体現

前回レビューで推奨した「Phase 1 で取り込み、Phase 2 以降は実運用の課題を見てから判断」を、**仕組み自体が体現**している。

- `report.md` の「テスト手順の記録」テーブル形式は、後から `.spec.ts` に機械的に変換しやすい
- 自然言語の操作手順 + アサーションという形式は、Claude Code が Playwright API に翻訳する際の良い入力になる
- Phase 2 で必要と判断された時点で、蓄積された `report.md` を元にテストコードを生成できる

この設計により、**今すぐ `.spec.ts` を書く複雑さを回避しつつ、将来の選択肢を残す**ことができている。

### 2. テンプレート適合性が大幅に向上

前回は skillup_mock 固有の15画面分のチェックリストがあり、テンプレート化に大幅な書き換えが必要だった。改善後は:

- `browser-test-checklist.md` が汎用的な共通項目（表示/レイアウト/レスポンシブ/インタラクション）のみ
- 画面固有の内容はすべて機能設計書の「4. ブラウザ評価計画」に集約
- **役割分離（汎用 vs 機能固有）がクリーンに**なった

テンプレートにそのまま取り込める形になっている。

### 3. エージェント・スキルの責任分担が明確化

| 主体 | 責務 |
|------|------|
| `/new-feature` | 機能設計書作成時に評価計画の記入を促す |
| `/design-review` | 評価計画のレビュー（8観点）、未記入時は草案生成 |
| `/browser-test` | テスト対象の特定、browser-tester への委譲、**report.md の保存責任** |
| `browser-tester` | ブラウザ操作の実行、スクリーンショット撮影、report.md 作成 |

エージェントがスキルを呼び、スキルが別のエージェントを呼ぶといった**ネストした委任構造が解消**されている。

### 4. フォールバック設計の思慮深さ

問題6への対応として、`browser-tester` が report.md を保存し忘れても `/browser-test` スキル側で検知・補完する2段構えの設計になっている。

- Step 5-1: report.md の存在確認
- Step 5-2: 存在しない場合は browser-tester の返答からメインエージェントが自力で作成
- Step 5-3: 存在する場合も不足をチェックして追記

**「AI は指示を忘れることがある」という前提に立った現実的な設計**として評価できる。

### 5. `/code-review` 警告の具体化

前回は単なる警告文言だったが、改善後は「レビュー結果ファイルの末尾に付記する」という**レビュー成果物への痕跡**として残る形に変わった。これにより:

- レビューを見返した時に「ブラウザテスト必要」が視覚的に残る
- コミット前の最終確認でリマインドとして機能する
- フロー順序変更と併せて実効性が大幅に上がっている

## 残る軽微な懸念

前回指摘の全問題は解消されているが、新たに気付いた軽微な点が1つある。

### 懸念: テスト手順の記録形式が「完璧ではないが許容範囲」

`report.md` の「テスト手順の記録」テーブルは以下の形式:

```
| # | 画面 | 操作（自然言語） | アサーション | 結果 |
```

これは Phase 2 での `.spec.ts` 変換を意識した良い形式だが、以下の情報が欠けている:

- **前提状態**: どのデータ状態を想定したテストか（初期データ/空データ/大量データ等）
- **セレクタのヒント**: `data-testid` や `aria-label` など、AI が Playwright API に変換する際に必要な識別子

**ただし、これは Phase 2 で導入する際に補完すればよい**情報であり、今の段階で入れると過剰設計になる。**Phase 1 としての完成度には影響しない**ため、将来の改善メモとして記録しておくに留める。

## Phase 2 への引き継ぎ事項

将来 `.spec.ts` 自動生成を検討する際の判断材料として、以下を記録しておく:

### Phase 2 導入の発動条件（変更なし）

- クロス機能の回帰バグが実際に発生した
- フルテストのトークンコストが月 $20〜30 を超える
- クリティカルパス（ログイン、決済、主要CRUD等）が明確になった

### Phase 2 実装時の改善アイデア

1. **`report.md` の記録形式を拡張**
   - 前提状態（データ状態）の列を追加
   - セレクタのヒント（`data-testid` 等）の列を追加
   - これにより `.spec.ts` への変換精度が向上

2. **`data-testid` 付与ルールの整備**
   - `.claude/01_development_docs/` に「テスタビリティのための命名ルール」を追加
   - 主要コンポーネントに `data-testid` を付ける運用を先行導入

3. **変換ツールの設計方針**
   - 全自動ではなく、**クリティカルパスのみ** Claude Code が `report.md` を元に `.spec.ts` を生成
   - 生成されたテストは人間がレビューしてからコミット

4. **CI 統合**
   - `npx playwright test tests/e2e/` を GitHub Actions 等で実行
   - MCP は開発時のみ、CI は純粋な Playwright で高速化
