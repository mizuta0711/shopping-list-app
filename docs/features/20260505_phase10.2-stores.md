## 1. 概要

### 背景

MVP リリース後に観察してきた「店舗別リスト要望」が、`docs/設計書/技術負債と将来課題.md` FUTURE-1 の閾値を満たしたため Phase 10.2 として着手する（**着手判断はユーザー判断 / Stage 2 着手前に技術負債.md の FUTURE-1 状態欄を「✅ Phase 10.2 で対応中」に更新する**）。

ユーザーは「スーパー」「薬局」「100均」「コンビニ」等、買う場所ごとに買い物計画を分けたいケースが多い。現在は単一リスト + 「今日 / また今度」スコープのみで、複数店舗を同時に管理しようとすると以下の問題が起きる：

- すべてのアイテムが同じビューに並び、店舗ごとの見通しが悪い
- スーパー用と薬局用が混ざり、レジ前で目的のアイテムを探しにくい
- 「今日」スコープを店舗代用に流用するユーザーがいるが、本来の用途（時期判断）と衝突する

### 目的

- ユーザーが**任意の名前 + 絵文字でリストを作成**し、買い物リストを店舗ごとに分けて管理できる
- メイン画面のヘッダー直下にあるリストタブから**ワンタップでアクティブリストを切り替え**できる
- 各リスト内では従来どおり「今日 / また今度」スコープ・並び替え・セット追加・編集モード・購入チェックが機能する
- ログイン済みユーザーはリスト構成と所属アイテムが**複数端末間で同期**される（Phase 9 + Phase 10.1b 同等のスキームで横展開）

### 方針（Stage 1 確定: 8 件の初期判断 + レビュー指摘を 2026-05-06 に対話で確定）

| 観点 | 判断 |
|---|---|
| リスト切替 UI | ヘッダー直下に**水平スクロール可能なピル風タブバー**（ScopeTabs と類似のスタイル）。タップで即切替 |
| **システムリスト「未分類」** | 🗂️ **「未分類」を system-managed リストとして常に存在させる**（削除・改名不可、絵文字固定）。ユーザー作成リストの削除時、所属アイテムは未分類へ自動移動。既存ユーザーのマイグレーション先も未分類 |
| 初期アクティブリスト | **未分類**（全リストはアグリゲート view のため add/edit セマンティクスが曖昧になる。メイン画面には配置しない） |
| リスト数上限 | ユーザー作成リスト **最大 20 件**（未分類はカウント外） |
| リスト名 | 1〜20 文字。未分類は固定文言で改名不可 |
| 絵文字 | ユーザーリスト: 任意（未指定なら 🛒 デフォルト）。**プリセット 12 個から選択**。未分類は 🗂️ 固定 |
| 新規リスト追加 UI | **`/lists/new` ページ遷移**（既存 `/sets/new` と統一） |
| アクティブリスト永続化 | **LocalStorage に保存**し、再起動・タブ切替後も維持。**端末ローカルでクラウド同期しない** |
| リスト切替時の編集モード | **OFF にリセット**（コンテキスト変更で誤操作防止） |
| **タブバーの表示条件 (PA-2)** | **ユーザー作成リストが 0 件**（未分類のみ）の状態では**タブバー全体を非表示**にし、画面圧迫を回避。1 件目のリスト作成と同時に出現 |
| セットとの関係 | **セットは全リスト共通**（リスト固有のセットを持たない）。セット追加時はアクティブリストに入る。未分類アクティブ時もセット追加可 |
| 履歴画面のリスト切替 (C2) | 履歴画面の**タブバー先頭に「全リスト」タブを配置**（全リストの PURCHASED アイテムをアグリゲート表示）+ それに続けて [未分類] [ユーザーリスト...] |
| 削除時のアイテム扱い (C1) | ユーザーリスト削除時、**所属アイテムは未分類へ自動移動**（物理削除なし）。リスト本体のみ tombstone 発行。confirm ダイアログで「N 件のアイテムを未分類に移動してリストを削除します」と明示 |
| クラウド同期 | リスト本体（`ShoppingList` テーブル）を Phase 10.1b と同パターンで LWW 同期。アイテムは `listId` 追加で対応 |
| クラウド DB の `listId` 互換戦略 | **`ShoppingItem.listId` は NOT NULL**。サーバー側で旧クライアント（Phase 10.2 未適用）の push に対し、`listId === undefined` の場合は**未分類** ID を補完して受け入れる |
| 未分類リストの ID 同期戦略 | 未分類は端末ごとに**独立した UUID** を持つ（端末初回起動時にクライアント発行）。クラウド同期で `system: true` フラグを LWW 衝突解決時に**保護**（system 同士の merge は createdAt が古い方を採用）。複数端末ログイン時、初回マージで「未分類」が両端末分作成されるリスクは Stage 2 で merge ロジックにより 1 件に集約 |
| データモデル変更 | `ShoppingItem.listId` 追加（破壊的）。`ShoppingList.system: boolean` 追加。LocalStorage v2 → v3 マイグレーション、書き込み前に旧データをバックアップ |
| BottomNav との関係 | 既存 4 タブ（ホーム/履歴/セット/設定）は据え置き。リスト切替は**ヘッダー直下のリストタブバー**で完結 |
| 設定画面の変更 | 設定画面に「リスト管理」リンクのみ追加。リスト管理 UI は独立ページ `/lists` |
| `/lists` ページのレイアウト | セット一覧画面 (`/sets`) と同じレイアウトパターン。未分類は最上部に固定表示 + 削除/改名ボタンを `disabled` |
| **初回アップデート時のコーチマーク (PA-5)** | 既存ユーザーが Phase 10.2 適用版に初めて起動した時、画面中央に**簡易バナー 1 枚**「店舗ごとにリストを作成できるようになりました。設定 → リスト管理から追加できます」を表示。閉じるボタンタップで非表示・以後表示しない（LocalStorage に表示済みフラグ） |
| **アイテムのリスト間移動（移動モード）** | 編集モードとは別の独立モードとして「**移動モード**」を導入。ヘッダーに **`FolderInput` アイコン**を追加し、タップで起動。移動モード中は行にチェックボックスが現れ、複数選択 → 決定ボタンで移動先 BottomSheet → 移動先タップで即時移動。**編集モードと完全排他**（同時 ON 不可、互いに disabled） |
| 移動モードの起動条件 | **ユーザー作成リストが 1 件以上**ある場合のみ enabled（未分類のみのときは移動先がないため disabled） |
| 移動モードでの Undo | **なし**（Phase 13 アンドゥ撤廃方針と整合）。誤操作時は移動先で再選択して元リストへ戻すことで対応 |
| 移動モードのスコープ切替時の挙動 | **選択は維持**（今日タブで 2 件選択 → また今度タブに切替 → 別途 1 件選択 → 計 3 件で決定可）|

## 3. 機能・画面設計（Stage 1）

### 3-1. 対象画面と操作フロー

| 画面 | URL | 新規/変更 | 概要 |
|------|-----|----------|------|
| メイン画面 | `/` | 変更 | ヘッダー直下に**リストタブバー**追加 + ヘッダーに **移動モードアイコン (`FolderInput`)** 追加。アクティブリストの items のみを表示 |
| リスト一覧 | `/lists` | 新規 | ユーザー作成のリスト一覧。新規追加・並び替え（並び替えは Stage 2 で要否判断） |
| リスト編集 | `/lists/new`, `/lists/[id]` | 新規 | リスト名 + 絵文字（任意）の編集。削除も含む |
| 履歴画面 | `/history` | 変更 | リストタブバーを追加し、アクティブリストの履歴のみ表示 |
| 設定画面 | `/settings` | 変更 | 「リスト管理」リンク（`/lists` への導線）追加 |
| セット選択シート | `SetPickerSheet` | 変更 | 追加先は**現在のアクティブリスト**になる（セット自体はリスト共通） |
| 移動先選択シート | `ListMoveSheet`（新規） | 新規 | 移動モード中の決定ボタンタップで開く BottomSheet。ユーザー作成リスト + 未分類から移動先を選択 |

#### 操作フロー A: 既存ユーザーが Phase 10.2 にアップデート

1. アプリ起動 → LocalStorage v2 検出 → v3 マイグレーション開始
2. 旧データを `shopping-list-app:state-backup-v2` にコピー（マイグレーション失敗時の復元用）
3. **「未分類」🗂️ システムリスト**（`system: true`）を新規作成し、既存アイテム全件に `listId = 未分類の id` を付与
4. アクティブリスト = 未分類
5. **コーチマーク 1 枚**「店舗ごとにリストを作成できるようになりました。設定 → リスト管理から追加できます」を画面中央に表示。閉じるボタンで非表示・以後表示しない
6. ユーザー作成リストが 0 件のため**タブバーは非表示**（PA-2）。UI は従来とほぼ同じ見た目で、AddItemForm から既存通り追加可能（追加先は未分類）

#### 操作フロー B: 新規リスト作成

導線は **2 系統**:
- **メイン画面のリストタブバー末尾 sticky 「+」ボタン** （タブバー表示中の場合のみ。横スクロール時も画面右端に固定して発見性を確保 / R8/P3 対応）
- **設定画面 → リスト管理 (`/lists`) → ヘッダー右上「+」ボタン**

1. 上記いずれかから `/lists/new` へ遷移
2. リスト名入力（1〜20 文字） + 絵文字選択（プリセット 12 個 / 任意） → ヘッダー右上の Check（保存）ボタン
3. 自動的に新リストがアクティブになり、メイン画面に戻る
4. **タブバーが初めて出現する場合は控えめなフェードイン演出** で気づきやすくする（実装は Stage 2）
5. 新リストは items 0 件の状態。AddItemForm から追加可能

#### 操作フロー C: リスト切替

1. メイン画面のリストタブバーから該当タブをタップ
2. アクティブリスト state を変更し、メイン画面のアイテム一覧と「今日 / また今度」カウントが切り替わる
3. アクティブリストは LocalStorage に永続化（再起動後も維持）
4. リスト切替時、編集モードは OFF にリセット（誤操作防止）

#### 操作フロー D: リスト編集

1. `/lists` 画面で該当リスト行をタップ → `/lists/[id]`
2. リスト名・絵文字を編集 → 「保存」 → `/lists` に戻る
3. 削除ボタンでリスト削除（後述 E）

#### 操作フロー E: リスト削除（C1 確定: 物理削除なし / 未分類へ移動）

1. `/lists/[id]` で削除ボタンタップ（**未分類リストでは削除ボタン disabled**）
2. `window.confirm`「『◯◯』を削除します。所属する N 件のアイテムは未分類リストに移動されます。よろしいですか？」
3. OK で:
   - 所属アイテムの `listId` を**未分類リストの ID に書き換え**（物理削除なし、アイテム自体は残る）
   - リスト本体のみ削除して `SetDeletionTombstone` 相当を発行
4. アクティブリストが削除された場合は、**未分類**を自動アクティブ化
5. 削除直後にトーストで「『◯◯』を削除しました。N 件のアイテムを未分類に移動しました」を表示

#### 操作フロー F: アイテム操作（既存のフロー）

- 各リスト内では既存の操作モデルが**そのまま機能**：
  - チェック切替（タップ）
  - 「また今度」移動
  - 編集モード（鉛筆ヘッダー）→ 編集 / 削除 / 並び替え
  - スコープタブ（今日 / また今度）
  - セットからの追加（アクティブリストに入る）

#### 操作フロー H: 移動モード（アイテムのリスト間移動）

ユーザー作成リストが 1 件以上ある場合のみ起動可能な、編集モードと**完全排他**の独立モード。

```
[通常メイン画面]
  ↓ ヘッダーの「移動」アイコン (FolderInput) をタップ
[移動モード ON / 0 件選択]
  ヘッダー右上 = [キャンセル]、各行の左端にチェックボックスが出現
  ↓ アイテム複数タップ
[移動モード ON / N 件選択 (N >= 1)]
  ヘッダー右上 = [決定 (N)]、選択行は背景 emerald-50
  ↓ 決定タップ
[ListMoveSheet 表示]
  「N 件のアイテムをどこへ移動？」見出し + 移動先リスト一覧（現在のアクティブリストは除外）
  ↓ 移動先タップ
[即時移動 → モード OFF / トースト「N 件を『◯◯』へ移動しました」]
  選択中 N 件の listId を移動先 ID に書き換え、移動モードを抜ける（アクティブリストは元のまま）
```

**移動モード中の操作制約**:
- アイテム行のタップは「選択トグル」のみ（チェック切替・また今度移動・編集・削除はすべて無効化）
- スコープタブ（今日/また今度）の切替は可能。**選択は維持**（タブをまたいで複数選択して一括移動可）
- リストタブの切替・編集モードボタンは disabled（モード排他）
- 「キャンセル」タップ / `Escape` キー / ListMoveSheet を閉じる動作で選択をクリアしモード解除
- ListMoveSheet 内では現在のアクティブリストはリストから除外（自分自身への移動は無意味）
- ユーザー作成リストが 0 件のとき（=未分類のみ）はヘッダーの移動モードアイコンが disabled（移動先がない）

#### 操作フロー G: 履歴画面でのリスト切替（C2 確定: 「全リスト」タブを先頭追加）

履歴画面のタブバーは以下の構成（メイン画面とは異なる）:

```
[全リスト] [🗂️ 未分類] [🛍️ スーパー] [💊 薬局] ...
```

- **「全リスト」タブ（先頭・デフォルト）**: 全リストの PURCHASED アイテムを横断アグリゲート表示。日付グルーピングは現状の実装を維持
- 各リストタブをタップ → そのリスト所属の PURCHASED のみ表示
- 履歴画面のアクティブタブはメイン画面の active list とは**別管理**（履歴閲覧時の都合に最適化）。LocalStorage には保存しない（履歴を開くたびに「全リスト」へリセット）
- アイテムの **復元（未購入に戻す）** や **削除** は所属リスト情報を維持したまま動作（全リスト表示中も同様）

### 3-2. 受け入れ条件（ユーザー目線）

#### 既存機能の維持
- [ ] アップデート後の初回起動で、既存アイテムは**未分類**リストに自動移行されている
- [ ] アクティブリスト = 未分類 の状態で、Phase 10.1 までの全機能（追加・編集・削除・並び替え・スコープ移動・セット追加・編集モード）が従来通り動作する
- [ ] 既存セットは引き続き全リスト共通で利用可能
- [ ] **アップデート初回起動時にコーチマーク 1 枚が表示される**（閉じるボタンで非表示・再表示しない）
- [ ] **ユーザー作成リストが 0 件の間はタブバーが非表示**（既存ユーザーの突然 UI 変化を抑制）

#### リスト管理
- [ ] `/lists` でリスト一覧が表示される（先頭に未分類が固定 + ユーザー作成リスト）
- [ ] `/lists/new` で新規リストを追加できる（名前 1〜20 文字 + 絵文字プリセット 12 個から選択）
- [ ] `/lists/[id]` でリスト名・絵文字を編集できる（**未分類は編集不可・disabled**）
- [ ] 設定画面 →「リスト管理」リンクから `/lists` に遷移できる
- [ ] **ユーザー作成リスト**数が上限（20 件）に達すると新規追加ボタンが disabled

#### リスト切替
- [ ] ユーザー作成リスト 1 件以上で、メイン画面ヘッダー直下にリストタブバーが表示される（水平スクロール可能、ピル風）
- [ ] タブをタップするとアクティブリストが切り替わり、items 一覧・今日/また今度カウントが連動
- [ ] **リスト切替時、編集モードは OFF にリセット**される
- [ ] アクティブリストはアプリ再起動後も維持される
- [ ] タブバー末尾の「+」ボタンは sticky 表示で常に画面右端に見える
- [ ] タブバーは `role="tablist"` + `aria-label="リスト選択"`、ScopeTabs は `role="tablist"` + `aria-label="表示期間"` で AT 区別される

#### 移動モード（アイテムのリスト間移動）
- [ ] ユーザー作成リスト 1 件以上で、ヘッダーの移動アイコン（`FolderInput`）が enabled になる
- [ ] ユーザー作成リスト 0 件（未分類のみ）のときは移動アイコンが disabled
- [ ] 移動アイコンタップで移動モードに入り、各行に選択チェックボックスが現れる
- [ ] 移動モード中、ヘッダー右上が「キャンセル」（0 件選択時）/「決定 (N)」（1 件以上選択時）に切り替わる
- [ ] 移動モード中はチェック切替・編集モード・行削除など他の行操作が無効
- [ ] 移動モード中はスコープタブを切り替えても**選択状態が維持される**（今日 + また今度を横断して複数選択可）
- [ ] 決定タップで `ListMoveSheet` が開き、現在のアクティブリストを除く全リスト（未分類含む）が表示される
- [ ] 移動先タップで選択アイテム N 件の `listId` が書き換わり、トースト「N 件を『◯◯』へ移動しました」が表示される
- [ ] 移動完了後は移動モードが自動 OFF になり、アクティブリスト自体は変わらない
- [ ] **編集モードと移動モードは同時 ON 不可**。一方が ON のとき他方のアイコンは disabled
- [ ] キャンセル / シート閉じで選択をクリアしモード解除
- [ ] 移動モードでは Undo は提供しない（Phase 13 アンドゥ撤廃方針と整合）

#### リスト削除（未分類への移動方式）
- [ ] 削除時に確認ダイアログ「『◯◯』を削除します。所属する N 件のアイテムは未分類リストに移動されます」が表示される
- [ ] OK でリストのみ削除され、**所属アイテムは未分類に移動**（物理削除されない）
- [ ] 削除後にトースト「『◯◯』を削除しました。N 件のアイテムを未分類に移動しました」が表示される
- [ ] 削除されたリストがアクティブだった場合、未分類が自動アクティブ化される
- [ ] **未分類リストは削除ボタンが disabled**

#### 履歴画面（C2 確定）
- [ ] 履歴画面にも同じタブバー構造が表示される
- [ ] **タブバー先頭に「全リスト」タブ**が固定で配置され、デフォルトでアクティブ
- [ ] 「全リスト」タブでは全リストの PURCHASED アイテムが横断アグリゲート表示される
- [ ] 各リストタブをタップでそのリストの履歴のみ表示
- [ ] 履歴のタブ状態はメイン画面と独立（履歴を開くたびに「全リスト」へリセット）

#### クラウド同期
- [ ] ログイン済みで端末 A にリストを作成 → 1.5s 後に端末 B にも同期される
- [ ] 端末 A でアイテムをリスト B に追加 → 端末 B でも同じリストに表示
- [ ] リスト削除時、tombstone が伝播し他端末でもリスト+アイテムが消える
- [ ] アクティブリスト state は端末ごとに独立（同期しない）

#### マイグレーション安全性
- [ ] v2 → v3 マイグレーション失敗時、`shopping-list-app:state-backup-v2` から既存データを復元できる
- [ ] マイグレーション実行は冪等（再実行で破壊されない）
- [ ] **マイグレーション中断（途中でブラウザを閉じる）後の次回起動で再試行が走る**（バックアップキーが残っていれば検知して再実行）
- [ ] **同期競合 UX**: 端末 A が削除したリストに端末 B が追加 → 端末 B で受信時、追加分は未分類へ自動振り分け、トースト「『◯◯』は別端末で削除されました。N 件のアイテムを未分類に移動しました」

### 3-4. データモデル変更（概要、Stage 2 で詳細化）

```typescript
// 新規型
type ShoppingList = {
  id: string;          // クライアント発行 UUID v4
  name: string;        // 1〜20 文字
  emoji: string | null; // 任意（プリセット 12 個から選択）
  system: boolean;     // true = 未分類（削除・改名不可）
  createdAt: string;
  updatedAt: string;
};

// 既存型の拡張（破壊的変更）
type ShoppingItem = {
  // ... 既存フィールド
  listId: string;  // 新規追加（必須・NOT NULL）
};
```

LocalStorage:
- `shopping-list-app:state` — version v3 に更新（v2 → v3 マイグレーション）
- `shopping-list-app:lists:state` — 新規（`useListsStore`、v1 開始）
- `shopping-list-app:active-list` — 新規（アクティブリスト ID のみ保持、軽量）
- `shopping-list-app:state-backup-v2` — マイグレーション前のバックアップ

DB（PostgreSQL）:
- `ShoppingList` テーブル新設（クラウド同期対象、`system: boolean` カラム含む）
- `ShoppingListDeletionTombstone` テーブル新設
- `ShoppingItem` に `listId` カラム追加（NOT NULL）。migration 時は既存行に未分類 ID を埋めて NOT NULL 化

詳細は Stage 2 で確定する。

### 3-5. エラー・例外シナリオ

| シナリオ | 期待動作 |
|---|---|
| LocalStorage v2 → v3 マイグレーション失敗 | エラー時は `shopping-list-app:state-backup-v2` から旧データを読み戻し、トーストで「データ移行に失敗しました。再起動してお試しください」を表示。アプリは v2 互換モードで継続動作 |
| マイグレーション中断（途中でブラウザを閉じる） | 次回起動時にバックアップキー (`shopping-list-app:state-backup-v2`) と未完了フラグの両方を検知し、**マイグレーションを冪等に再試行**。再試行も冪等で何度実行しても破壊しない |
| マイグレーション再試行も失敗 | バックアップ値で v2 互換モード継続。設定画面に「データ整合性の確認」リンクを追加し、ユーザーが手動で再試行できるようにする（Stage 2 で UI 確定） |
| アクティブリスト ID が存在しない（削除済み等） | **未分類**を自動アクティブ化（必ず存在するため fallback が安全） |
| すべてのリスト（未分類含む）が消失（DB 異常等） | 起動時に検知し、**未分類**を自動再作成。orphan アイテムも未分類へ |
| クラウド同期で `listId` 未指定のアイテムが返ってきた（旧クライアント送信） | サーバー側で受信時に**未分類**ID を補完して保存（DB 制約違反を防ぐ）。クライアントは保存済みの listId 付きでレスポンスを受け取る |
| リスト名重複 | 許容（同名リスト OK。区別は ID で行う） |
| 削除中のリストにアイテム追加が走った（同期競合） | tx 内で削除を優先、後続の追加は **未分類**へ自動振り分け。クライアント側でトースト「『◯◯』は別端末で削除されました。N 件のアイテムを未分類に移動しました」 |
| 端末 A と端末 B で別々に「未分類」が作成された（初回同期前に両方で操作） | merge 時に `system: true` のリストが複数あれば、`createdAt` が**古い方を採用**してもう一方の所属アイテムを移動。Stage 2 でロジック詳細化 |

#### 4-2-1. Prisma スキーマ変更

```prisma
/// 買い物リスト（Phase 10.2 で追加。ユーザーが任意に作成する複数リストの実体）
/// id はクライアント発行 UUID。updatedAt は LWW 判定キー（クライアント発行）
/// system: true は「未分類」固定リスト。クライアント側 / API 側双方で削除・改名を弾く
model ShoppingList {
  id        String   @id /// クライアント発行 UUID v4
  userId    String   /// 所有ユーザー
  name      String   /// リスト名（最大 20 文字、未分類は固定文言）
  emoji     String?  /// 絵文字（任意、未分類は 🗂️ 固定）
  system    Boolean  @default(false) /// true = 未分類（システム管理）
  createdAt DateTime /// クライアント発行のタイムスタンプ
  updatedAt DateTime /// LWW 判定キー（クライアント発行）

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  items ShoppingItem[]

  @@index([userId, updatedAt])
  @@index([userId, system])
}

/// リスト削除イベントのトラッキング（Phase 10.2）
/// アイテム所属の listId は削除前にサーバー側で未分類へ書き換えて保持する
model ShoppingListDeletionTombstone {
  id        String   @id @default(uuid())
  userId    String   /// 削除を行ったユーザー
  listId    String   /// 削除された ShoppingList の id
  deletedAt DateTime @default(now()) /// サーバー時刻

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, listId])
  @@index([userId, deletedAt])
}

/// 買い物アイテム（Phase 10.2 で listId 追加）
model ShoppingItem {
  id          String    @id
  userId      String
  listId      String    /// 所属リスト ID（NOT NULL、新規追加）
  name        String
  scope       String
  status      String
  order       Int
  createdAt   DateTime
  updatedAt   DateTime
  purchasedAt DateTime?

  user User           @relation(fields: [userId], references: [id], onDelete: Cascade)
  list ShoppingList   @relation(fields: [listId], references: [id], onDelete: Restrict)

  @@index([userId, updatedAt])
  @@index([userId, scope, status])
  @@index([userId, listId, scope, status]) // リスト別フィルタ用に追加
}

model User {
  // ...
  shoppingLists      ShoppingList[]
  listTombstones     ShoppingListDeletionTombstone[]
}
```

**重要な制約**:
- `ShoppingItem.list` は `onDelete: Restrict`。**リスト削除時にアイテムが残っていれば DB レベルでブロック**される。サーバー側はリスト削除前に必ず所属アイテムの `listId` を未分類へ書き換える tx を組む（誤って物理削除しない安全弁）
- `system: true` のリストは**ユーザー 1 人につき必ず 1 件**を merge ロジックで保証（`@@index([userId, system])` でクエリ高速化）

#### 4-4-2. `PUT /api/sync/lists`

リクエスト（`ListsSyncPushRequest`）:
```json
{
  "upserts": [
    {
      "id": "uuid-1",
      "name": "薬局",
      "emoji": "💊",
      "system": false,
      "createdAt": "2026-05-06T08:00:00.000Z",
      "updatedAt": "2026-05-06T09:00:00.000Z"
    }
  ],
  "deletedIds": ["uuid-2"],
  "since": "2026-05-06T07:00:00.000Z"
}
```

レスポンス: Phase 10.1b の sets 版と同構造。差分:
- `rejected` の `reason` に `"SYSTEM_PROTECTED"` 追加: クライアントが `system: true` のリストを `name`/`emoji` 変更しようとした、もしくは `deletedIds` に system list を含めた場合 → 拒否し `serverList` を返す
- `deletedIds` 処理時、サーバー tx 内で:
  1. 対象リストの `system: true` なら拒否（`rejected`）
  2. `ShoppingItem.update where { listId: target } set { listId: <userの未分類id> }`
  3. `ShoppingList.delete`
  4. `ShoppingListDeletionTombstone.upsert`
- `upserts` で `system: true` を新規送信されてもサーバーは無視（既存の system list との重複防止）

実装ロジック擬似コード:
```ts
await prisma.$transaction(async (tx) => {
  const unclassified = await ensureUnclassifiedList(tx, userId);
  for (const id of deletedIds) {
    const list = await tx.shoppingList.findUnique({ where: { id } });
    if (!list || list.userId !== userId) continue;
    if (list.system) {
      rejected.push({ id, reason: "SYSTEM_PROTECTED" });
      continue;
    }
    await tx.shoppingItem.updateMany({
      where: { userId, listId: id },
      data: { listId: unclassified.id, updatedAt: new Date() },
    });
    await tx.shoppingList.delete({ where: { id } });
    await tx.shoppingListDeletionTombstone.upsert({
      where: { userId_listId: { userId, listId: id } },
      create: { userId, listId: id },
      update: { deletedAt: new Date() },
    });
  }
  // upserts は LWW
  for (const u of upserts) {
    if (u.system) continue; // クライアントから system list 作成は不可
    const existing = await tx.shoppingList.findUnique({ where: { id: u.id } });
    // 他ユーザーのリストへの上書きを禁止（ID 衝突攻撃防御 / Stage 2 レビュー T3 対応）
    if (existing && existing.userId !== userId) continue;
    if (existing && existing.system) {
      rejected.push({ id: u.id, reason: "SYSTEM_PROTECTED", serverList: listToDTO(existing) });
      continue;
    }
    if (existing && existing.updatedAt >= new Date(u.updatedAt)) {
      rejected.push({ id: u.id, reason: "SERVER_NEWER", serverList: listToDTO(existing) });
      continue;
    }
    await tx.shoppingList.upsert({
      where: { id: u.id },
      // create では userId を強制上書き（クライアント発行 userId を信頼しない）
      create: { id: u.id, userId, name: u.name, emoji: u.emoji, system: false, createdAt: u.createdAt, updatedAt: u.updatedAt },
      // update でも name / emoji / updatedAt のみ。userId は不変
      update: { name: u.name, emoji: u.emoji, updatedAt: u.updatedAt },
    });
    applied.push(u);
  }
});
```

#### 4-4-3. `POST /api/sync/lists/merge`

初回ログイン時（`useInitialMerge`）。リクエスト: `localLists` + `localUnclassifiedId`。

レスポンス（`ListsSyncMergeResponse`）:
- `finalLists`: マージ後のサーバー全リスト
- `unclassifiedId`: サーバー側で正本となる未分類 ID（古い方を採用）
- `remappedUnclassifiedIds`: クライアント側で別 ID として作られていた未分類 ID（クライアントはこれらの所属アイテムを `unclassifiedId` に書き換える）
- `uploadedCount` / `downloadedCount`: Phase 10.1b と同等

実装方針:
```ts
await prisma.$transaction(async (tx) => {
  // 1. 既存サーバー側未分類を取得 / なければ作成
  let unclassified = await tx.shoppingList.findFirst({
    where: { userId, system: true }, orderBy: { createdAt: "asc" },
  });
  if (!unclassified) {
    if (localUnclassifiedId) {
      // クライアントの未分類を採用
      const localUL = localLists.find((l) => l.id === localUnclassifiedId);
      unclassified = await tx.shoppingList.create({
        data: { ...localUL!, userId, system: true },
      });
    } else {
      unclassified = await tx.shoppingList.create({
        data: {
          id: crypto.randomUUID(),
          userId, name: "未分類", emoji: "🗂️", system: true,
          createdAt: new Date(), updatedAt: new Date(),
        },
      });
    }
  }
  // 2. 端末の未分類 ID とサーバーの未分類 ID が異なる場合 → remap 対象
  const remapped: string[] = [];
  if (localUnclassifiedId && localUnclassifiedId !== unclassified.id) {
    remapped.push(localUnclassifiedId);
  }
  // 3. ローカルの非 system リストを LWW upsert
  for (const l of localLists) {
    if (l.system) continue;
    // 既存と LWW 比較 → upsert
  }
  return { finalLists, unclassified, remapped, ... };
});
```

クライアント側 (`mergeListsOnLogin` ハンドラー):
- `remappedUnclassifiedIds` を受け取ったら、`useShoppingStore.items` を走査して `listId` がそれらに該当するアイテムを `unclassifiedId` に書き換える（LocalStorage に保存）
- `useListsStore.lists` から remapped な未分類エントリを削除

#### `reconcileLists` 純粋関数（Stage 2 レビュー対応 / Phase 9・10.1b と対称）

```ts
/**
 * サーバー pull の lists / serverDeletes を、ローカル lists にマージした最終状態を返す純粋関数。
 * テスタブルにするため副作用なしの設計とする。
 */
export function reconcileLists(params: {
  local: ShoppingList[];
  serverChanges: ShoppingList[];
  serverDeletes: string[];
}): { upserts: ShoppingList[]; deletes: string[] } {
  const localById = new Map(params.local.map((l) => [l.id, l]));
  const upserts: ShoppingList[] = [];
  for (const s of params.serverChanges) {
    const existing = localById.get(s.id);
    // LWW: サーバー updatedAt がローカルより新しい時のみ採用
    if (!existing || s.updatedAt > existing.updatedAt) upserts.push(s);
  }
  return { upserts, deletes: params.serverDeletes };
}
```

### 4-9. LocalStorage マイグレーション v2 → v3

#### 手順（`src/features/shopping/migrations/v2ToV3.ts`）

```ts
type V2State = { items: V2Item[]; sort: SortKey; hasOnboarded: boolean };
type V2Item = { id: string; name: string; scope; status; order; createdAt; updatedAt; purchasedAt };
type V3State = { items: V3Item[]; sort: SortKey; hasOnboarded: boolean };
type V3Item = V2Item & { listId: string };

/**
 * v2 形式かどうかの型ガード（Stage 2 レビュー T10 対応）。
 * - object でかつ items 配列を持つ
 * - items[0] に listId フィールドが**ない**（あれば既に v3 化済み）
 * - 空配列の場合は v2 として扱う（マイグレーションを冪等に通すため）
 */
function isV2(persisted: unknown): persisted is V2State {
  if (!persisted || typeof persisted !== "object") return false;
  const s = persisted as { items?: unknown };
  if (!Array.isArray(s.items)) return false;
  if (s.items.length === 0) return true;
  const first = s.items[0] as { listId?: unknown };
  return typeof first.listId !== "string";
}

export function migrateShoppingV2ToV3(persisted: unknown): V3State | undefined {
  if (!isV2(persisted)) return undefined;

  // 1. v2 全データをバックアップ
  try {
    localStorage.setItem(STATE_BACKUP_V2_KEY, JSON.stringify(persisted));
    localStorage.setItem(`${STATE_BACKUP_V2_KEY}:in-progress`, "1");
  } catch {
    // QuotaExceeded 時はバックアップ失敗を許容して通常マイグレーションに進む
  }

  // 2. 未分類 list を ensure（useListsStore 経由）
  const unclassifiedId = useListsStore.getState().ensureUnclassified();

  // 3. 全アイテムに listId = unclassifiedId を付与
  const v3Items: V3Item[] = persisted.items.map((it) => ({ ...it, listId: unclassifiedId }));

  // 4. アクティブリストを未分類に
  useActiveListStore.getState().setActiveListId(unclassifiedId);

  // 5. マイグレーション完了フラグ
  localStorage.removeItem(`${STATE_BACKUP_V2_KEY}:in-progress`);

  return { items: v3Items, sort: persisted.sort, hasOnboarded: persisted.hasOnboarded };
}
```

`useShoppingStore` の `persist({ migrate })` で:
```ts
migrate: (state, version) => {
  if (version < 2) state = migrateToV2(state);
  if (version < 3) state = migrateShoppingV2ToV3(state);
  return state;
}
```

#### 中断時の再試行

起動時に `STATE_BACKUP_V2_KEY` と `:in-progress` フラグの両方が存在 → 中断と判断。`migrate` が再実行されることで冪等に復元（v2 形式が persist にまだ残っていればそのまま v3 化、既に v3 化済みならスキップ）。

#### 失敗時の復元

`migrate` 内で例外発生 → `catch` でトースト表示 + `STATE_BACKUP_V2_KEY` を読み戻して v2 互換モードで起動。設定画面に「データ整合性の確認」リンクを追加（手動再試行 UI は Stage 2 末尾で詳細化）。

```ts
  onSelect: (id: string) => void; // setActiveListId を呼ぶ
  onAddNew: () => void;           // /lists/new へ遷移
  disabled?: boolean;             // 移動モード中は true
};
```

実装:
- 横スクロール `overflow-x-auto scrollbar-hide`
- 未分類タブ → ユーザーリストタブ → 末尾「+」（`position: sticky; right: 0;`）
- ARIA: `role="tablist" aria-label="リスト選択"`、各タブ `role="tab" aria-selected`
- 表示条件: `lists.filter((l) => !l.system).length === 0` のとき**コンポーネント自体を non-render**（PA-2）

### 4-12. エラー処理

| 種別 | 動作 |
|------|------|
| API 401 | `useSyncStore.setStatus("logged_out")` |
| API 5xx | 指数バックオフ（1s / 2s / 4s）リトライ |
| Migration 失敗 | バックアップから読み戻し + 「データ整合性の確認」リンクを設定画面に表示 |
| Migration 中断 | 次回起動時に再試行（冪等） |
| `system: true` への delete / rename | API 側で `rejected (SYSTEM_PROTECTED)` 返却、クライアント側でも UI で disabled |
| 移動モード中の 0 件選択で決定 | 決定ボタン disabled（押下不可） |
| 移動先リストが移動中に削除された（同期競合） | クライアントが存在しない `listId` で items を push → §4-4-4 の listId 補完層で**未分類に自動振替**。クライアント側は次回 `pullListsOnce` で当該 list の `serverDeletes` を受信した時点でトースト「『◯◯』は別端末で削除されました。N 件のアイテムを未分類に移動しました」を表示 |
| ユーザー作成リスト 20 件超過 | 「+」ボタン disabled + トースト「リストは最大 20 件まで作成できます」 |

### 5-2. UX 評価観点

| 観点 | 確認 |
|------|------|
| **発見性** | 新規ユーザーがリスト追加機能に気づけるか（sticky 「+」ボタン の視認性） |
| **タブバーの操作性** | 4 件以上のリストでスムーズに横スクロールできるか、誤タップしないか、sticky 「+」が他タブを覆わないか |
| **学習コスト** | 既存ユーザーが単一リスト運用から複数リスト運用に移行する際の混乱（コーチマーク 1 枚で十分か） |
| **アクティブ状態の視認性** | 多リスト時にどれが active か一目で分かるか（タブのコントラスト + `/lists` の `●` バッジ） |
| **データ消失の不安緩和** | 削除時の confirm 文言「未分類へ移動」が安心感を与えているか |
| **同期競合 UX** | 削除されたリストへのアイテム追加が試行された場合、ユーザーに何が起きたか視覚的に伝わるか |
| **画面圧迫** | リストタブ + ScopeTabs の二段重ねがスクロール領域を不快なほど削っていないか（実機 375px で実測） |
| **コーチマーク** | 表示タイミングが邪魔でないか、文言が機能を正しく説明しているか |
| **移動モードの発見性** | 移動アイコン (`FolderInput`) が「リスト間移動」用と直感的に分かるか。ラベル / aria-label の妥当性 |
| **移動モードの分かりやすさ** | モード ON / OFF の状態がヘッダー右上の「キャンセル / 決定 (N)」と行の checkbox で明示的か。誤って通常モードに戻らないか |
| **移動モードと編集モードの排他** | 一方が ON のとき他方の disabled が視覚的に十分伝わるか（opacity / aria-disabled） |

## 7. 注意事項

- 既存ユーザーへの破壊的変更を含むため、**マイグレーション失敗時の復元手段を絶対に確保する**（バックアップキー + 中断時の冪等再試行）
- マイグレーションは**冪等**にすること（再実行で壊れない）
- `listId` 追加で全アイテム検索クエリの構造が変わる。既存セレクタ・フィルター関数の総点検が Stage 2 で必要
- アクティブリストの永続化キーは**端末ローカル**で、クラウド同期しない（端末ごとに開いているリストが異なるのが自然）
- リスト数上限・名前文字数等の magic number は `src/features/shopping/types.ts` に定数として定義する
- **「未分類」は system list として保護**: API / Store の delete・rename 系操作で `system: true` を弾く層を必ず設ける（クライアント / サーバー両側）
- **セットは全リスト共通**を Phase 10.2 の確定方針とする。将来「リスト固有セット」要望が来た場合は別フェーズで設計（本フェーズでは取り込まない / レビュー P5 への明示的応答）
- **横断課題への影響**:
  - UX-2（破壊的操作 UI 統一）: 本フェーズで `window.confirm` を 1 件追加（リスト削除）。`docs/設計書/技術負債と将来課題.md` の状態欄に記録する
  - UX-3（UI モデル変更時のユーザー周知）: 本フェーズでコーチマーク 1 枚を導入し**部分解消**。横断方針はまだ未確立
  - **新規候補 UX-4**: 複数 tablist 並置による AT ナビゲーション複雑化を技術負債.md に登録予定（Stage 2 完了時）
