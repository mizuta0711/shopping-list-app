## 1. 概要

### 背景

Phase 10.2 で「店舗別リスト」機能（複数リスト + 移動モード）が完成し、ユーザーは買う場所ごとにリストを分けて管理できるようになった。一方、**セット機能は引き続き「全リスト共通」のまま**で、Phase 10.2 §6 注意事項にも「セットは全リスト共通を Phase 10.2 の確定方針とする。将来『リスト固有セット』要望が来た場合は別フェーズで設計（本フェーズでは取り込まない）」と明記されていた。

実際に Phase 10.2 リリース直後にユーザーから以下の改修要望が挙がった:

1. セットが増えてくると `/sets` 一覧が煩雑になる → 紐付けリストでグルーピングしたい
2. 薬局リスト中に「カレーセット」（明らかにスーパー用）が並ぶのが冗長 → 現在のリスト用セットを優先表示したい
3. 設定画面の「リスト管理」「セット管理」リンクが BottomNav の「セット」タブと重複している → ナビゲーション整理が必要

### 目的

- セットを **1:1 でリストに紐付け**できるようにし、セットの所属を明示する
- `/sets` 一覧を **紐付けリスト別にグルーピング**して視認性を上げる
- `SetPickerSheet`（ホーム → AddItemForm のセット選択シート）で **アクティブリスト紐付けセットを上部優先 + その他をリスト別**に表示し、よく使うセットを素早く選べる
- BottomNav に **リスト管理を追加して 5 タブ化**し、設定画面からリスト・セット管理リンクを削除してナビゲーションの一貫性を確保

### 方針（Stage 1 ドラフト時点 / ユーザー対話で 2026-05-06 確定）

| 観点 | 判断 |
|---|---|
| データモデル | **`ShoppingSet.listId` を NOT NULL で追加**（1:1 関係）。デフォルト = 未分類 |
| 紐付けの効果 | **セットの並び順 / 絞り込みのための分類のみ**。アイテム追加先には影響しない（C1 採用）|
| /sets 一覧 | **紐付けリスト別にグルーピング表示**。共通（未分類紐付け）は最下部固定 |
| /sets/new, /sets/[id] | **「紐付けるリスト」セクションを追加**。BottomSheet 形式の `SetListPickerSheet` で選択 |
| SetPickerSheet (ホーム経由) | 上部「現在のリスト用」+ 下部「その他のリスト別」のセクション分け |
| 追加先 | **アクティブリスト固定（C1）**。紐付けは並び順分類だけに使う |
| リスト削除時の連鎖 | 削除リスト紐付けセットの listId を**未分類へ自動移動**（クライアント / サーバー両側）|
| BottomNav | **5 タブ化**: `ホーム / 履歴 / リスト / セット / 設定` |
| 設定画面 | 「リスト管理」「セット管理」リンクを**両方削除** |
| クラウド同期 | Phase 10.1b の sets 同期パイプラインを `listId` 追加で拡張。LWW 更新 + 不正 listId のサーバー側補完（未分類フォールバック、items API と同パターン）|
| マイグレーション | LocalStorage `shopping-list-app:sets:state` v1 → v2: 全セットに `listId = 未分類 ID` を埋める。バックアップ + 中断時冪等再試行 |
| 後方互換 | 旧クライアント（Phase 10.4 未適用）が `listId` なしで push してきた場合、サーバーは未分類で補完（items と同パターン）。Zod の `ShoppingSetSchema` は `.strict()` を維持しつつ `listId: z.string().uuid().optional()` 追加で旧クライアント push の 400 を回避 |
| 表記ルール | リストエンティティ名は **「未分類」**（システムリスト名、`/lists` / ListTabs / `SetListPickerSheet` で使用）。一方、セット表示文脈（`/sets` 一覧、`SetPickerSheet` 第 1 段）の「未分類紐付けセットのグループ」は **「共通」** と表記。文脈で使い分け（受け入れ条件 §3-2 でも統一）|

## 3. 機能・画面設計（Stage 1）

### 3-1. 対象画面と操作フロー

| 画面 | URL | 新規/変更 | 概要 |
|------|-----|----------|------|
| メイン画面 | `/` | 変更（軽微）| ヘッダーの ShoppingCart アイコン → Home に変更（リストタブとの混同防止）|
| BottomNav | 全ページ | 変更 | 4 タブ → **5 タブ化**（リスト管理を追加）|
| `/sets` 一覧 | `/sets` | 変更 | 紐付けリスト別グルーピング |
| `/sets/new` | `/sets/new` | 変更 | 「紐付けるリスト」セクション追加 |
| `/sets/[id]` | `/sets/[id]` | 変更 | 「紐付けるリスト」セクション追加 |
| `SetPickerSheet` | (シート) | 変更 | 上部「現在のリスト用」+ 下部「その他のリスト別」のセクション分け |
| `SetListPickerSheet`（新規）| (シート) | 新規 | セット編集画面でのリスト選択 BottomSheet |
| 設定画面 | `/settings` | 変更 | 「リスト管理」「セット管理」リンクを削除 |
| `/lists` 一覧 | `/lists` | 軽微変更 | ヘッダーの「← 設定に戻る」ボタンを削除（BottomNav からの遷移になるため）|

#### 操作フロー A: 既存ユーザーが Phase 10.4 にアップデート

1. アプリ起動 → `setsStore` LocalStorage v1 検出 → v2 マイグレーション開始
2. 旧データを `shopping-list-app:sets:state-backup-v1` にコピー（マイグレーション失敗時の復元用）
3. `useListsStore.ensureUnclassified()` で未分類 ID を取得
4. 全セットに `listId = 未分類 ID` を付与し v2 形式で保存
5. 既存ユーザーは `/sets` 一覧で「共通」セクションに既存セットがすべて並ぶ状態となる
6. BottomNav が 5 タブ化して「リスト」が登場、設定画面のリンクは消える

> **コーチマーク不要**（§3-X-7 推奨）: 機能の追加ではなく既存機能の整理が中心で、操作モデル変更が軽微なため。

#### 操作フロー B: 新規セット作成（紐付け指定あり）

1. BottomNav「セット」タブ → `/sets` → 右上「+」 → `/sets/new`
2. セット名入力（既存）
3. **「紐付けるリスト」セクション**で現在の選択（デフォルト = 🗂️ 未分類）をタップ
4. `SetListPickerSheet` が下から開く → リスト一覧から選択 → シート閉じる
5. 商品入力（既存）
6. 右上 `✓` で保存 → `/sets` に戻る → 該当リストのセクションに新セットが表示される

#### 操作フロー C: 既存セットの紐付け変更

1. `/sets` で該当セットタップ → `/sets/[id]`
2. 「紐付けるリスト」セクションのプルダウンタップ → `SetListPickerSheet` 開く → 別リスト選択
3. `✓` で保存 → `/sets` に戻る → 移動先セクションに表示される

#### 操作フロー D: リスト削除時のセット連鎖

1. ユーザーが `/lists/[id]` でリスト「スーパー」を削除
2. confirm 文言（M = 紐付くセット件数で分岐）:
   - **M ≥ 1**: 「『スーパー』を削除します。**所属する N 件のアイテム + M 件のセットは未分類リストに移動されます**。よろしいですか？」
   - **M = 0**: 「『スーパー』を削除します。**所属する N 件のアイテムは未分類リストに移動されます**。よろしいですか？」（既存 Phase 10.2 文言を維持）
3. OK で:
   - 所属アイテムの `listId` を未分類 ID に書き換え（既存）
   - **所属セットの `listId` を未分類 ID に書き換え**（新規、M ≥ 1 のみ実効）
   - リスト本体を削除して tombstone 発行
4. トースト（同様に M で分岐）:
   - **M ≥ 1**: 「『スーパー』を削除しました。N 件のアイテム + M 件のセットを未分類に移動しました」
   - **M = 0**: 「『スーパー』を削除しました。N 件のアイテムを未分類に移動しました」（既存維持）

#### 操作フロー E: SetPicker から追加（アクティブリスト = スーパー）

1. ホーム画面（アクティブ = 🛍️ スーパー）で AddItemForm の「セットから追加」ボタンタップ
2. `SetPickerSheet` 第 1 段が開く:
   - 上部セクション「🛍️ スーパー（現在のリスト）」: スーパー紐付けセット
   - 下部セクション「その他のセット」: 他リスト紐付けセット + 共通（未分類紐付け）
3. ユーザーが任意のセットをタップ → 第 2 段（アイテムチェックボックス）が開く
   - **第 2 段ヘッダーのセット名直下に「→ 🛍️ スーパーに追加」サブテキストを表示**（P1 対応 / 認知摩擦軽減）
   - 「その他のセット」から薬局紐付けセットを選んだ場合も同じく「→ 🛍️ スーパーに追加」と表示し、追加先がアクティブリストである旨を明示
4. 「追加」タップで **アイテムは現在のアクティブリスト（= スーパー）に追加される**（C1: セットの紐付け先は無視）

#### 操作フロー F: BottomNav 5 タブの遷移

1. 任意の画面で BottomNav の「リスト」タブをタップ → `/lists` へ遷移
2. リスト一覧が表示される（既存 ListListView）
3. 戻る矢印は不要（BottomNav から他タブへ移動するか、ホーム選択でメインに戻る）

### 3-4. データモデル変更（概要、Stage 2 で詳細化）

```typescript
// 既存型の拡張（破壊的変更）
type ShoppingSet = {
  // ... 既存フィールド
  listId: string;  // 新規追加（必須・NOT NULL）。デフォルト = 未分類リスト ID
};
```

LocalStorage:
- `shopping-list-app:sets:state` — version v1 → v2 へマイグレーション
- `shopping-list-app:sets:state-backup-v1` — マイグレーション前のバックアップ
- `shopping-list-app:sets:state-backup-v1:in-progress` — 中断検知用フラグ

DB（PostgreSQL）:
- `ShoppingSet.listId` カラム追加（NOT NULL、外部キー `ShoppingList.id` **ON DELETE Restrict**）
- `ShoppingSet` に `@@index([userId, listId])` を追加（リスト別取得高速化）
- migration 時は既存 `ShoppingSet` 行に各ユーザーの未分類 ID を埋めて NOT NULL 化

**ON DELETE Restrict 採用根拠 (C1 対応)**:
- items と同じパターン。リスト削除は「先に所属セット/アイテムの listId を未分類へ UPDATE → リスト本体 DELETE」のトランザクション内で実施
- Restrict は「先行 UPDATE が同一 tx 内で失敗した場合に tx 全体がロールバックされ、リスト本体も削除されない」という保護機構として機能する
- `SET DEFAULT` / `CASCADE` は採用しない: SET DEFAULT は user 単位の未分類 ID を表現できない（DEFAULT は静的値）。CASCADE はセット本体の不本意な消失リスクがある

API Zod スキーマ整合 (C2 対応):
- `src/lib/api/sync-schemas.ts` の `ShoppingSetSchema` に `listId: z.string().uuid().optional()` を追加
- `.strict()` は維持（`userId` 偽装等の防御）。`listId` のみ任意扱い
- 旧クライアント（Phase 10.4 未適用）の push で `listId` が抜けた場合、サーバー側で `ensureUnclassifiedList(userId)` の戻り値で補完
- 受信した `listId` が「該当ユーザーに存在しないリスト」を指していた場合も同じく未分類で補完（不正値防御）

詳細は Stage 2 で確定する。

```prisma
  list ShoppingList  @relation(fields: [listId], references: [id], onDelete: Restrict)

  @@index([userId, updatedAt])
  @@index([userId, listId])  // ← 新規追加。リスト別セット取得高速化
}
```

**ON DELETE Restrict 採用根拠（Phase 10.2 items と同パターン / C1 対応）**:
- リスト削除は「先に所属セット/アイテムの `listId` を未分類へ UPDATE → リスト本体 DELETE」の tx 内で実施
- Restrict は「先行 UPDATE が同一 tx 内で失敗した場合に tx 全体がロールバックされ、リスト本体も削除されない」保護機構

### 4-9. LocalStorage マイグレーション v1 → v2

**Phase 10.2 の v2→v3 と同パターン。**

#### 手順（`src/features/shopping/migrations/v1ToV2.ts`）

```ts
type SetsStateV1 = {
  sets: V1Set[];
  version: number;
};
type V1Set = {
  id: string;
  name: string;
  items: string[];   // 既存通り string[]（listId なし）
  createdAt: string;
  updatedAt: string;
  // listId なし
};
type V2Set = V1Set & { listId: string };

/**
 * v1 形式かどうかの型ガード（Phase 10.2 の isV2 と同パターン）。
 * - object でかつ sets 配列を持つ
 * - sets[0] に listId フィールドが ない（あれば既に v2 化済み）
 * - 空配列の場合は v1 として扱う（マイグレーションを冪等に通すため）
 */
function isV1(s: unknown): s is SetsStateV1 {
  if (!s || typeof s !== "object") return false;
  const obj = s as { sets?: unknown };
  if (!Array.isArray(obj.sets)) return false;
  if (obj.sets.length === 0) return true;
  const first = obj.sets[0] as { listId?: unknown };
  return typeof first.listId !== "string";
}

export function migrateSetsV1ToV2(persisted: unknown): SetsState | undefined {
  if (!isV1(persisted)) return undefined;

  // 1. v1 全データをバックアップ
  try {
    localStorage.setItem(SETS_BACKUP_V1_KEY, JSON.stringify(persisted));
    localStorage.setItem(`${SETS_BACKUP_V1_KEY}:in-progress`, "1");
  } catch {
    // QuotaExceeded 時はバックアップ失敗を許容して通常マイグレーションに進む
  }

  // 2. 未分類 ID を取得（useListsStore 経由）
  const unclassifiedId = useListsStore.getState().ensureUnclassified();

  // 3. 全セットに listId = unclassifiedId を付与
  const v2Sets: V2Set[] = persisted.sets.map((s) => ({ ...s, listId: unclassifiedId }));

  // 4. マイグレーション完了フラグ
  localStorage.removeItem(`${SETS_BACKUP_V1_KEY}:in-progress`);

  return { sets: v2Sets, version: 2 };
}
```

`useSetsStore` の `persist({ migrate })` で（Phase 10.2 の `migrateToV3` パターンと統一）:
```ts
migrate: (persistedState: unknown, version: number): SetsState => {
  let state = persistedState;
  if (version < 2) {
    state = migrateSetsV1ToV2(state);
  }
  return state as SetsState;
},
```

#### 中断時の再試行

起動時に `SETS_BACKUP_V1_KEY` と `:in-progress` フラグの両方が存在 → 中断と判断。`migrate` が再実行されることで冪等に復元（v1 形式が persist にまだ残っていれば v2 化、既に v2 化済みなら `isV1` ガードがスキップ）。

#### 失敗時の復元

`migrate` 内で例外発生 → `catch` でトースト「セットデータ移行に失敗しました。再起動してお試しください」+ `SETS_BACKUP_V1_KEY` を読み戻して v1 互換モードで起動。

### 4-12. エラー処理

| 種別 | 動作 |
|------|------|
| LocalStorage v1 → v2 マイグレーション失敗 | `SETS_BACKUP_V1_KEY` から読み戻し + トースト「セットデータ移行に失敗しました。再起動してお試しください」。アプリは v1 互換モード継続 |
| マイグレーション中断（途中でブラウザを閉じる）| 次回起動時に `SETS_BACKUP_V1_KEY` と `:in-progress` フラグを検知し、冪等に再試行 |
| リスト削除時の sets 連鎖失敗（ローカル）| `useSetsStore.applyListDeleted` が失敗した場合、`useListsStore.deleteList` 全体を try/catch でラップしてロールバック + トースト |
| リスト削除時の sets 連鎖失敗（サーバー）| `ShoppingSet.updateMany` が tx 内で失敗 → tx ロールバックでリスト本体も削除されない（Restrict 制約の安全弁）|
| `PUT /api/sync/sets` で `listId` 未指定（旧クライアント）| サーバーが `ensureUnclassifiedList` の戻り値で補完して保存（400 を返さない）|
| `PUT /api/sync/sets` で「ユーザーに存在しない listId」| `userListIds` ホワイトリストで検出 → 未分類で補完（不正 listId 防御）|
| 同期競合（A が削除中の List X に B が紐付くセット作成）| サーバー tx で List X 削除を優先。後続のセット push は listId 補完層で未分類に振替。クライアント次回 pull でトースト「N 件のセットを未分類に移動しました」|
| `POST /api/sync/sets/merge` 時に未分類が未存在 | `UNCLASSIFIED_NOT_FOUND` エラー（`lists/merge` が先行実行されている前提だが、旧クライアントや順序違反の防御として）|
| API 401 | `useSyncStore.setStatus("logged_out")`（既存）|
| API 5xx | 指数バックオフ（1s / 2s / 4s）リトライ（既存）|

## 7. 注意事項

- マイグレーション失敗時の復元手段を必ず確保する（バックアップキー + 中断時の冪等再試行 / Phase 10.2 と同パターン）

- セットの並び順保持: `/sets` 一覧でリスト別にグルーピングしても、各セクション内では従来通り `createdAt` 順を維持
- **横断課題への影響**:
  - UX-2（破壊的操作 UI 統一）: リスト削除時の confirm 文言拡張で内容追加だが `window.confirm` ベースは継続。新規追加なし
  - UX-3（UI モデル変更時のユーザー周知方針）: 本フェーズでは軽微な変更のためコーチマーク不要（§3-X-7）。横断方針は引き続き未確立
  - MAINT-3（API route の Service 層方針統一）: sets API の Service 層化は本フェーズでは見送り（lists 同様の `setSyncService.ts` 化は別途）
  - **MAINT-5**（ListPickerSheet 系の共通化 / 本レビューで新設）: `SetListPickerSheet` を `ListMoveSheet` ベースで `selectedId?` prop 追加して新設するが、本フェーズでは別コンポーネントとして実装する（影響範囲を抑制）。共通化は別フェーズで対応
