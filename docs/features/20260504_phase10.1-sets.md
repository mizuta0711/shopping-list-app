## 3. データモデル

### 3-1. 型定義

`src/features/shopping/types.ts` に追加する。既存の `ShoppingItem` には**手を入れない**。

```typescript
/// 買い物セット定義（カレーセット・調味料セット 等）
export type ShoppingSet = {
  id: string;        // crypto.randomUUID()
  name: string;      // セット名（最大50文字、必須）
  items: string[];   // 商品名の配列（順序保持・最大100件・各最大50文字・必須1件以上）
  createdAt: string; // ISO 8601
  updatedAt: string; // ISO 8601（将来クラウド同期で LWW キーになる想定）
};

export const SETS_STORAGE_KEY = "shopping-list-app:sets:state";
export const SETS_STORAGE_VERSION = 1;
```

**設計判断**:

| 項目 | 採用 | 理由 |
|------|------|------|
| `items` の型 | `string[]`（商品名のみ） | セット内のアイテムには `scope` / `status` / `order` の概念がない（追加時のアクティブ scope に展開される）。`ShoppingItem` を流用すると不要なフィールドが残り型が嘘になる |
| 重複の扱い | 配列内で同名重複は許容しない（保存時に dedupe） | UX 的に「同じものが2つ並ぶセット」は無意味 |
| 順序 | 配列のインデックス順を保持 | 一括追加時の `order` 採番に使う |
| `scope` の保持 | **持たない** | 追加時のアクティブタブが採用される（Stage 1 で確定済み） |
| `updatedAt` | 必須フィールドとして持つ | Phase 9 のクラウド同期適用時に LWW キーとして再利用できるよう先取りで定義 |

### 3-2. LocalStorage 配置

| キー | バージョン | 内容 |
|------|----------|------|
| `shopping-list-app:state`（既存） | 2 | アイテム・並び替え・onboarding。**変更なし** |
| `shopping-list-app:sets:state`（新規） | 1 | セット定義 |

**設計判断**: 既存ストアと**別キーに分離**する。理由:
- main store の `STORAGE_VERSION` を上げずに済み、既存ユーザーの migrate を発生させない
- 将来 Phase 9 のクラウド同期適用時、セットだけ別エンドポイントに切り出す柔軟性が残る
- セット機能未使用のユーザーにとって、空配列の sets が main store に混じるのを避けられる

**migrate**: 初期版（v1）のため migration ロジックは持たない。`partialize` だけ定義。将来 v2 が必要になった際に追加。

## 4. ストア設計

### 4-1. `setsStore`（新規）

`src/features/shopping/stores/setsStore.ts` を新規作成する。

```typescript
import { create } from "zustand";
import { createJSONStorage, persist } from "zustand/middleware";
import {
  SETS_STORAGE_KEY,
  SETS_STORAGE_VERSION,
  type ShoppingSet,
} from "../types";

type SetsState = {
  sets: ShoppingSet[];
};

type SetsActions = {
  addSet: (name: string, items: string[]) => void;
  updateSet: (id: string, name: string, items: string[]) => void;
  deleteSet: (id: string) => void;
  reset: () => void;
};

const initialState: SetsState = { sets: [] };

const sanitizeItems = (raw: string[]): string[] => {
  const seen = new Set<string>();
  const result: string[] = [];
  for (const r of raw) {
    const t = r.trim();
    if (!t || seen.has(t)) continue;
    seen.add(t);
    result.push(t);
    if (result.length >= 100) break; // 上限
  }
  return result;
};

export const useSetsStore = create<SetsState & SetsActions>()(
  persist(
    (set) => ({
      ...initialState,

      addSet: (name, items) => {
        const trimmedName = name.trim().slice(0, 50);
        const cleanItems = sanitizeItems(items).map((s) => s.slice(0, 50));
        if (!trimmedName || cleanItems.length === 0) return; // 呼び出し側で事前バリデーション必須（無効入力は黙ってスキップ）
        const id = crypto.randomUUID();
        const now = new Date().toISOString();
        set((state) => ({
          sets: [
            ...state.sets,
            { id, name: trimmedName, items: cleanItems, createdAt: now, updatedAt: now },
          ],
        }));
      },

      updateSet: (id, name, items) => {
        const now = new Date().toISOString();
        const trimmedName = name.trim().slice(0, 50);
        const cleanItems = sanitizeItems(items).map((s) => s.slice(0, 50));
        if (!trimmedName || cleanItems.length === 0) return;
        set((state) => ({
          sets: state.sets.map((s) =>
            s.id === id
              ? { ...s, name: trimmedName, items: cleanItems, updatedAt: now }
              : s,
          ),
        }));
      },

      deleteSet: (id) => {
        set((state) => ({ sets: state.sets.filter((s) => s.id !== id) }));
      },

      reset: () => set(initialState),
    }),
    {
      name: SETS_STORAGE_KEY,
      version: SETS_STORAGE_VERSION,
      storage: createJSONStorage(() => localStorage),
      partialize: (state) => ({ sets: state.sets }),
    },
  ),
);
```

**設計判断**:
- `addSet` は **戻り値なし**（既存 `useShoppingStore.addItem` と同じ流儀）。id はストア内部で生成し外部には漏らさない。`/sets/new` 保存後の遷移は `router.replace('/sets')` 固定で、id を呼び出し側が知る必要がない
- 無効入力（空名 / 空配列）は黙ってスキップ。呼び出し側（`SetEditView`）で**事前バリデーションを必須**とし、保存ボタンの活性条件で防ぐ
- `sanitizeItems` を**ストア内**で持つ。UI 側でも一次バリデーションは行うが、ストアに渡る時点で重複・空文字が混ざらないことを保証する（防御的プログラミング）
- `name` の最大長は 50 文字、`items` 1件あたり 50 文字、配列上限 100 件。これらは UI 側のバリデーションメッセージと連動させる

### 4-2. 既存 `useShoppingStore` への変更

**なし**。`addItems` の既存重複スキップ仕様（同 `scope`, `PENDING` で同名）をそのまま流用する。

### 4-3. セレクター

`src/features/shopping/stores/selectors.ts` に追加:

### 5-4. `SetEditView`

| Props | `mode: "new" \| "edit"`, `setId?: string` |
| バリデーション | セット名: 必須・最大50文字 / 商品名: split 後 1件以上・各最大50文字・最大100件 |
| プレビュー | 入力中に `split(/[\n、。]/)` → trim → 空除外 → dedupe して件数表示「X品が登録されます」（既存 `AddItemForm` と同じ split ロジックを `utils/parseItemNames.ts` として切り出して共通化） |
| 保存 | new / edit いずれも保存後は `router.replace('/sets')` で**統一**（deeplink 経由でも一貫した挙動にする / 履歴汚染を避ける） |
| 削除（編集モードのみ） | `window.confirm` で確認後 `deleteSet` → `router.replace('/sets')` |
| 不存在の id | hydration 後に対象セット未発見なら「セットが見つかりません」+ 戻るボタン（hydration 中は skeleton 表示で誤判定を防ぐ） |

**抽出する共通関数**: `parseItemNames(text: string): string[]`

```typescript
// src/features/shopping/utils/parseItemNames.ts（新規）
export const parseItemNames = (text: string): string[] => {
  const seen = new Set<string>();
  const result: string[] = [];
  for (const raw of text.split(/[\n、。]/)) {
    const t = raw.trim();
    if (!t || seen.has(t)) continue;
    seen.add(t);
    result.push(t);
  }
  return result;
};
```

`AddItemForm` の現行ロジック（[AddItemForm.tsx:41-42](../../src/features/shopping/components/AddItemForm.tsx#L41-L42)）も将来この関数に寄せると重複が減る。本フェーズでは新規利用箇所（`SetEditView`）のみ呼び出し、`AddItemForm` のリファクタは別タスクとする（スコープを膨らませない）。

### 5-5. `SetPickerSheet`（ボトムシート）

| 要素 | 仕様 |
|------|------|
| 起動 | `AddItemForm` 左の「📋」アイコンタップ |
| レイアウト | 画面下部から固定。`max-h-[80dvh]`, `overflow-y-auto`, `safe-area-inset-bottom` 対応。背景に半透明の backdrop |
| ヘッダー | タイトル「セットから追加」/ 右上「✕」 |
| 一覧 | セット名 + 件数 + 「📋」のような視覚アイコン（emoji なしでも可）。タップで確認 |
| 確認 | `window.confirm("{セット名} の N 品を {scope ラベル} に追加します")` を使用（既存 `SettingsView` の削除確認と同方式で一貫性確保。モーダル on モーダルの ARIA 課題を回避） |
| 追加処理 | `useShoppingStore.getState().addItems(set.items, activeScope)` を呼ぶ。**追加件数の計算は `AddItemForm` と同じ before/after 差分パターン**（下記コード例参照）。トーストは「{セット名} を追加しました（N件 / M件は重複）」 |
| 0件時 | 「セットがまだありません」+「セットを作成する」CTA（`/sets/new` へ） |
| ESC / backdrop タップ | 閉じる |

**実装方針**:
- 新規 npm パッケージは導入しない。`fixed inset-0` の overlay + `transform translate-y-*` の transition で十分
- `OnboardingModal` の overlay 構造を参考に、`role="dialog"` + `aria-modal="true"` を必ず設定
- **z-index**: overlay は `z-50`（既存ヘッダー `z-10` / sticky bottom コンテナ `z-10` を確実に上回るため）。トーストの sonner はデフォルト `z-[9999]` 相当のため重ね順は問題なし
- フォーカス管理: 開いたとき先頭ボタンへ `ref.current?.focus()`、閉じたら起動ボタンに戻す

**追加件数の計算パターン**（`AddItemForm.tsx:43-46` と同形）:

```tsx
const handlePickSet = (set: ShoppingSet) => {
  const ok = window.confirm(`${set.name} の ${set.items.length} 品を「${scopeLabel}」に追加します`);
  if (!ok) return;
  const before = useShoppingStore.getState().items.length;
  useShoppingStore.getState().addItems(set.items, activeScope);
  const added = useShoppingStore.getState().items.length - before;
  const skipped = set.items.length - added;
  if (added === 0) {
    toast(`${set.name} はすべて既存のため追加されませんでした`);
  } else if (skipped === 0) {
    toast.success(`${set.name} を追加しました（${added}件）`);
  } else {
    toast.success(`${set.name} を追加しました（${added}件 / ${skipped}件は重複）`);
  }
  onClose();
};
```

| フィールド | ルール | UI 制約 | エラー表示 |
|-----------|--------|---------|----------|
| 名前 | trim 後 1〜50 文字 | `<input maxLength={50}>` で 51 文字目以降の入力をブラウザレベルでブロック | 「セット名を入力してください」（空時のみ。文字数超過は input が抑止するためエラー文言不要） |
| 商品名（合計） | parse 後 1〜100 件 | プレビュー表示で件数を可視化。100 件超は dedupe + slice (100) で打ち切り | 「商品名を1つ以上入力してください」/ 100件超時は「100件まで登録できます」 |
| 商品名（個別） | trim 後 1〜50 文字 | textarea 全体には maxLength を付けない（複数行入力のため）。parse 段階で 50 文字超は **slice する**（保存処理側で防御） | UI 上は警告のみ（ストア層で slice してデータ整合を保つ） |

「保存」ボタンは無効化条件: `name.trim().length === 0 || parsed.length === 0`。

### 7-2. ボトムシート追加時

- 既存の `addItems` の dedupe ロジックに依存する
- 追加件数 0 の場合のみ全重複扱いのトーストを出す

## 8. クラウド同期との関係（将来検討メモ）

Phase 9（クラウド同期）が完了したあと、セットも同期したい場合の方針メモ:

| 項目 | 想定 |
|------|------|
| DB スキーマ | `ShoppingSet` テーブル（`id, userId, name, items (jsonb), createdAt, updatedAt`）+ `DeletionTombstone` を共有 or 専用化 |
| 同期方式 | LWW（`updatedAt` 比較）。`ShoppingItem` と同じ debounce + reconcile スキームに合流 |
| 互換性 | 本フェーズで **`ShoppingSet.id` を UUID v4** で発行しているため、サーバ側の `id` をクライアント発行 UUID とする方針と整合する |
| `updatedAt` の扱い | Phase 9 の決定事項に従い、`@updatedAt`（Prisma 自動更新）は使わず**クライアント発行値をサーバが保存**する。本フェーズの `setsStore` で `addSet` / `updateSet` のたびに `new Date().toISOString()` をセットしているのは、この将来方針と一致する |

本フェーズではこれらは実装しない（親設計書 §3-A の方針通り）。

## 10. ロールアウト / リスク

| リスク | 対策 |
|--------|------|
| ボトムシートと既存ドラッグの競合 | ボトムシート起動はアイコン**タップ**のみ。長押し / ドラッグは使わない |
| LocalStorage 容量圧迫 | セット 100 件 × 100 商品 × 50 文字 = 約 500KB 上限。LocalStorage 5MB に対し十分余裕 |
| 既存ユーザーへの影響 | 別 storage key にしているため main store の migrate は走らない。**既存データに一切影響しない** |
| Phase 9 クラウド同期との後付け統合 | `updatedAt` を最初から持たせて LWW 適用余地を残している（§8 参照） |
