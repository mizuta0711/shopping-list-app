## 運用ガイド

<!--
この設計書の運用フロー（作成後、このセクションは削除してよい）:

### 設計フェーズ
1. `/new-feature {機能名}` でこのテンプレートをコピーして `docs/features/yyyymmdd_機能名.md` を作成
2. メタ情報の「変更の性質」「変更規模」を記入（AI 推測 → ユーザー承認）
3. 全体ステータスを「🔵 設計中」に設定
4. タスク一覧を記入

L 規模の場合（2段階設計）:
5a. Stage 1（機能・画面設計）を記入、メタ情報の Stage 1 ステータスを「🔵 記入中」に
6a. /design-review feature でレビュー → ユーザー承認 → Stage 1 ステータス「✅ 確定」
7a. Stage 2（技術設計）を記入、Stage 2 ステータスを「🔵 記入中」に
8a. /design-review tech でレビュー → Stage 2 ステータス「✅ 確定」

M 規模（UX変更なし）の場合（1段階設計）:
5b. Stage 1 のメタ情報を「⚪ 不要」、Stage 1 セクションは空のまま
6b. Stage 2（技術設計）を記入
7b. /design-review tech でレビュー → Stage 2 ステータス「✅ 確定」

### 実装フェーズ（フェーズごとに繰り返す）
1. 全体ステータスを「🟡 実装中」に更新
2. coding-specialist でフェーズの実装を実施
3. 実装完了したタスクのステータスを「✅ 完了」に更新
4. 保留・却下を決定したタスクのステータスと理由を更新
5. code-reviewer でコードレビューを実施
6. /browser-test でスポットテストを実施（UI 変更ありの場合）
7. npm run build でビルド確認
8. docs/設計書/ 配下の関連ドキュメントを更新（/update-docs）
9. git commit（フェーズ単位）

### 完了時
1. 全タスクのステータスが最終状態（✅/⏸️/❌）であることを確認
2. 全体ステータスを「🟢 完了」または「⏸️ 一部保留」に更新
3. 完了 → docs/features/completed/ に移動
4. 一部保留 → そのまま docs/features/ に残す（保留タスクの対応時に再開）

### Stage 2 で技術制約が発覚して Stage 1 に戻る場合
1. Stage 2 の記入を中断
2. Stage 1 ステータスを「🟡 レビュー中」に戻す
3. Stage 1 を修正
4. /design-review feature を再実行
5. ユーザー再承認後、Stage 2 を再開
-->
