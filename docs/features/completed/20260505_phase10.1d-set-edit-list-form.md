## 1. 概要

### 背景

Phase 10.1a で実装したセット編集画面では、商品リストの入力を **textarea**（改行/読点/句点で区切り）で行っていた。実運用ではセット内の各商品を視覚的に確認しづらく、削除も「テキストの行を削除」という間接的な操作になっていた。

### 目的

商品リストを **通常の買い物リストと同じ「行ベースのリスト UI」** に変更し、追加 UI も既存の AddItemForm スタイルに揃えることで、操作モデルをアプリ全体で統一する。

### 方針（対話で合意）

| 観点 | 判断 |
|---|---|
| アイテム表示 | 各行に商品名 + 右端に ✕ 削除ボタン。チェックボックスはセットに購入概念がないため省略 |
| 追加 UI | AddItemForm 風の `textarea` + ➕ ボタン。改行/読点/句点で複数追加可 |
| 並び替え | 本フェーズではスコープ外 |
| ローカル状態 | 編集中は `items: string[]` をローカル保持。「保存」で確定 |
| 重複処理 | 追加時に既存と重複したらスキップ |
| ネストフォーム回避 | 追加部は `<form>` ではなく `<div>` + onClick + onKeyDown |
| 既存設計書 §5-4 のプレビュー件数 | 「件数表示」は引き続き維持。ただし表示位置は items リストの直上 or フッター近辺 |
| 100 件超過時の打ち切り表示 | 既存の `overLimit` 警告（amber 色）は維持 |

## 3. 技術設計

### 3-1. 変更ファイル

| ファイル | 操作 | 内容 |
|---------|------|------|
| `src/features/shopping/components/SetEditView.tsx` | 変更 | textarea を撤去し、`items: string[]` を直接編集する UI に書き換え |

**変更しないもの**:
- `useSetsStore`（既存の `addSet` / `updateSet` / `deleteSet` で対応可）
- `parseItemNames` ユーティリティ（追加フォームから流用）
- 親 Phase 10 / Phase 10.1a 設計書（型・データモデルは変更なし）

### 3-2. 内部 state 設計

```tsx
const [name, setName] = useState("");
const [items, setItems] = useState<string[]>([]);    // 編集中のアイテム配列
const [addInput, setAddInput] = useState("");        // 追加 textarea の入力値
const [hydrated, setHydrated] = useState(false);
const [initialized, setInitialized] = useState(mode === "new");
```

編集モードでは hydrate 後に target.items を items state にロード（一度だけ）。

### 3-3. ハンドラ

```tsx
// 追加: 改行/読点/句点で複数追加、既存と重複したらスキップ、上限到達で打ち切り
const handleAddInputSubmit = useCallback(() => {
  const { items: parsed } = parseItemNames(addInput);
  if (parsed.length === 0) return;
  setItems((prev) => {
    const seen = new Set(prev);
    const next = [...prev];
    for (const name of parsed) {
      if (seen.has(name)) continue;
      if (next.length >= SET_ITEMS_MAX_COUNT) break;
      next.push(name);
      seen.add(name);
    }
    return next;
  });
  setAddInput("");
}, [addInput]);

// Cmd/Ctrl + Enter で送信（既存の AddItemForm と同じ）
const handleAddKeyDown = (e: KeyboardEvent<HTMLTextAreaElement>) => {
  if ((e.metaKey || e.ctrlKey) && e.key === "Enter") {
    e.preventDefault();
    handleAddInputSubmit();
  }
};

// 行削除
const handleRemove = (name: string) => {
  setItems((prev) => prev.filter((n) => n !== name));
};

// 保存（既存）: items state を addSet/updateSet に渡す
```

### 3-4. レイアウト（最終版: メイン買い物リスト画面と統一）

実装過程の追加要望により、メイン画面と同じ「上：ヘッダー固定 / 中：スクロール / 下：追加入力固定」レイアウトに統一。

```
┌──────────────────────────────────┐
│ ← セットを編集     🗑   ✓        │ ← ヘッダー（保存/削除は右上アイコン）
├──────────────────────────────────┤
│ セット名                          │ ← スクロール領域
│ [ティラミス___________________]   │
│                                   │
│ 商品 (9 / 100)                    │
│  クリームチーズ              ✕   │
│  生クリーム                  ✕   │
│  卵黄                        ✕   │
│  ...                              │
├──────────────────────────────────┤
│ [追加したい商品名… ]        ➕    │ ← 下部固定の追加フォーム
└──────────────────────────────────┘
```

**コンテナ**: `<main h-[100dvh] flex-col>` で画面高さに固定。`<form>` がヘッダー〜スクロール〜下部入力まで包み、保存ボタンは header 内の `type="submit"`、削除ボタンは `type="button"`、追加ボタンも `type="button"` で nested form を回避。スクロール領域は `flex-1 overflow-y-auto`。下部入力欄は `pb-[calc(0.75rem+env(safe-area-inset-bottom))]` で iOS のホームバー領域を確保。

### 3-5. アクセシビリティ

| 観点 | 対応 |
|------|------|
| アイテム行 | `<li>` で `<button aria-label="{name} を削除">` の ✕ ボタン（h-11 w-11 = 44px タップ領域） |
| 追加フォーム | textarea に `aria-label="追加する商品名（改行で複数追加可能）"`。送信ボタンは `aria-label="追加"` |
| 件数表示 | `<p>` で「{count} / 100 品」。100件超で amber color + 警告文「(上限)」 |
| 保存ボタン | ヘッダー右の Check アイコン。`aria-label="作成"` または `"保存"`。name または items が空の時 disabled（灰色） |
| 削除ボタン | ヘッダー右の Trash アイコン（赤）。edit モードのみ表示。`aria-label="このセットを削除"` |
| キーボード | textarea 内で Cmd/Ctrl+Enter で追加。name input で Enter は form submit を発火（保存と同等） |

### 3-6. バリデーション・上限

| 項目 | ルール |
|---|---|
| セット名 | 1〜50 文字（既存） |
| アイテム名（個別）| trim 後 1〜50 文字（`parseItemNames` 内で slice） |
| アイテム数 | 100 件まで（`SET_ITEMS_MAX_COUNT`）。超過時は追加で打ち切り |
| 重複（追加時）| 同名のアイテムが既に items にあればスキップ |
| 保存可否 | name.trim() && items.length > 0 |

### 3-7. 削除 UX

行の ✕ ボタンタップで即時削除。アンドゥは付けない（編集中の操作で保存前に取り消したい場合は「再度追加」で十分。Phase 11 の削除アンドゥは LocalStorage 確定済みアイテムが対象なので別物）。

## 6. 注意事項

- 追加フォームを `<form>` にするとネストフォーム（外側の Save 用 `<form>` と二重）になり HTML 仕様違反になるため、`<div>` 内で onClick + onKeyDown のパターンを使う
- アイテム行の削除はアンドゥなしで即時。保存前なら未確定のため再追加で対応可能
- 行のドラッグ並び替えは本フェーズではスコープ外。要望があれば別フェーズで dnd-kit を流用して追加可能
- Phase 10.1a 設計書 §5-4 の `parseItemNames` プレビュー機能は、本フェーズでは「追加時に消費」する形になり、items リストが結果として残る
