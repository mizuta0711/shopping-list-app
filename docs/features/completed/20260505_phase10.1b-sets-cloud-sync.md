## 1. 概要

### 背景

Phase 10.1a でセット機能（`ShoppingSet` + `useSetsStore`）を LocalStorage 単独で導入した。Phase 9 で買い物アイテムはクラウド同期化済みのため、ログイン済みユーザーは複数端末で買い物リストが同期される一方、**セットは端末ごとにバラバラ**になっている。本フェーズでは Phase 9 の LWW + DeletionTombstone スキームを `ShoppingSet` に横展開する。

### 目的

- ログイン済みユーザーのセット作成・編集・削除を複数端末間で同期する
- 既存の Phase 9 同期パイプラインの設計（LWW、tombstone、subscribe + debounce、初回ログインマージ）をそのまま流用し、**新しい設計判断を最小化**する
- UX は変更しない（裏で同期するだけ。同期インジケーターのドットは既存のものを流用）

### 方針（対話で合意）

| 観点 | 判断 |
|---|---|
| 同期粒度 | **セット単位の LWW**（updatedAt 比較で勝者を決め、items 配列はセット全体で置換）。アイテム単位の差分マージは複雑性過剰 |
| DB スキーマ | `ShoppingSet` テーブル新設 + `SetDeletionTombstone` テーブル新設（`DeletionTombstone` とは別。`itemId` と `setId` は ID 体系が独立しており、共有すると `kind` 列で分岐が必要になり煩雑） |
| API エンドポイント | 既存 `/api/sync/items` パターンを踏襲し、`/api/sync/sets` (GET/PUT) と `/api/sync/sets/merge` (POST) を新規追加 |
| クライアント同期サービス | 既存 `syncOrchestrator` を拡張し、items と sets の両方を扱う。**1 つの orchestrator が両ストアを subscribe + 1.5s debounce + online/offline/focus イベントを共有**。pull/push の HTTP は別呼び出しで分離 |
| `ShoppingSet` 型 | 既に `id`/`createdAt`/`updatedAt` を持っているので追加不要。`userId` は API DTO から除外（既存 `ShoppingItemDTO` と同じ） |
| LocalStorage マイグレーション | `SETS_STORAGE_VERSION` は v1 据え置き（既存スキーマで同期に十分。v1 で `updatedAt` 既存） |
| 初回ログインマージ | `useInitialMerge` を拡張し、items とセットの両方をマージ。アイテム同様、ローカルとサーバーの双方を取り込む |
| エラー処理 | Phase 9 と同等（401 で sync 停止、5xx で指数バックオフ、ネットワーク断は次回 online 復帰時に retry） |
| 削除 tombstone のクリーンアップ | Phase 9.1 と合わせて将来対応（30 日経過後にバッチ削除）。本フェーズではテーブル定義のみ |

## 3. 技術設計

### 3-1. DB スキーマ追加

`prisma/schema.prisma` に以下を追加：

```prisma
/// 買い物セット（クラウド同期対象。Phase 10.1b で追加）
/// id はクライアント発行 UUID、updatedAt は LWW 判定キー（クライアント発行値を保存）
model ShoppingSet {
  id        String   @id /// クライアント発行 UUID v4
  userId    String   /// 所有ユーザー
  name      String   /// セット名（最大 50 文字、サーバー側でも slice）
  items     String[] /// アイテム名の配列（PostgreSQL ARRAY 型、最大 100 件）
  createdAt DateTime /// クライアント発行のタイムスタンプ
  updatedAt DateTime /// LWW 判定キー（クライアント発行）

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, updatedAt])
}

/// セット削除イベントのトラッキング（since 以降の削除を別端末に伝播）
model SetDeletionTombstone {
  id        String   @id @default(uuid())
  userId    String   /// 削除を行ったユーザー
  setId     String   /// 削除された ShoppingSet の id
  deletedAt DateTime @default(now()) /// サーバー時刻

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, setId])
  @@index([userId, deletedAt])
}
```

`User` モデルに relations 追加：

```prisma
model User {
  // ... 既存フィールド
  shoppingSets       ShoppingSet[]
  setTombstones      SetDeletionTombstone[]
}
```

**バックアップ運用** (`tools/export-to-sql.ts`):
- `ORDERED_TABLES` に `shopping_sets`, `set_deletion_tombstones` を追加（DEPENDENCY 順: User の後）
- `DB_TABLE_MAP` に対応エントリ追加

### 3-2. 共有型定義（`src/types/sync.ts` 追加）

    id: string;

### 3-3. API エンドポイント

#### `GET /api/sync/sets?since=ISO`

リクエスト: クエリパラメータ `since`（任意）、`Authorization` は session cookie。

レスポンス（200, `SetsSyncPullResponse`）:
```json
{
  "success": true,
  "data": {
    "sets": [
      {
        "id": "uuid-1",
        "name": "カレーセット",
        "items": ["玉ねぎ", "じゃがいも", "カレールー"],
        "createdAt": "2026-05-04T12:00:00.000Z",
        "updatedAt": "2026-05-05T08:00:00.000Z"
      }
    ],
    "serverDeletes": ["uuid-2"],
    "serverTime": "2026-05-05T10:00:00.000Z",
    "lastUpdatedAt": "2026-05-05T08:00:00.000Z"
  }
}
```

実装方針:
- `since` 未指定なら全セット取得 / 指定時は `updatedAt > since` のみ
- `serverDeletes` は `SetDeletionTombstone.deletedAt > since` の `setId` 配列
- `lastUpdatedAt` はクライアントが次回 `since` として使う基準時刻（最新 `updatedAt` または `serverTime`）

#### `PUT /api/sync/sets`

リクエスト（`SetsSyncPushRequest`）:
```json
{
  "upserts": [
    {
      "id": "uuid-1",
      "name": "カレーセット",
      "items": ["玉ねぎ", "じゃがいも", "カレールー"],
      "createdAt": "2026-05-04T12:00:00.000Z",
      "updatedAt": "2026-05-05T08:00:00.000Z"
    }
  ],
  "deletedIds": ["uuid-2"],
  "since": "2026-05-05T07:00:00.000Z"
}
```

レスポンス（200, `SetsSyncPushResponse`）:
- `applied`: サーバーで採用された upsert（クライアント updatedAt がサーバー以上）
- `rejected`: サーバー側が新しいため拒否されたもの（クライアントは serverSet を採用）
- `serverChanges`: since 以降にサーバーで更新された他セット（push しているクライアント以外の更新）
- `serverDeletes`: since 以降の他端末削除

実装方針:
- `prisma.$transaction` で:
  1. `upserts` を 1 件ずつ `findUnique` → サーバー `updatedAt < クライアント` なら upsert、そうでなければ rejected
  2. `deletedIds` を `delete` + `setDeletionTombstone.upsert`
  3. since 以降の他更新を取得して `serverChanges` / `serverDeletes` に積む

#### `POST /api/sync/sets/merge`

初回ログイン時のマージ（`useInitialMerge` から呼ぶ）。リクエスト: `localSets`（端末上のローカルセット全件）。

レスポンス（`SetsSyncMergeResponse`）:
- `finalSets`: マージ後のサーバー上の全セット（id 衝突なしで両方の和集合 + 同 id は LWW）
- `uploadedCount`: ローカル → サーバーへ反映された件数
- `downloadedCount`: サーバー → ローカルへ反映された件数

実装方針:
- 既存 `/api/sync/merge` の sets 版。`prisma.$transaction` 内でアップサート + 全件取得。

### 3-4. クライアント同期サービス拡張

#### `useSetsStore` への追加アクション

```ts
// 既存の addSet/updateSet/deleteSet/reset に加えて:

setSets: (sets: ShoppingSet[]) => void;
applyServerChanges: (params: {
  upserts: ShoppingSet[];
  deletes: string[];
}) => void;
```

`setSets` は merge 完了時にサーバーから取得した全件で上書き、`applyServerChanges` は pull/push のレスポンス反映。実装は `useShoppingStore` と同じ Map ベース。

**pendingIds 管理**: `useShoppingStore` と同じく `useSyncStore.pending` Set にセット ID を積む（既存 set 型のまま）。プレフィクス `set:` で衝突回避（例: `set:uuid-1`）。あるいは別 Set として `pendingSetIds` を新設する案もあるが、後者の方が責務分離が明確。**推奨: 別 Set `pendingSetIds`**。

#### `syncClient` 拡張

```ts
// 既存 pull/push/mergeOnLogin に加えて:
pullSets: (since: string | null, signal?: AbortSignal) => Promise<SetsSyncPullResponse>;
pushSets: (req: SetsSyncPushRequest, signal?: AbortSignal) => Promise<SetsSyncPushResponse>;
mergeSetsOnLogin: (req: SetsSyncMergeRequest, signal?: AbortSignal) => Promise<SetsSyncMergeResponse>;
```

#### `syncOrchestrator` 拡張

```ts
// 既存 createSyncOrchestrator() の戻り値に追加:
pullSetsOnce(): Promise<void>;
pushPendingSetsNow(): Promise<void>;
mergeSets(localSets: ShoppingSet[]): Promise<{ uploadedCount: number; downloadedCount: number }>;
```

内部:
- `useSetsStore.subscribe((state, prev) => { ... })` で sets 配列の差分検知（参照変化）
- 差分があれば `pendingSetIds` に積み、1.5s debounce で `pushPendingSetsNow` 実行
- `pullOnce` / `pushPendingNow` と並列ではなく**直列**実行（順序不要・サーバー負荷分散）

#### `useInitialMerge` 拡張

既存の items merge と並列で sets merge を実行：

```ts
const [itemsResult, setsResult] = await Promise.all([
  orchestrator.merge(localItems),
  orchestrator.mergeSets(localSets),
]);

const total = {
  uploadedCount: itemsResult.uploadedCount + setsResult.uploadedCount,
  downloadedCount: itemsResult.downloadedCount + setsResult.downloadedCount,
};

toast(`同期完了: ↑${total.uploadedCount} ↓${total.downloadedCount}`);
```

### 3-5. 同期インジケーター

UI 変更なし。既存の `SyncStatusDot` / `SyncStatusSheet` は items と sets の両方の状態を集約して表示する。具体的には `useSyncStore.pending` (items) または `pendingSetIds` のいずれかが空でなければ「同期中」表示。

### 3-6. エラー処理

Phase 9 と同等：
- 401 → `useSyncStore.setStatus("logged_out")`
- 5xx → 指数バックオフ（1s/2s/4s）リトライ
- ネットワーク断 → 次回 online 復帰時に retry
- レスポンス JSON が `success: false` → エラーログ + 同期失敗状態

## 4. ブラウザ評価計画

### 4-1. 主要シナリオ

| # | 操作 | 期待結果 |
|---|------|---------|
| 1 | ログイン状態でセット作成 | サーバーに保存される（DB 直接確認 or Network タブで PUT 200） |
| 2 | 同ユーザーでもう 1 ブラウザ起動 → ログイン | 既存セットが pull される |
| 3 | 端末 A でセット名変更 → 端末 B で 1.5s 待つ | 端末 B でセット名が更新される |
| 4 | 端末 A でセット削除 → 端末 B で待つ | 端末 B でセットが消える |
| 5 | 端末 A オフラインでセット作成 → online 復帰 | 端末 B にも反映される |
| 6 | 同 ID で端末 A/B 同時編集（updatedAt 異なる） | 後勝ち（updatedAt が新しい方） |
| 7 | 初回ログイン（ローカルにセットあり、サーバーにも別セットあり）| マージ完了トーストで両方反映 |
| 8 | ログアウト → 別ユーザーでログイン | 前ユーザーのセットが見えない |
| 9 | 100 件超のセットを push | 全件サーバーに反映、エラーなし |

### 4-2. UX 評価観点

| 観点 | 確認 |
|------|------|
| 同期の透明性 | 既存 SyncStatusDot に sets の同期状態が反映されるか |
| パフォーマンス | sets 数 100 程度でも 1.5s debounce + push が体感的に重くないか |
| エラー伝達 | 5xx 発生時にトースト or ドット赤化で通知されるか |

## 6. 注意事項

- **DB スキーマ変更前に必ず `npm run db:backup` を実行**（CLAUDE.md ルール）。`tools/export-to-sql.ts` の `ORDERED_TABLES` も更新
- セットの items 配列は PostgreSQL の TEXT[] 型を使う。Prisma の `String[]` 型は ARRAY にマップされる
- LWW 衝突時のサーバー側 rejected は、クライアントが必ず serverSet で上書きする（既存 items 同期と同じパターン）
- pendingSetIds は LocalStorage に persist しない（既存 pending 同様。再起動で失われるが、起動時の pull で同期回復する）
- セットのクラウド同期はログインユーザーのみ。ログアウト中は LocalStorage 単独動作（既存 Phase 10.1a 通り）
