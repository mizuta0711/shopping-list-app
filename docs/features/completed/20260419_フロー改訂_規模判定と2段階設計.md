## 1. 概要

### 背景

現在のテンプレートには以下の曖昧さ・不足がある:

1. **S/M/L の判定基準が曖昧**: 「機能追加」と「新機能」の境界、ファイル数の基準、誰が判定するか等が未定義
2. **バグ修正が S 固定扱い**: 複数機能にまたがる不具合やスキーマ変更を伴う大規模修正まで S 扱いされてしまう
3. **設計フェーズが1段階**: 「何を作るか（UX）」と「どう作るか（技術）」を同じレビューで扱っており、要件の誤解が DB/API 設計後に発覚する非効率がある
4. **`/design-review` は技術レビュー中心**: 企画・UX 観点のレビューがなく、`product-advisor` エージェントがフロー外で孤立している
5. **改修 vs 新規の違いが意識されていない**: アプローチが異なるのに同じ扱い

### 目的

- 規模判定を明確化し、誰が・いつ・どう判定するかを定義する
- バグ修正も規模に応じた適切なフローに乗せる
- 設計フェーズを2段階化し、要件レベルの誤解を早期検出できるようにする
- `product-advisor` を Stage 1 レビューに統合し、形骸化を解消する
- 改修/新規/バグ修正の性質差を意識する補助軸を追加する

### 方針

1. **規模軸 (S/M/L) と性質軸 (新規/改修/バグ修正) を直交させる**
2. **判定プロセスは「AI 推測 → ユーザー承認」** の2ステップ
3. **設計書は1ファイルで維持**、Stage 1/2 はセクション構造で分離
4. **`/design-review` にモード追加**（`feature` / `tech`）、新規スキルは増やさない
5. **バグ修正専用テンプレートは作らない**、機能設計書テンプレートの柔軟性で対応

## 3. 変更詳細

### 3-1. 規模判定の基準とプロセス（CLAUDE.md への追加）

#### 主軸: 変更の規模（S/M/L）

| 観点 | S | M | L |
|------|---|---|---|
| 変更ファイル数 | 1〜2 | 3〜10 | 10超 または広範囲 |
| 新規API | なし | 既存パターンに沿った 1〜2 | 新パターン or 複数 |
| DBスキーマ変更 | なし | カラム追加のみ | 新テーブル、FK変更、破壊的変更 |
| UX変更 | なし（既存挙動維持） | 限定的（1画面内） | 複数画面・フロー全体 |
| 設計書更新 | 不要 | 必要 | 必要（複数種類） |

**判定ルール**:
- 最も該当する項目が多い規模を採用
- 1つでも L 相当の項目があれば **L に引き上げる**

**判定プロセス**:
1. ユーザーが作業を依頼
2. メインClaude が指示内容と影響範囲から **S/M/L を推測して提示**
   - 例: 「これは M 規模と判断します。既存API に類似エンドポイントを1つ追加、関連ファイル5つ程度、DB変更なし。進めてよいですか？」
3. ユーザーが承認 or 修正（「いや、L で」等）
4. 承認された規模でフローに入る

**スコープ拡大時の再判定**:
- 原則: **現在のフェーズを完了させてから判断する**（作業途中で止めると中途半端な状態になる）
- 判定タイミング:
  - **着手前に気づいた場合**: 即座に停止して規模の再判定
  - **フェーズ実行中に気づいた場合**: **現フェーズを完了**させ、フェーズ完了時の報告で「当初 S 規模と判定したが、実装中にファイル数が当初予想を超えたため、次フェーズから M に引き上げたい」のように報告・確認
  - **ただし破壊的変更やスキーマ変更が必要と判明した場合は即停止**（L 相当の変更は戻しが困難なため）
- AI 単独でフロー切り替えをしない
- 引き上げ判断後は、必要な設計書追記・レビュー等を**遡って実施**

**AI 判定のバイアス補正ルール**:
- 判定に迷ったら **L 寄りで提示する** ことを原則とする
- ユーザーが「S で良い」と判断して下げる方が、「S で進めたが実は L だった」より安全
- 「ファイル数が不明」「DB スキーマ変更が必要か未確定」など判断材料が揃わない場合は M 以上で提示

#### 補助軸: 変更の性質

| 区分 | 意識すべき点 |
|------|-------------|
| 新規追加 | 既存パターンを先に確認してから設計する |
| 改修 | 「何を変えるか・何を変えないか」の境界を明示する |
| バグ修正 | 根本原因を確定してから修正する（CLAUDE.md「バグ修正の原則」参照） |

性質は S/M/L とは独立して判定する（例: 「M規模の改修」「L規模のバグ修正」等）。

### 3-2. バグ修正の扱い

**現状**: バグ修正 = S 固定扱い（実態と合っていない）

**変更後**: バグ修正も S/M/L で判定する

| バグの例 | 規模 | 設計書 |
|---------|------|--------|
| タイポ、表示ずれ、null チェック漏れ | S | 不要 |
| 複数機能にまたがる不具合、API ロジック修正 | **M** | 軽量な設計書を作成 |
| スキーマ変更を伴う修正、認証フロー変更 | **L** | 通常の設計書を作成（2段階設計） |

**バグ修正フローの順序（CLAUDE.md「バグ修正の原則」との整合）**:

既存の「原因特定前の修正禁止」原則を尊重するため、以下の順序で進める:

```
1. バグ発見
   ↓
2. 原因特定フェーズ（対話的、設計書なし）
   - エラーメッセージ全文確認
   - ログ追加 or 最小再現手順で原因を確定
   - 「何が原因だと考えたか」を短く示す
   ↓
3. 規模判定フェーズ
   - 原因が確定した段階で、修正に必要な変更範囲を見積もる
   - AI が S/M/L を推測・提示 → ユーザー承認
   ↓
4. フロー選択
   ├─ S: 直接修正 → /build-check → コミット → /done
   ├─ M: 軽量な設計書作成 → /code-review → /build-check → /update-docs → コミット → /done
   └─ L: 通常の設計書作成（2段階設計）→ 以降 L フローと同じ
```

重要:
- **設計書は原因確定後に作成する**（原因不明のまま設計書を書き始めない）
- 「2回修正失敗したら立ち止まる」原則も継続（doom loop 対処）
- 緊急バグ（本番障害等）で原因特定が追いつかない場合は、S で暫定修正 → 後日 M/L で恒久対応 という2段構えを許容する

**バグ修正の設計書（M/L 規模）の書き方**:
- `docs/features/TEMPLATE.md` をそのまま使う
- セクションを読み替える:
  - 「1. 概要 > 背景」 = 問題の発見経緯・症状
  - 「1. 概要 > 方針」 = 根本原因の分析・修正アプローチ
  - 「3. 設計詳細」 = 修正内容・影響範囲・再発防止策
- 専用ディレクトリは作らない（運用で問題が出たら `docs/features/bugfixes/` 等を後から検討）

### 3-3. 2段階設計フロー

**対象**: **L 規模（必須）**

**「M 規模 + UX 変更あり」は L に引き上げる**方針に統一。理由:
- 2段階設計が推奨レベルだとスキップされる可能性が高い
- UX 変更がある時点で product-advisor による企画レビューの価値が出る
- 「M だけど 2段階」という例外ケースを作らない方が判定ロジックが単純

つまり:
- **UX 変更あり** → 自動的に L 扱い（2段階設計）
- M 規模 = UX 変更なしのケースに限定（1段階設計）

**Stage 1: 機能・画面設計**
- 対象画面と操作フロー
- 受け入れ条件（ユーザー目線）
- → `/design-review feature` でレビュー（**code-reviewer + product-advisor** を並列実行）
- → **ユーザー承認**
- → Stage 2 へ

**Stage 2: 技術設計**
- API 仕様（リクエスト・レスポンス）
- テーブル構造 / スキーマ変更
- サービス・フック設計
- → `/design-review tech` でレビュー（code-reviewer のみ）
- → 実装へ

**更新後のフロー図（L 規模）**:

```
作業依頼
  ↓
AI が S/M/L を推測・提示
  ↓
ユーザー承認（L 規模確定）
  ↓
/new-feature で設計書作成
  ↓
Stage 1 記入（機能・画面設計）
  ↓
/design-review feature
  (code-reviewer + product-advisor を並列実行)
  ↓
ユーザー承認（Stage 1 確定）
  ↓
Stage 2 記入（技術設計）
  ↓
/design-review tech
  (code-reviewer のみ)
  ↓
実装 → /code-review → /browser-test → /build-check → /update-docs → コミット → /done → プッシュ
```

**M 規模（UX 変更なし）** は従来通り1段階設計で OK。

### 3-4. `/design-review` のモード分割

**現状**: 単一モードで全てをチェック

**変更後**: モード引数を追加

```
/design-review feature  # Stage 1 レビュー（機能・画面・UX）
/design-review tech     # Stage 2 レビュー（技術設計）
/design-review          # 1段階設計用（従来動作と互換）
```

**`feature` モードの動作**:
1. 設計書の「3. 機能・画面設計（Stage 1）」セクションを読む
2. `code-reviewer` と `product-advisor` を**並列起動**（役割分担は §3-5 参照）
3. 両者の結果を統合してレビュー結果ファイルに保存（統合フォーマットは後述）
4. ユーザーには判定・件数・重要指摘の概要を報告

**`feature` モードでの `code-reviewer` の観点**:

従来の設計レビュー観点に加え、Stage 1 では **UI/UX 設計観点** も担当する:

- 設計書整合性（既存）:
  - 設計書間の矛盾チェック
  - ブラウザ評価計画の記入状況
- **UI/UX 設計観点（Stage 1 で追加）**:
  - デザインシステム準拠: `.claude/02_design_system/` との整合性
  - 既存画面のレイアウトパターンとの一貫性
  - 情報階層・視覚優先度の妥当性
  - アクセシビリティの考慮（ARIA、キーボードナビ、コントラスト）
  - レスポンシブ対応の方針（スマホ375px / PC1280px）
  - コンポーネント再利用の可否（既存コンポーネントで足りるか）

**並列レビュー結果の統合フォーマット**:

レビュー結果ファイル（`docs/reviews/YYYYMMDD_HHMMSS_design-review-feature.md`）の構造:

```markdown
# Stage 1 設計レビュー結果

- 実施日時: YYYY-MM-DD HH:MM
- 対象: （設計書パス）
- モード: feature (Stage 1)

## 総合判定: ✅ 承認 / ⚠️ 条件付き承認 / ❌ 差し戻し

## code-reviewer（設計書整合性）

| # | 重要度 | 内容 | 対象 | 対応状況 |
|---|--------|------|------|---------|

## product-advisor（企画・UX）

| # | 優先度 | 観点 | 指摘内容 | 対応状況 |
|---|--------|------|---------|---------|

## 両者の指摘が競合する場合

- 技術的に実現不可と code-reviewer が判断した項目を product-advisor が推奨している場合等
- 競合表を別途作成し、ユーザーに判断を仰ぐ
```

**矛盾・タイムアウト時の扱い**:
- 片方のエージェントがタイムアウトまたは失敗した場合、**成功した側の結果だけで一旦レポート** し、失敗したエージェントは再実行を promptする
- 両者の指摘が真逆のケース（技術制約 vs UX 要求）は自動解決せず **「要判断」として明示** しユーザーに委ねる
- 統合処理は `/design-review` スキルが担当（エージェント側は自分の領域だけ返す）

**スキル実装指針（部分読み戦略）**:

設計書が大規模化した際のトークンコスト削減のため、スキル側は**必要なセクションだけを部分読み**する。

- `/design-review feature`: 「メタ情報 + 3. 機能・画面設計」のみを読む（Stage 2 は読まない）
- `/design-review tech`: 「メタ情報 + 4. 技術設計」のみを読む（Stage 1 は参考のため軽く見る程度）
- `/update-docs`: メタ情報 + タスク一覧のみ（全文ロード不要）

**実装イメージ**:
```bash
# Stage 1 セクションの開始行を特定
grep -n "^## 3\\. 機能・画面設計" {設計書パス}
# 例: "100:## 3. 機能・画面設計（Stage 1）"

# Stage 2 セクションの開始行を特定（= Stage 1 の終端）
grep -n "^## 4\\. 技術設計" {設計書パス}
# 例: "250:## 4. 技術設計（Stage 2）"

# メタ情報（冒頭30行）+ Stage 1（100〜249行）のみを読む
# Read ツールの offset / limit を活用
```

これにより設計書が 1000 行を超えるような大規模機能でも、実質コストは小さい設計書と同程度に抑えられる。

**`tech` モードの動作**:
1. 設計書の「4. 技術設計（Stage 2）」セクションを読む
2. `code-reviewer` のみ起動（従来の設計レビュー観点）
3. 結果を保存・報告

**従来モード (`/design-review`)**: 1段階設計（M 規模で UX 変更なし）の場合に使用、互換維持

### 3-5. `product-advisor` の Stage 1 統合と役割分担

**現状**: どのフローからも呼ばれていない孤立エージェント

**変更後**: `/design-review feature` から自動起動される

#### Stage 1 レビューエージェントの役割分担

UI/UX レビューは2つの観点（企画・UXフロー vs UIレイアウト・デザインシステム）が混在するため、既存の2エージェントで分担する:

| エージェント | 担当観点 |
|-------------|---------|
| **product-advisor**<br>（企画・UX体験） | ・課題-解決策フィット<br>・ユーザーフローの自然さ<br>・摩擦ポイント・感情曲線<br>・代替案の検討余地<br>・優先度の妥当性<br>・競合・代替手段との差別化 |
| **code-reviewer**<br>（設計整合性 + UIデザイン） | ・設計書間の矛盾（既存）<br>・ブラウザ評価計画の記入状況（既存）<br>・デザインシステム準拠<br>・既存画面のレイアウトパターンとの一貫性<br>・情報階層・視覚優先度<br>・アクセシビリティの考慮<br>・レスポンシブ対応方針<br>・コンポーネント再利用性 |

詳細な code-reviewer の観点は §3-4 を参照。

#### YAGNI 判断: `design-reviewer` エージェントは作らない

UI/UX 専用の `design-reviewer` エージェントを新設する案もあったが、以下の理由で**見送る**:

1. **エージェント数の肥大化回避**: 既に5エージェント（coding-specialist / code-reviewer / documentation-manager / product-advisor / browser-tester）あり、これ以上増やすと入門ガイドで警告した「AI slop」（形骸化）の元
2. **code-reviewer は既にUI観点を持っている**: `/code-review` のUI変更時チェックリストに「既存レイアウトとの一貫性」「レスポンシブ対応」が既にある。拡張するだけで済む
3. **デザインシステムの規模が限定的**: 本テンプレの `.claude/02_design_system/` は2ファイル程度。本格的なデザインシステム運用（Figma連携、大規模コンポーネントライブラリ等）はまだない
4. **Stage 1 レビューの並列度を抑える**: 3並列（code-reviewer + product-advisor + design-reviewer）は統合が複雑。2並列で十分

**将来の再検討判断基準**:
- デザインシステムが大規模化する（Figma連携、50+コンポーネント）
- デザイナーが独立した職種として関わる開発体制になる
- Visual Regression Testing 等の専門ツールを導入する

これらの状況になったら、`design-reviewer` 専門化を再検討する。

#### product-advisor のレビュー観点（Stage 1 用）

- 課題-解決策フィット: 本当にこの機能で解決できるか
- ユーザーフローの自然さ
- 代替案の検討余地
- 優先度の妥当性
- 既存機能との整合性（重複・矛盾）

**呼び出しイメージ**:

```
Agent(subagent_type: "product-advisor", model: "sonnet", prompt:
  "Stage 1 設計レビュー。設計書 {path} の「3. 機能・画面設計」を読み、
   以下の観点で企画妥当性を評価して。
   - 課題-解決策フィット

   結果は短くまとめて、code-reviewer の結果と統合できる形式で返して。")
```

Stage 2 (`tech`) では呼ばない（技術論に立ち入らない原則）。

#### 1段階設計用の扱い（確定: 案A）

**M 規模で UX 変更なし** の場合は、**Stage 1 セクションは空のまま残し、ステータスを「⚪ 不要」に設定する**方針で統一する（案A）。

検討した選択肢:
- 案A: Stage 1 を「⚪ 不要」ステータスで空のまま残す ← **採用**
- 案B: 1段階用の軽量テンプレ `TEMPLATE_simple.md` を別に用意 ← 不採用（YAGNI、テンプレ数増加を避ける）
- 案C: `/new-feature` で規模に応じて不要セクションを自動削除 ← 不採用（スキル複雑化）

案A の利点:
- テンプレが1種類で済む
- 後から「やっぱり UX 変更もあった」と気づいた場合、Stage 1 セクションを埋めて L に引き上げる移行が容易
- スキル・ツール側の分岐ロジックが不要

#### セクションマーカー規約（部分読み対応）

設計書が大規模化しても低トークンで必要部分だけ読めるよう、セクション構造のルールを定める。

1. **主要セクションは `## N. セクション名` 形式で統一**
   - 例: `## 3. 機能・画面設計（Stage 1）`, `## 4. 技術設計（Stage 2）`
   - grep で開始行が特定できる形式にする

2. **Stage 1 / Stage 2 はそれぞれ独立した H2 セクション**
   - 入れ子や混在を避ける
   - 区切りの `---` と誘導文（`→ /design-review feature でレビュー...`）は継続

3. **メタ情報は先頭 30 行以内に収める**
   - 全体ステータス、Stage 1/2 ステータス、変更の性質、変更規模を冒頭で参照可能にする
   - `Read(offset=0, limit=30)` だけで全体把握できるようにする

4. **タスク一覧もメタ情報直後に配置**
   - 進捗確認用途で頻繁に読まれるため、冒頭近くに置く

スキルはこの規約に従って部分読みを実装する（§3-4 実装指針参照）。

#### Stage 2 で技術制約が発覚し Stage 1 に戻る手順

L 規模の実装では「技術設計中に実現不可と判明して UX を変える」ケースがある。対応:

1. Stage 2 記入中に矛盾を発見したら **Stage 2 を中断**
2. 設計書メタ情報の「Stage 1 ステータス」を `🟡 再レビュー中` に戻す
3. 該当の Stage 1 セクション（受け入れ条件・操作フロー等）を修正
4. `/design-review feature` を再実行（必要なら product-advisor も再レビュー）
5. ユーザー再承認
6. Stage 2 を再開

改訂履歴には「Stage 1 を再修正した版」として明記する。

## 6. 注意事項・オープンな論点

### 後方互換性

- 既存の `docs/features/yyyymmdd_*.md`（旧テンプレート構造）は**そのまま残す**
- 新テンプレートは今後作成する設計書から適用
- 既存設計書のマイグレーションは行わない

### 1段階設計の扱い（決定済み）

**案A で確定**。Stage 1 セクションを「⚪ 不要」ステータスで空のまま残す方式。§3-6 参照。

### スキル肥大化の懸念

`/design-review` にモードを追加するとスキル定義が複雑化する。**過剰な分岐ロジックは入門ガイドで警告した「AI slop」の原因**になる。シンプルに保つこと。

実装時の原則:
- モード分岐は最小限（`feature` / `tech` / 引数なし の3パターン）
- 各モードの処理は `Step 1`〜`Step 4` 程度に抑える
- 統合フォーマットも必要最小限に（§3-4 参照）

### 規模判定の AI バイアス補正（タスク#4 で対応）

AI が判定に迷った際に S 寄りの判断をする傾向を補正する。CLAUDE.md に以下を明記する（Phase 1 のタスク#4）:

- 判定に迷ったら **L 寄りで提示する**
- ファイル数や DB 変更の有無が不明確なら **M 以上で提示する**

これにより「S のつもりが実は L だった」の発生を抑える。

### トークンコストと部分読み戦略

1ファイル方式では設計書全体の読み込みコストが懸念されるが、**Read ツールの offset/limit を活用した部分読み戦略**で対応する。

- スキル側は該当 Stage のみを部分読み（§3-4 スキル実装指針 参照）
- メタ情報は先頭 30 行以内に収める（§3-6 セクションマーカー規約 参照）
- 設計書が 1000 行を超える場合は「1 機能」の粒度を見直す

試算（L 規模機能・設計書 400 行の場合）:
- 最適化なし: 約 56,000 トークン / 機能 ≒ $0.84
- 部分読み最適化後: 約 30,000 トークン / 機能 ≒ $0.45
- 2ファイル分割と同等の効率を1ファイル維持のまま実現できる
