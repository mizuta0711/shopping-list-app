## 1. 概要

### 背景

[企画書 v0.3](../企画書.md) Phase 10。MVP コアを使い続けるユーザーが感じる「もっと便利にしたい」課題に応える、3つの**派生機能**を1つのフェーズにまとめて設計する。

| 機能 | 解決する課題 | スコープ判定（Stage 1 レビュー後） |
|---|---|---|
| **F-22/F-23 セット機能（10.1）** | カレーセット・調味料セットなど、決まった組み合わせを毎回入力するのが面倒 | ✅ 着手 |
| **F-25 店舗別リスト（10.2）** | スーパー / 薬局 / 100均 など、店舗ごとに買うものを分けて管理したい | ⏸ 保留（既存「今日/また今度」での代用可否を MVP リリース後に観察してから判断） |
| **F-24 自動分類（10.3）** | リストが長くなると、買い物中にカテゴリ別に整列されてた方が動きやすい | ⏸ 保留（リスト規模・ニーズの実データを観察してから判断） |

3つは独立して機能する。Stage 1 設計レビュー（[`docs/reviews/20260504_055236_design-review-feature-phase10.md`](../reviews/20260504_055236_design-review-feature-phase10.md)）で**3機能同時導入によるコンセプト毀損リスク**が指摘されたため、**着手対象を 10.1 のみに縮退**し、10.2 / 10.3 は Stage 1 で構想を残しつつ実装は将来判断とする。

セット機能の差別化（既存の JSON エクスポート/インポートとの違い）: **ワンタップ追加の速度・買い物中の即応性**。JSON は永続バックアップ用途、セットは日常運用での再利用に最適化。

### 目的

- ユーザーが**頻繁に買う組み合わせをセット保存**し、ワンタップで買い物リストに追加できる
- ユーザーが**複数の店舗ごとに独立したリスト**を持てる（例: スーパー用、薬局用）
- 商品が**自動でカテゴリ分けされ**、買い物中にカテゴリでグルーピング表示できる

### 方針

| 観点 | 判断 |
|---|---|
| 実装順序 | **セット機能（10.1）のみ Stage 2 → 実装着手**。10.2 / 10.3 はニーズ観察後に再判断 |
| 機能の独立性 | 3機能は互いに**疎結合**にする。例えば 10.2 を入れない設定でも 10.1 / 10.3 が単体で動作する |
| データモデルの破壊的変更 | 10.1 のみで動かす場合、`ShoppingItem` 型は変更不要。10.2 着手時に `ShoppingItem.listId` 追加 + migrate 必要 |
| ストレージバージョン採番ルール | **Phase 9 完了済み**（main store は v2 据え置き）。Phase 10.1 は別 storage key (`shopping-list-app:sets:state`) で v1 開始のため main store のバージョンは据え置き。Phase 10.2 着手時に main store を v3、Phase 10.3 着手時に v4 へ採番予定 |
| クラウド同期との関係 | Phase 9 完了済み。**Phase 10.1 は 10.1a（LocalStorage 単独）と 10.1b（クラウド同期化）に分割**し、10.1a 完了後に 10.1b を別ファイルで Stage 2 設計する。Phase 10.2 / 10.3 は Phase 9 のスキームに後付け統合する方針 |
| カテゴリの実装方針（10.3 着手時） | MVP は**ハードコードされた語彙ベースのキーワード分類**（外部 ML API 不使用）。誤判定はユーザーが手動で修正可能 |
| カテゴリのマスター編集 | カテゴリそのものの追加・編集機能は **本フェーズではスコープ外**（将来検討） |
| シンプルさ維持 | 全ての機能を**オプショナル**にする。デフォルトでは「単一リスト・カテゴリ非表示・セットなし」で MVP 体験を維持 |
| 編集起動方式 | ロングプレスは**採用しない**（dnd-kit の 250ms drag delay と競合するため）。代わりに**カテゴリ別表示モード時のみ行末に編集アイコンを表示**（10.3 適用時）。10.1 のセット編集は通常の `/sets/[id]` ルート遷移で行う |
| リスト切替 UI | ヘッダーのドロップダウン置換は**採用しない**（モバイル発見性が低い）。代わりに **ScopeTabs と同居する「リストタブバー」**（水平スクロール可能なピル風タブ）として再設計（10.2 適用時） |
| カテゴリ別表示の位置付け | `SortKey` には**追加しない**。並び替えとは別の「**表示モード**」フラグとして分離（10.3 適用時） |

## 3. 機能・画面設計（Stage 1）

### 3-A. セット機能（F-22/F-23） — 着手対象

#### 3-A-1. 対象画面と操作フロー

| 画面 | URL | 新規/変更 | 概要 |
|------|-----|----------|------|
| セット一覧 | `/sets` | 新規 | ユーザーが作成したセット一覧。新規作成・編集・削除 |
| セット編集 | `/sets/new`, `/sets/[id]` | 新規 | セット名 + アイテム名（改行/句読点区切り）の編集 |
| メイン画面 | `/` | 変更 | 「セットから追加」ボタンを**入力欄左**（AddItemForm の隣）に配置（ヘッダーは過密のため避ける） |

**操作フロー: セットを使って買い物リストに一括追加**

1. メイン画面で「セットから追加」アイコンをタップ
2. ボトムシートに**セット一覧**（カレーセット 4品 / 調味料セット 6品 / etc.）が表示される
3. セットをタップ → 確認モーダル「カレーセットの 4 品を『今日』に追加します」
4. 「追加」をタップ → アイテムが**現在のアクティブタブ**（scope）に一括追加される（既存の `addItems` を流用、重複は自動スキップ）
5. ボトムシートが閉じてメイン画面に戻る

**セット定義は `scope` を持たない**。追加時のアクティブタブが採用される。

**操作フロー: セットの作成**

1. 設定画面 → 「セット管理」リンク → `/sets` ページ
2. `/sets` のヘッダー右上「+」ボタンから `/sets/new` へ
3. セット名入力 + 商品名（改行/句点/読点区切り）入力。**入力中に「X品が登録されます」のプレビュー表示**
4. 「保存」→ `/sets` に戻る

**操作フロー: セットがゼロ件の状態（CTA）**

- ボトムシート空状態に「セットを作成する」CTA ボタン
- `/sets` 空状態にも同様に「最初のセットを作成」ボタン

#### 3-A-2. 受け入れ条件

- [ ] `/sets` でセット一覧が表示される。0件なら「セットがまだありません」+ **「最初のセットを作成」CTA** が表示される
- [ ] `/sets/new` で名前と商品名を入力 → 保存できる
- [ ] `/sets/new` 入力中に「X品が登録されます」のプレビューが動的に更新される
- [ ] `/sets/[id]` でセットを編集・削除できる
- [ ] メイン画面の「セットから追加」アイコンで、現在のアクティブタブに一括追加できる
- [ ] セット 0 件のときボトムシートに「セットを作成する」CTA が表示される
- [ ] 一括追加時、既にリストにあるアイテムは（既存の重複スキップ仕様で）スキップされる
- [ ] セットは LocalStorage に永続化される（クラウド同期も将来対応可能な構造）

#### 3-B-4. データモデル変更

```typescript
type ShoppingList = {
  id: string;
  name: string;
  emoji: string | null;  // 任意
  createdAt: string;
  updatedAt: string;
};

type ShoppingItem = {
  // ... 既存フィールド
  listId: string;  // 追加: 所属リスト
};
```

`addItem` / `addItems` の引数に `listId` を必須として追加（`activeListId` は ShoppingMainView の state で保持）。
重複スキップキーは Phase 10.2 適用時に `(name, listId, scope, status)` に拡張。

LocalStorage migrate（Phase 9 が先か後かでバージョン番号が変わる）:
- 「メイン」リスト1件を作成
- 既存 items 全てに `listId` = メインの id を割り当て
- 書き込み前に旧 state を `shopping-list-app:state-backup-v{prev}` に保存（migrate 失敗時の復元用）

### 3-C. 自動分類（F-24） — ⏸ 保留（実装は MVP リリース後に再判断）

> **注意**: 本セクションは Stage 1 の構想記録。実装着手時は別ファイルで Stage 2 設計書を作成する。
> Stage 1 レビューで「リスト平均件数が分類を要するレベルか不明」「キーワード分類の誤判定がコンセプトを毀損」「ロングプレス編集は発見性が低い + dnd-kit と競合」との指摘あり。**カテゴリ別表示は SortKey ではなく独立した「表示モード」フラグとして分離**し、**編集起動はカテゴリ別表示時のみ行末に編集アイコンを表示**する形に再設計。
>
> **⚠️ Phase 13 完了後の方針見直し**: Phase 11（スワイプ方式）は実機での操作感が悪く Phase 13（[20260505_phase13-edit-mode.md](./completed/20260505_phase13-edit-mode.md)）で**編集モード切替型 UI に置換**された。本セクション §3-C-1 / §3-C-3 / §3-C-4 の「カテゴリ別表示モード時のみ行末に編集アイコン」方針は**不要化**されており、Phase 10.3 着手時は編集 UI 部分を「Phase 13 と同じ編集モードトグル方式」に統一する前提で再設計すること（カテゴリ別表示と編集モードは独立した直交フラグとして両立可能）。

#### 3-C-1. 対象画面と操作フロー

| 画面 | URL | 新規/変更 | 概要 |
|------|-----|----------|------|
| メイン画面 | `/` | 変更 | **「カテゴリ別表示」トグル**（並び替えとは独立した表示モード）追加 |
| アイテム編集 | （ダイアログ） | 新規 | **カテゴリ別表示モード時のみ行末に表示される編集アイコン** からダイアログを開く（ロングプレスは採用しない） |

**操作フロー: 自動分類**

1. ユーザーがアイテムを追加（例: 「玉ねぎ」）
2. クライアント側のキーワード分類ロジックで `category: '野菜'` が自動セット
3. ヘッダーの「カテゴリ別表示」トグルを ON → カテゴリでグルーピング表示

**操作フロー: 分類の手動修正**

1. カテゴリ別表示モードでアイテムを表示中、行末の **編集アイコン**（鉛筆）をタップ
2. 編集ダイアログが開く（名前 + カテゴリ）
3. カテゴリのドロップダウンから別のカテゴリを選択
4. 保存 → アイテムの `category` 更新、カテゴリ別表示にも反映

通常の表示モード（PENDING リスト）では編集アイコンは表示しない。誤分類が気になるユーザーだけ「カテゴリ別表示」をオンにして修正できる。

#### 3-C-2. カテゴリ設計

固定の語彙を持つ:
- 🥬 野菜（玉ねぎ、にんじん、キャベツ、もやし、…）
- 🥩 肉
- 🐟 魚
- 🥛 乳製品（牛乳、チーズ、ヨーグルト、…）
- 🍞 主食（パン、米、麺、…）
- 🧂 調味料（醤油、砂糖、塩、…）
- 🥫 食品（缶詰、レトルト、お菓子、…）
- 🧴 日用品
- 💊 薬・医療
- ❓ その他（既定値、未分類）

キーワードマッピングは静的 JSON で管理（数十〜数百キーワード）。
不明なら `その他` に入る。

#### 3-C-3. 受け入れ条件

- [ ] 新規追加時、商品名から自動で `category` がセットされる（マッチがなければ `その他`）
- [ ] migrate 時、既存アイテムは `category: 'その他'` として初期化される
- [ ] **「カテゴリ別表示」トグル**を ON にすると、カテゴリでグルーピングされ、各グループ内は登録日時順
- [ ] カテゴリ別表示モード時、各行末に編集アイコンが表示され、タップで編集ダイアログが開く（ロングプレス・既存ドラッグとの競合なし）
- [ ] 通常表示モード時は編集アイコンを表示しない
- [ ] カテゴリ別表示モード時、ドラッグ並べ替えは無効化（順序がカテゴリで規定されるため）
- [ ] カテゴリは LocalStorage に永続化される

## 5. ブラウザ評価計画

⚪ サブフェーズごとに Stage 2 で詳細化。Stage 1 時点での主要シナリオ骨子:

### セット機能（Phase 10.1）
- セット作成 → メインで一括追加 → 重複スキップが効く
- セットの編集・削除
- 空状態の CTA → セット作成フローに到達できる
- 入力プレビューが正しく件数表示する

### 店舗別リスト（Phase 10.2、保留）
- 既存ユーザーが Phase 10.2 にアップデート → 既存データが「メイン」リストに自動移行
- migrate 失敗時にバックアップから復元できる
- リスト切替・追加・削除
- 1つしかないリストは削除不可
- リストタブバーが水平スクロールで全リスト見える

### 自動分類（Phase 10.3、保留）
- 様々な商品名を入力して期待カテゴリに分類されるか
- 不明な商品は「その他」に入るか
- 手動修正の永続化
- **カテゴリ別表示モード時のみ編集アイコンが表示される**
- **編集アイコンタップが既存のドラッグジェスチャーと競合しない**
- カテゴリ別表示モード時、ドラッグ並べ替えが無効化されている

## 7. 注意事項

- **3機能を一気に実装しない**。本設計書は Stage 1 のみ。Stage 2 はサブフェーズ独立で別ファイル（`20260504_phase10.1-sets.md` 等）として作る
- **着手は Phase 10.1（セット機能）のみ**。10.2 / 10.3 は MVP リリース後の利用ログ・ニーズ観察を経て着手判断する
- **店舗別リスト導入時のデータ移行**は最重要。移行失敗で既存ユーザーがリストを失わないよう、`migrate` 関数を冪等にし、書き込み前にバックアップキーを保存する
- カテゴリの語彙設計は**英語ローカライズの可能性も視野**。MVP は日本語のみで OK
- **ロングプレスでの編集ダイアログは採用しない**（既存ドラッグ 250ms と競合）。代わりに「カテゴリ別表示モード時のみ行末に編集アイコンを表示」する方式
